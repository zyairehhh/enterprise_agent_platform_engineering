# Chapter 52 Compliance and Regulation

---

Compliance for enterprise Agent platforms should be implemented as engineering evidence. Regulations and frameworks describe risk, transparency, data rights, human oversight, and accountability. The platform has to translate these requirements into controls: policy configuration, audit logs, data lineage, model routing, deletion chains, cross-border boundaries, and review records. This chapter does not give legal advice. It explains how compliance requirements can be mapped into platform mechanisms so that teams can prove what happened, who approved it, which data was used, and which policy version was active.

The engineering question is usually concrete. A reviewer may ask why an Agent saw a specific customer field, which model processed the request, whether the data left a region, who approved the final report, and whether the same output can be reconstructed after an incident. A compliance answer cannot rely on memory or a slide deck. It needs evidence produced by the running system: identity, data source, semantic version, model route, policy decision, human approval, artifact status, and retention rule. The platform should make these records part of normal execution so review packets can be assembled from source systems instead of written manually after the fact.

---

## 52.1 Compliance engineering framework

A compliance engineering framework connects requirement, control, evidence, owner, and review cadence. Each compliance requirement should map to a platform control and a record that can be reviewed later. The mapping should be specific enough for engineers to implement. "Protect personal information" is too broad to test. A platform-level version should say which fields are sensitive, where masking runs, which logs exclude raw values, who can approve exceptions, and which sample verifies the behavior. This conversion from legal language to executable control is the core work of compliance engineering.

The mapping also needs a failure path. If a control fails, the platform should know whether to refuse the request, degrade the answer, require human review, or create an incident. For example, if the semantic layer cannot confirm a field's sensitivity label, a DataAgent should not silently continue with raw fields. It can return an unsupported-state message, ask for a safer aggregate, or route the case to a data owner. Compliance controls become useful when they define both normal behavior and the recovery path for missing evidence.

*Table 52-1: Compliance engineering mapping. Source: Compiled by this book.*

| Requirement | Platform control | Evidence |
| --- | --- | --- |
| Data minimization | Context assembly policy | Selected fields and redaction record |
| Human oversight | HITL state | Approval decision and actor |
| Transparency | Evidence reference | Source, version, and explanation |
| Accountability | Ownership matrix | Policy owner and reviewer |
| Deletion right | Deletion workflow | Object deletion and derived-index cleanup |

## 52.2 NIST AI RMF risk management

NIST AI RMF frames AI risk through governance, mapping, measurement, and management. For an Agent platform, these ideas become concrete when the system records task type, model route, tool permissions, evaluation result, risk category, and mitigation action. The platform should not treat RMF as a checklist detached from runtime. Risk mapping should influence routing, Guardrails, approval, logging, and incident response.

For example, a low-risk drafting assistant may need output review and basic logging. A high-risk workflow that influences financial disclosure needs stronger data lineage, human confirmation, retention, and incident response. The same framework can cover both, but the platform must attach the risk level to actual runtime controls.

## 52.3 EU AI Act risk classification

EU AI Act risk classification matters when Agent systems affect regulated decisions, employment, finance, education, public services, or other high-impact domains. The platform should record which applications fall into stricter review categories and which controls are required before release. Risk classification should be attached to the application and task as well as the model. The same model may be low risk in an internal drafting tool and higher risk in a workflow that influences customer eligibility.

This task-level classification should appear in release records. When an application changes from recommendation to execution, its risk class may change. When a tool changes from read-only to write-capable, the required oversight may change. Compliance review should therefore be tied to capability changes and calendar reviews.

## 52.4 China generative AI compliance requirements

China's generative AI compliance requirements emphasize content safety, data legality, personal information protection, labeling, and service accountability. For platform teams, this means data-source review, content policy, sensitive-data masking, model-output review, and incident handling. The platform should keep evidence for data source, prompt template, model provider, output policy, and user feedback. This helps teams answer compliance questions without reconstructing history from chat logs.

Chinese compliance requirements also make content governance part of the platform. If an output is refused, revised, or reported, the platform should preserve policy version and reason code. If a model provider changes, the platform should record which traffic moved and which data categories are allowed on the new route.

## 52.5 Content provenance and C2PA

Content provenance records where generated content came from and how it changed. C2PA-style metadata can help mark generated images, documents, or reports, but provenance is broader than a watermark. It includes model version, input references, artifact version, reviewer, and publication status. For reports and artifacts, provenance should connect to EvidenceRef and Trace. A reader should know whether a paragraph came from model generation, human editing, or approved publication.

Provenance should be designed for normal users as well as auditors. A report viewer does not need internal prompt text, but should be able to inspect source data, generated sections, review state, and publication version. This makes provenance part of product trust as well as compliance review.

## 52.6 Compliance control matrix generation and validation

A compliance control matrix lists requirements, controls, owners, evidence, and test methods. The matrix should be generated from platform configuration where possible, then reviewed by compliance and platform teams.

*Table 52-2: Compliance control matrix fields. Source: Compiled by this book.*

| Field | Purpose |
| --- | --- |
| Requirement id | Link to policy or regulation |
| Control | Platform mechanism |
| Owner | Responsible team |
| Evidence | Record used for audit |
| Test | How the control is verified |
| Review cadence | When it is rechecked |

Validation should test whether evidence is actually produced during normal runs. A control that exists in a document but leaves no runtime record is weak. The matrix should also capture negative evidence. If a request is blocked because a user lacks permission, that denial is evidence. If a deletion request cannot remove an audit record for legal reasons, the retained record and reason should be evidence. Compliance often depends on proving that the platform refused or limited an action.

## 52.7 Platform landing points for compliance requirements

Compliance requirements land in several platform modules. Identity and tenant isolation enforce access. Context assembly enforces data minimization. Tool Registry enforces permission. Guardrails enforce content policy. Runtime records human oversight. Trace and audit store evidence. Evaluation checks behavior drift. When a new compliance requirement appears, the team should decide which module owns it and which evidence proves it. This ownership decision prevents scattered implementations. If every application masks data differently, audit becomes expensive and inconsistent. If masking belongs to Guardrails and context assembly, the platform can test it once and reuse the evidence across applications.

Side channels should be included in the same design. Generated reports, debug screenshots, failed samples, evaluation datasets, exported CSV files, support tickets, and collaboration-tool messages may contain the same sensitive content as the final answer. Many compliance incidents come from these secondary artifacts, not from the main response shown in the chat window. The platform should define who can view traces, who can export artifacts, how long failed samples are retained, and which fields are removed before a sample enters evaluation. Treating side channels as system outputs gives compliance teams a realistic view of where data travels.

## 52.8 Platform landing points for compliance evidence

Evidence should be collected close to the control. A tool permission decision belongs near the executor. A human approval decision belongs in Runtime. A masking decision belongs in Guardrails. A metric-definition decision belongs in the semantic layer. Central audit stores can aggregate evidence, but the source module should still own correctness. Aggregation without source ownership creates logs that are hard to trust.

Evidence should also be queryable by incident and review workflows. A compliance reviewer may start from a user complaint, a run id, an artifact id, a policy version, or a data subject request. The audit model should support these entry points without requiring manual log stitching.

Evidence views should answer review questions directly. A reviewer should be able to open a Run and see purpose, user role, data categories, model route, tool calls, human approvals, generated artifacts, retention status, and policy versions. Raw traces are still useful for engineers, but compliance teams need a curated view that connects each control to the record that proves it. That view should also show responsible owners. If an output was approved, the record should identify the approver and the scope of approval. If a request was denied, the record should show the policy and reason code without exposing sensitive prompt content unnecessarily.

## 52.9 Configuration governance under regulatory changes

Regulations and internal policies change. The platform should treat compliance configuration as versioned material: policy rules, risk labels, masking fields, retention periods, model-provider allowlists, and cross-border routing rules. Configuration changes should go through review, staged rollout, and regression tests. The platform should know which runs used which configuration version.

Regulatory changes should not become emergency code edits. If retention period, masking field, provider allowlist, or review threshold is configuration, the platform can test and release it through a controlled path. Code changes are still needed for new control types, but many policy adjustments should be data-driven.

## 52.10 Data subject rights and deletion chain

Deletion requests are difficult for Agent systems because data may appear in raw files, parsed chunks, embeddings, memory, logs, artifacts, and evaluation samples. The platform should define a deletion chain that traces derived objects and confirms cleanup.

The deletion chain should distinguish operational deletion from audit retention. Some records must be removed from retrieval, memory, and generated artifacts so they no longer influence future answers. Other records may need to be retained in minimized form to prove that the platform handled the request correctly. The platform should avoid deleting evidence so aggressively that later review becomes impossible, while also avoiding broad retention of raw personal data. A practical implementation stores hashes, object ids, retention reasons, and deletion receipts, and it keeps raw content only where policy allows.

*Table 52-3: Deletion chain objects. Source: Compiled by this book.*

| Object | Deletion action |
| --- | --- |
| Raw file | Remove or archive according to policy |
| Parsed text | Delete extracted content |
| Embedding | Remove vector entry |
| Memory | Delete user or session memory |
| Artifact | Delete or mark unavailable |
| Audit log | Retain minimal legal evidence when required |

The deletion chain should record what was removed, what was retained by legal basis, and who approved the action. Deletion is also an evaluation problem. The platform should test that deleted content no longer appears in retrieval, memory, generated artifacts, or future evaluation samples. Otherwise a file can be removed from storage while its embedding or extracted text continues to affect answers.

## 52.11 Engineering input for compliance review

Compliance review needs engineering input. Reviewers should receive architecture diagrams, data-flow descriptions, tool permission lists, model-provider routes, retention configuration, evaluation results, incident history, and user-facing explanations. The platform team should prepare these materials as part of release evidence, not after a review request arrives. Review packets should be standardized. A small internal template can ask for task description, data categories, model route, tool list, policy controls, evaluation results, and known limitations. This reduces review time and makes it easier to compare applications.

## 52.12 Cross-border and external model-call boundaries

External model calls and cross-border routing require clear boundaries. The platform should know which data can leave which region, which providers are allowed, which fields must be masked, and which logs must remain local. Routing decisions should be visible in Trace. If a request uses an external provider, the record should show provider, region, model, data category, masking result, and policy version.

Fallback routing needs the same control. If the primary model fails and the gateway sends traffic to another provider, the fallback must satisfy the same data and region policy. A reliable gateway should fail closed for restricted data instead of quietly using an unauthorized route.

## 52.13 Compliance samples and continuous regression

Compliance samples should test policy behavior over time. Samples may include sensitive-data masking, deletion requests, cross-border routing, content refusal, human oversight, and provenance records. These samples should run after model, prompt, policy, retrieval, and tool changes. Continuous regression keeps compliance from becoming a one-time release document. When behavior drifts, the platform should identify the affected requirement and owner.

Regression results should feed governance meetings. If a new model improves quality but fails a masking sample, the release decision should include that trade-off explicitly. If a new policy reduces leakage but creates many false refusals, business owners should review the affected workflows before full rollout.

## 52.14 Compliance evidence package and release rhythm

The deliverable of compliance engineering should not be a meeting note from one review. The platform needs reusable compliance evidence packages. A package should contain scenario description, user scope, model and tool list, data sources, data classification, cross-border-call notes, content-safety policy, output-retention policy, deletion chain, red-team and compliance sample results, human-review points, and incident-response plan. When an Agent or capability goes live, legal, internal control, security, and business owners can review material extracted from the package. Compliance review then avoids asking the same basic questions from the beginning each time.

The evidence package should be tied to release rhythm. Model upgrades, external model calls, new data sources, enabled write tools, expanded content-generation scope, and user expansion from internal to external groups all change compliance risk. A release record should state which parts of the evidence package were triggered: whether personal information processing was added, whether cross-border transfer was added, whether content generation and distribution changed, whether deletion requests are affected, and whether compliance samples need to be rerun. If no risk item is triggered, the release can follow a light path. If a high-risk item is triggered, formal review is required.

The package should also handle regulatory change. When a regulation or supervisory guideline changes, the team needs to know which Agents, data sources, model calls, and output scenarios are affected. If evidence is scattered across project documents and chat history, impact analysis becomes manual search. With structured evidence, the platform can query affected scenes by data type, user scope, model source, output channel, and retention policy. Compliance teams then see reviewable risk facts instead of a list of Agent names.

The first platform version can start with a small required field set: scenario, users, data, model, tools, output, retention, deletion, audit, and owner. The fields do not need to cover every legal clause at the beginning, but they should answer the common questions needed for launch and review. As new regulation, business, and incidents appear, the fields can expand into more detailed controls. Compliance maturity depends on whether evidence can be updated and reused over time, not on how long the first review document is.

## 52.15 Retention and deletion boundaries for compliance evidence

Compliance engineering is not a request to save more logs. Evidence retention should state what is stored, how long it is stored, who can access it, when it is deleted, and how deletion is proven. Agent runs generate user input, tool parameters, retrieval fragments, model output, approval comments, Trace, evaluation samples, and exported artifacts. These materials have different compliance requirements. Some need longer retention for audit. Some should be masked or deleted quickly. Some should keep only fingerprints and version identifiers.

Deletion boundaries must cover related assets. When a user or business process requires deletion, the platform should inspect conversation text together with Memory, vector indexes, evaluation samples, report Artifacts, caches, logs, and backups. If some evidence must remain for audit, the record should state the legal basis, retention period, and access restriction. Compliance teams need to see how engineering systems execute the policy, along with the policy text.

A compliance evidence package should support external accountability. When auditors or regulators ask about a decision, the platform should provide model version, data source, permission judgment, human approval, output destination, retention policy, and deletion state. Clear evidence packages reduce reliance on verbal explanation. Chapter 52 translates regulatory requirements into engineering records that can be executed, reviewed, and deleted according to policy.

## 52.16 Shared rhythm between compliance evidence and product release

Compliance evidence should be part of the product release rhythm. Many teams prepare compliance material after a capability is already live. By then architecture, data flow, model routing, permission policy, and user messages may already be fixed, leaving documentation to explain risk after the fact. A steadier approach is to mark data categories, user scope, external calls, output destinations, and retention requirements during requirement review; confirm Trace, approval, and deletion paths during development; and generate the evidence package and run compliance samples before release. Compliance then becomes a platform capability, not a late document.

The release rhythm should also detect compliance triggers. A new external model, personal-information processing, external users, export channels, cross-border routing, and automatic write actions should trigger stricter review. A wording change or a low-risk read-only optimization can follow a lighter path. If the release system chooses review depth based on triggers, teams do not need to send every change through the same heavy process, and high-risk changes are less likely to be treated as ordinary features.

A first version can use a structured release note to support this rhythm. The note should cover scenario, users, data, model, tools, output, retention, deletion, audit, and owner. The fields do not need to map every legal clause at the beginning, but they should answer common launch questions. As regulation changes and incidents produce new lessons, the fields can expand. The value of compliance engineering is that evidence can move with product iteration instead of being rewritten as static material for each review.

## 52.17 Productizing compliance audit queries

Compliance audit should not depend on engineers manually searching logs. Auditors need to query evidence by user, Run, artifact, data category, model provider, tool, approver, and policy version. If every audit requires ad hoc log stitching, the platform cannot prove that a control works over time. Audit query should become a product capability, at least through controlled views for compliance, security, and internal-control teams.

Productized audit views should control information detail. Auditors need to see whether evidence exists, whether a control executed, who approved the action, and where output went. They do not always need full user input or raw sensitive fields. Engineering views and audit views can share the same Trace foundation while showing different fields by role. This supports audit and reduces leakage risk from audit material itself.

A first version can support several query types: provenance chain for an artifact, data-access record for a user, data category for an external model call, and hits and exceptions for a policy. As scenarios expand, the platform can add data-subject requests, cross-border calls, and deletion proof.

## 52.18 Compliance evidence as an internal product

Compliance evidence becomes useful when internal teams can consume it without opening engineering tickets. Legal, internal audit, security, data governance, and business owners ask different questions about the same Run. Legal may ask whether the output falls under a regulated use case. Audit may ask who approved an export. Data governance may ask which data category entered the model. Business owners may ask whether a user correction changed a published artifact. The platform should provide role-based evidence views, because one raw log export gives every reviewer the same sensitive material and still leaves common questions hard to answer.

An internal evidence product needs stable vocabulary. Terms such as data category, model provider, policy version, artifact state, approval state, and deletion receipt should mean the same thing across dashboards, release records, and incident reports. If each team renames the same field, compliance review turns into translation work. A shared vocabulary also lets teams compare cases across Agents and business domains.

The product should include guided queries. Auditors should not need to know table names or Trace internals to answer common questions. They should be able to start from an artifact, user, provider, policy, or time window, then drill into the relevant evidence. Sensitive details can remain masked unless the user has a specific approval path. This design keeps audit work efficient while reducing the chance that audit access becomes another leakage channel.

A first version can expose a small catalog of evidence queries and a request path for deeper review. The important point is that compliance evidence becomes part of platform operation. When teams can retrieve the same evidence during release, incident review, and external audit, compliance stops being a parallel documentation process and becomes a shared engineering record.

## 52.19 Evidence freeze and review before external audit

Before an external audit, customer security assessment, or regulatory inquiry, the platform should freeze a reviewable evidence package. Freezing evidence does not mean stopping business activity. It means fixing the audit window, use-case scope, model version, policy version, data category, approval record, Trace samples, and deletion receipts so the team does not keep overwriting historical facts with the latest system state. Agent behavior changes as models, policies, prompts, tools, and knowledge bases evolve. Without an evidence freeze, a team may explain a past event using today's configuration and create inconsistency across documents, logs, and reviewer answers.

The frozen package should preserve the minimum evidence needed for review. Auditors usually need to confirm whether a control executed, who approved an action, whether data entered an external model, whether output was published, and whether users had an appeal path. They do not always need full prompts, raw attachments, or sensitive fields. A practical evidence package can separate summary, verifiable reference, and controlled detail. The summary states the control result. The references point to Trace, artifacts, approvals, and policy versions. Controlled details open only through an approved path. This structure lets the team answer audit questions while reducing data exposure during the audit itself.

The review process should also handle historical changes. If the audit window includes a model switch, policy exception, manual approval bypass, data deletion, or artifact withdrawal, the evidence package should state the before-and-after state, reason, approver, and affected scope. For deletion requests or data-subject rights requests, the platform should show which stores were updated, which derived artifacts retain required audit records, and which caches were invalidated. Without this evidence, compliance teams can only give principle-level statements, which are weak when customers or regulators ask for a concrete sequence of events.

A first version can turn evidence freeze into a shared capability for release and audit. Each high-risk use case can produce a periodic audit snapshot containing use-case definition, model and policy versions, external calls, human supervision, user appeals, deletion requests, and sample replay results. When an audit starts, the team begins from the snapshot and adds context instead of rebuilding the story from logs. This makes compliance response repeatable. It also makes discrepancies easier to find before external reviewers see them, because release records, incident reviews, and audit packages rely on the same evidence structure.

## 52.20 Change management for the control matrix

A compliance control matrix is not a one-time document. Use-case scope, model provider, data source, cross-border path, user group, output form, and human supervision can all change. The matrix has to change with them. If the control matrix is written only before launch, it may no longer describe the real system six months later. The platform should manage it as a versioned asset. Whenever a high-risk use case is added, an external model is switched, a data category changes, an approval path is adjusted, or a deletion policy is modified, the team should decide whether a control item is affected.

Control-matrix changes need engineering input. Compliance teams need to know which Runs are affected, which policy versions change, which evidence fields are added or removed, and which historical artifacts still operate under an older control. Platform teams need executable requirements from the matrix: which policy to add, which Trace field to capture, which approval state to expose, which retention rule to enforce, and which evaluation sample to run. A matrix that lists only regulatory clauses and owners is hard for engineers to translate into system behavior.

Changes also need regression samples. Moving a control from required human approval to sampling changes who carries risk. Reclassifying a data category as sensitive affects permission, masking, retention, and export. Switching a model provider to an overseas service affects cross-border evidence and contractual records. Each type of change should have samples that prove the control still executes. After those samples pass, the matrix version, policy version, and release record should be linked, so auditors can recover the basis used at the time.

A first version can use a lightweight change flow. A control-matrix change request needs to answer four questions: which use cases are affected, which controls are added or changed, where evidence is produced, and which samples prove the control works. This flow does not turn engineers into compliance specialists. It turns compliance requirements into fields, states, and samples the platform can execute. The closer the matrix stays to runtime evidence, the less external audit depends on human memory.

## 52.21 Evidence sampling for compliance controls

Compliance controls should not be checked only during launch review. The platform should periodically sample real Runs and verify whether fields in the control matrix still have supporting evidence. Sampling can be stratified by risk level, tenant, model route, tool type, data domain, and export behavior. Each sample should answer who the user was, where the data came from, where the model ran, which tools were called, whether the output entered a downstream process, whether human review occurred, and whether retention or deletion policy applied.

Sampling results should feed the control matrix. If a class of Runs often lacks an owner, the release template is incomplete. If export samples lack masking records, the tool or frontend is not writing evidence. If external model calls lack region information, the gateway log fields are insufficient. Compliance teams should turn missing fields into platform backlog items and retest before the next release or audit freeze.

A first version can sample a small number of high-risk Runs each month and produce an evidence-sampling report. The report does not need to cover every regulatory clause. It should focus on whether the evidence chain is complete, whether fields can be reused, and whether responsible owners can explain the decision. Compliance engineering then moves from a static checklist into continuous verification.

Evidence sampling should also compare declared controls with actual user paths. Many compliance reviews describe an approval step, a deletion path, or a masking rule, but production users may reach the same outcome through a shortcut, batch job, admin tool, or copied artifact. Sampling should therefore start from real Runs and artifacts, then compare them with the documented controls. For each selected case, reviewers should trace the path from user request to model call, tool call, output, approval, storage, export, and deletion state. If the path differs from the control matrix, the platform should update either the system or the matrix. Leaving the two inconsistent creates risk during customer assessments and regulatory inquiries.

The sampling method should include negative cases. A compliant Run is useful, but a denied export, a failed deletion request, a blocked external model call, and an appealed content decision often reveal more about the control design. These cases show whether users receive accurate messages, whether reviewers can reconstruct context, whether policy exceptions have expiry, and whether the platform can prove that the denied action did not continue through another path. Compliance evidence is stronger when it covers refusal, recovery, and successful use in the same review model.

Compliance teams also need a way to turn sampling findings into engineering work. A missing field in Trace, an unclear data category, an unowned exception, or an unverifiable deletion receipt should become a backlog item with owner and retest condition. Otherwise monthly sampling produces reports that do not change the platform. The backlog should state the control affected, the failing evidence, the system component to change, the sample that will prove repair, and the next review date. This makes compliance governance visible to product and engineering teams without forcing engineers to interpret legal text on their own.

External model calls deserve special attention in sampling. The evidence package should state provider, region, data category, contract boundary, fallback route, retention statement, and whether user content or retrieved evidence entered the external service. A model gateway can record these fields once and reuse them across release review, audit query, and incident review. Without gateway-level evidence, compliance teams must infer behavior from application logs and vendor settings, which is fragile when applications change model routes or use multiple providers.

Deletion and retention sampling should cover derived artifacts. Agent outputs often become reports, dashboards, evaluation examples, cached responses, Memory entries, vector chunks, and shared documents. A deletion request that removes only the original conversation can leave derived material visible. The platform does not need to erase every audit record, but it should distinguish live content, derived content, retained evidence, and legally required records. Each category needs an action and receipt. This distinction lets compliance teams explain why some references remain while user-facing content has been removed.

Compliance review should also inspect human review. HITL records are part of compliance evidence when the platform claims human supervision. The record should show who reviewed, what material was visible, which alternatives were available, what decision was made, and whether the decision changed the output or tool action. A checkbox that says "reviewed" does not prove meaningful supervision. The platform should record enough context to show that review happened before the risky action and that the reviewer had authority to make the decision.

For a first version, the compliance program can focus on a small sample set: external model calls, sensitive data output, regulated content generation, high-risk write actions, and deletion requests. Each sample should be traceable to a control matrix item and a release record. The goal is to establish a repeatable evidence habit. Once the platform can sample, repair, and retest these paths, it can add more regulations and business domains without rewriting the compliance process from scratch.

## 52.22 Human-supervision evidence and deletion-retention checks

Many compliance frameworks mention human supervision, but a platform should not store only a checkbox that says a human reviewed the case. Useful supervision evidence states who reviewed, what material was visible before the decision, which actions were available, whether the reviewer had authority, and whether the decision changed an output or tool action. If review happens after the risky action, or if the reviewer cannot see key evidence, the supervision claim is weak. HITL records should bind to Run, Trace, policy version, and artifact version.

Human supervision should also cover appeals and exceptions. A user appeal of a compliance block, a reviewer release of an export, or a business-owner approval of a temporary exception all become compliance evidence. The platform should record scope, expiry, retest requirement, and follow-up owner. An exception without expiry becomes a long-term bypass. A release without retest requirements leaves future releases unable to judge whether the risk still exists.

Deletion and retention checks must cover derived artifacts. Agent outputs often become reports, dashboards, evaluation samples, caches, Memory entries, vector chunks, and shared documents. A deletion request that removes only the original conversation can leave derived content visible. The platform does not have to delete every audit record, but it must distinguish live content, derived content, retained evidence, and legally required records, then provide an action and receipt for each category. Compliance teams can then explain why user-facing content was removed while some audit references remain.

A first version can include human supervision and deletion retention in the same high-risk sampling program. Sampling checks external model calls, sensitive data output, regulated content generation, high-risk write actions, and deletion requests. Each sample should connect to a control matrix item and release record. The aim is not to cover every regulation at once. It is to build evidence that can be sampled, repaired, and retested.

## 52.23 Service interface for the compliance evidence catalog

The compliance evidence catalog needs a service interface. Legal, internal audit, security, data-governance teams, and business owners ask different questions about the same Run. If every query depends on engineers exporting logs, audit response is slow and may expose too much sensitive material. The platform can expose the catalog as a controlled service: query by artifact, user, model provider, policy version, approver, data category, and time window, then return role-specific detail.

The service should also keep query records. Who viewed which evidence, when sensitive fields were expanded, and whether material was exported should all enter audit logs. Compliance evidence itself then becomes governed. A first version does not need a complex portal. Fixed query templates and an approval path for deeper review can turn common audit questions into repeatable product capability.

## 52.24 Evidence response for customer inquiries

Customer security inquiries test whether compliance evidence is usable. Customers often ask about model providers, data transfer, log retention, deletion requests, human approval, incident notification, and security certification. If evidence is scattered across procurement, engineering, legal, and business documents, responses are slow and inconsistent.

A first version can prepare customer-inquiry templates that map common questions to evidence-catalog fields and owners. The template does not replace formal legal review, but it helps the platform team find the source of facts quickly. Compliance capability then supports sales, security assessment, customer trust, and internal audit through the same evidence base.

## Chapter Recap

Compliance becomes practical when requirements map to controls, evidence, owners, and tests. Agent platforms need versioned policy configuration, data-rights workflows, cross-border boundaries, provenance, review inputs, and continuous regression. The goal is not to turn engineers into lawyers, but to make compliance claims traceable in the platform.
