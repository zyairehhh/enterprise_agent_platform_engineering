# Chapter 52 Compliance and Regulation

---

Compliance for enterprise Agent platforms should be implemented as engineering evidence. Regulations and frameworks describe risk, transparency, data rights, human oversight, and accountability. The platform has to translate these requirements into controls: policy configuration, audit logs, data lineage, model routing, deletion chains, cross-border boundaries, and review records.

This chapter does not give legal advice. It explains how compliance requirements can be mapped into platform mechanisms so that teams can prove what happened, who approved it, which data was used, and which policy version was active.

---

## 52.1 Compliance engineering framework

A compliance engineering framework connects requirement, control, evidence, owner, and review cadence. Each compliance requirement should map to a platform control and a record that can be reviewed later.

The mapping should be specific enough for engineers to implement. "Protect personal information" is too broad to test. A platform-level version should say which fields are sensitive, where masking runs, which logs exclude raw values, who can approve exceptions, and which sample verifies the behavior. This conversion from legal language to executable control is the core work of compliance engineering.

*Table 52-1: Compliance engineering mapping. Source: Compiled by this book.*

| Requirement | Platform control | Evidence |
| --- | --- | --- |
| Data minimization | Context assembly policy | Selected fields and redaction record |
| Human oversight | HITL state | Approval decision and actor |
| Transparency | Evidence reference | Source, version, and explanation |
| Accountability | Ownership matrix | Policy owner and reviewer |
| Deletion right | Deletion workflow | Object deletion and derived-index cleanup |

## 52.2 NIST AI RMF risk management

NIST AI RMF frames AI risk through governance, mapping, measurement, and management. For an Agent platform, these ideas become concrete when the system records task type, model route, tool permissions, evaluation result, risk category, and mitigation action.

The platform should not treat RMF as a checklist detached from runtime. Risk mapping should influence routing, Guardrails, approval, logging, and incident response.

For example, a low-risk drafting assistant may need output review and basic logging. A high-risk workflow that influences financial disclosure needs stronger data lineage, human confirmation, retention, and incident response. The same framework can cover both, but the platform must attach the risk level to actual runtime controls.

## 52.3 EU AI Act risk classification

EU AI Act risk classification matters when Agent systems affect regulated decisions, employment, finance, education, public services, or other high-impact domains. The platform should record which applications fall into stricter review categories and which controls are required before release.

Risk classification should be attached to the application and task as well as the model. The same model may be low risk in an internal drafting tool and higher risk in a workflow that influences customer eligibility.

This task-level classification should appear in release records. When an application changes from recommendation to execution, its risk class may change. When a tool changes from read-only to write-capable, the required oversight may change. Compliance review should therefore be tied to capability changes and calendar reviews.

## 52.4 China generative AI compliance requirements

China's generative AI compliance requirements emphasize content safety, data legality, personal information protection, labeling, and service accountability. For platform teams, this means data-source review, content policy, sensitive-data masking, model-output review, and incident handling.

The platform should keep evidence for data source, prompt template, model provider, output policy, and user feedback. This helps teams answer compliance questions without reconstructing history from chat logs.

Chinese compliance requirements also make content governance part of the platform. If an output is refused, revised, or reported, the platform should preserve policy version and reason code. If a model provider changes, the platform should record which traffic moved and which data categories are allowed on the new route.

## 52.5 Content provenance and C2PA

Content provenance records where generated content came from and how it changed. C2PA-style metadata can help mark generated images, documents, or reports, but provenance is broader than a watermark. It includes model version, input references, artifact version, reviewer, and publication status.

For reports and artifacts, provenance should connect to EvidenceRef and Trace. A reader should know whether a paragraph came from model generation, human editing, or approved publication.

Provenance should be designed for normal users as well as auditors. A report viewer does not need internal prompt text, but should be able to inspect source data, generated sections, review state, and publication version. This makes provenance part of product trust as well as compliance review.

## 52.6 Compliance control matrix generation and validation

A compliance control matrix lists requirements, controls, owners, evidence, and test methods. The matrix should be generated from platform configuration where possible, then reviewed by compliance and platform teams.

*Table 52-2: Compliance control matrix fields. Source: Compiled by this book.*

| Field | Purpose |
| --- | --- |
| Requirement id | Link to policy or regulation |
| Control | Platform mechanism |
| Owner | Responsible team |
| Evidence | Record used for audit |
| Test | How the control is verified |
| Review cadence | When it is rechecked |

Validation should test whether evidence is actually produced during normal runs. A control that exists in a document but leaves no runtime record is weak.

The matrix should also capture negative evidence. If a request is blocked because a user lacks permission, that denial is evidence. If a deletion request cannot remove an audit record for legal reasons, the retained record and reason should be evidence. Compliance often depends on proving that the platform refused or limited an action.

## 52.7 Platform landing points for compliance requirements

Compliance requirements land in several platform modules. Identity and tenant isolation enforce access. Context assembly enforces data minimization. Tool Registry enforces permission. Guardrails enforce content policy. Runtime records human oversight. Trace and audit store evidence. Evaluation checks behavior drift.

When a new compliance requirement appears, the team should decide which module owns it and which evidence proves it.

This ownership decision prevents scattered implementations. If every application masks data differently, audit becomes expensive and inconsistent. If masking belongs to Guardrails and context assembly, the platform can test it once and reuse the evidence across applications.

## 52.8 Platform landing points for compliance evidence

Evidence should be collected close to the control. A tool permission decision belongs near the executor. A human approval decision belongs in Runtime. A masking decision belongs in Guardrails. A metric-definition decision belongs in the semantic layer.

Central audit stores can aggregate evidence, but the source module should still own correctness. Aggregation without source ownership creates logs that are hard to trust.

Evidence should also be queryable by incident and review workflows. A compliance reviewer may start from a user complaint, a run id, an artifact id, a policy version, or a data subject request. The audit model should support these entry points without requiring manual log stitching.

## 52.9 Configuration governance under regulatory changes

Regulations and internal policies change. The platform should treat compliance configuration as versioned material: policy rules, risk labels, masking fields, retention periods, model-provider allowlists, and cross-border routing rules.

Configuration changes should go through review, staged rollout, and regression tests. The platform should know which runs used which configuration version.

Regulatory changes should not become emergency code edits. If retention period, masking field, provider allowlist, or review threshold is configuration, the platform can test and release it through a controlled path. Code changes are still needed for new control types, but many policy adjustments should be data-driven.

## 52.10 Data subject rights and deletion chain

Deletion requests are difficult for Agent systems because data may appear in raw files, parsed chunks, embeddings, memory, logs, artifacts, and evaluation samples. The platform should define a deletion chain that traces derived objects and confirms cleanup.

*Table 52-3: Deletion chain objects. Source: Compiled by this book.*

| Object | Deletion action |
| --- | --- |
| Raw file | Remove or archive according to policy |
| Parsed text | Delete extracted content |
| Embedding | Remove vector entry |
| Memory | Delete user or session memory |
| Artifact | Delete or mark unavailable |
| Audit log | Retain minimal legal evidence when required |

The deletion chain should record what was removed, what was retained by legal basis, and who approved the action.

Deletion is also an evaluation problem. The platform should test that deleted content no longer appears in retrieval, memory, generated artifacts, or future evaluation samples. Otherwise a file can be removed from storage while its embedding or extracted text continues to affect answers.

## 52.11 Engineering input for compliance review

Compliance review needs engineering input. Reviewers should receive architecture diagrams, data-flow descriptions, tool permission lists, model-provider routes, retention configuration, evaluation results, incident history, and user-facing explanations.

The platform team should prepare these materials as part of release evidence, not after a review request arrives.

Review packets should be standardized. A small internal template can ask for task description, data categories, model route, tool list, policy controls, evaluation results, and known limitations. This reduces review time and makes it easier to compare applications.

## 52.12 Cross-border and external model-call boundaries

External model calls and cross-border routing require clear boundaries. The platform should know which data can leave which region, which providers are allowed, which fields must be masked, and which logs must remain local.

Routing decisions should be visible in Trace. If a request uses an external provider, the record should show provider, region, model, data category, masking result, and policy version.

Fallback routing needs the same control. If the primary model fails and the gateway sends traffic to another provider, the fallback must satisfy the same data and region policy. A reliable gateway should fail closed for restricted data instead of quietly using an unauthorized route.

## 52.13 Compliance samples and continuous regression

Compliance samples should test policy behavior over time. Samples may include sensitive-data masking, deletion requests, cross-border routing, content refusal, human oversight, and provenance records. These samples should run after model, prompt, policy, retrieval, and tool changes.

Continuous regression keeps compliance from becoming a one-time release document. When behavior drifts, the platform should identify the affected requirement and owner.

Regression results should feed governance meetings. If a new model improves quality but fails a masking sample, the release decision should include that trade-off explicitly. If a new policy reduces leakage but creates many false refusals, business owners should review the affected workflows before full rollout.

## Chapter Recap

Compliance becomes practical when requirements map to controls, evidence, owners, and tests. Agent platforms need versioned policy configuration, data-rights workflows, cross-border boundaries, provenance, review inputs, and continuous regression. The goal is not to turn engineers into lawyers, but to make compliance claims traceable in the platform.
