# Chapter 50 Security and Offense-Defense

---

Enterprise Agent security starts from the execution chain. User input, retrieved documents, tool results, memory, screenshots, email, tickets, and generated artifacts may all enter the model context. The Agent may also hold tool permissions that can read data, write records, send messages, or trigger workflow steps. This makes the attack surface wider than a conventional chatbot.

Security work should therefore be connected to platform governance. Prompt Injection, indirect injection, tool overreach, data leakage, red teaming, baseline verification, and incident response need repeatable controls and evidence. This chapter explains how to turn offense-defense work into platform mechanisms instead of one-time testing.

---

## 50.1 Enterprise Agent attack surface

An Agent attack surface includes all inputs that can affect model behavior and all tools that can create side effects. The attacker may not call an internal API directly. They may place instructions in a document, webpage, ticket, email, or field description that later enters retrieval context.

*Table 50-1: Enterprise Agent attack surface. Source: Compiled by this book.*

| Surface | Example | Control |
| --- | --- | --- |
| User input | Direct malicious instruction | Input policy and intent classification |
| Retrieval content | Poisoned document or webpage | Source trust, chunk policy, citation review |
| Tool result | Malicious text returned by API | Tool output sanitization |
| Memory | Stored instruction pollution | Write admission and expiry |
| UI artifact | Fake approval or hidden action | Component allowlist and backend confirmation |

Security review should ask which surfaces can change the model context and which actions can create external effects. The higher the side effect, the stronger the policy gate should be. The attack surface should be reviewed whenever a new input source, retrieval collection, tool, memory type, or UI artifact is added. A document collection that was safe for search may become unsafe when its text is inserted into model context. A read-only API may become risky when the Agent uses its result to decide a write action. This is why the attack surface inventory should be part of release evidence instead of a one-time architecture diagram.

## 50.2 Prompt Injection and indirect injection

Direct Prompt Injection comes from the current user input. Indirect injection comes from external content retrieved or loaded into context. Indirect injection is harder to see because the user may ask a benign question while the malicious instruction sits inside a document. The platform should separate instruction hierarchy from evidence content. Retrieved text is evidence, not policy. Tool returns are data, not instructions. Memory is context, not authority. The model prompt should state these boundaries, but runtime controls must enforce them through tool permissions, policy checks, and review states.

Indirect injection should be tested with realistic content. A malicious sentence hidden in a policy document, spreadsheet comment, email footer, or webpage can be more dangerous than a direct jailbreak because the user may not know it exists. The defense should therefore mark source trust, strip executable instructions from retrieved content when possible, and require tool-layer checks before any side effect.

## 50.3 Tool overreach and data leakage

Tool overreach occurs when an Agent uses a tool outside the user's permission, outside the task boundary, or with unsafe parameters. Data leakage occurs when the Agent exposes data through an answer, artifact, tool call, log, or frontend event.

*Table 50-2: Tool and data risks. Source: Compiled by this book.*

| Risk | Common cause | Mitigation |
| --- | --- | --- |
| Unauthorized read | Tool trusts Agent intent only | Bind tool call to user and tenant permissions |
| Unsafe write | Model chooses side-effect action | Require approval and idempotency key |
| Excessive data return | Tool returns raw records | Return scoped views or references |
| Log leakage | Prompt or result logged in full | Store ids, hashes, and redacted summaries |

Least privilege should be implemented at the tool layer as well as in prompts. A prompt can tell the model to avoid unauthorized access, but the executor must reject unauthorized parameters. Tool responses should also be scoped. If a tool returns more records than the task needs, the model may leak data even when the call was technically authorized. A safer executor returns a task-specific view, aggregates sensitive fields, or provides a reference that the frontend can inspect under permission. This keeps the model from becoming a second access-control layer.

## 50.4 AI Red Teaming method system

AI Red Teaming should become a repeatable engineering process. Test cases should cover direct injection, indirect injection, tool overreach, data exfiltration, jailbreak attempts, memory pollution, unsafe output handling, and approval bypass. Each test should define input, expected refusal or degradation, observed behavior, severity, and fix status. Red teaming should run before release and after policy changes, model changes, retrieval changes, and tool additions. The test set should include successful attacks as regression cases so fixed issues do not reappear.

The red-team process should produce engineering work items. Each finding should map to a failing control: prompt hierarchy, retrieval source policy, tool schema, permission check, output validation, UI rendering, or incident response. Findings that cannot be mapped are hard to fix and tend to become generic warnings.

## 50.5 Security baseline and incident response

A security baseline defines the minimum controls that every Agent application must satisfy: identity binding, tenant isolation, tool permission, retrieval source control, output validation, audit retention, and incident escalation. Incident response defines what the team does after a security event: freeze risky tools, preserve Trace, identify affected runs, notify owners, patch policy, and replay tests.

Incident response should be rehearsed. Teams should know how to disable a tool, revoke a route, quarantine a memory namespace, roll back a policy, and search affected runs. A playbook that has never been tested is unreliable during a real event, especially when the incident involves generated artifacts that may have already been exported or shared.

*Table 50-3: Security baseline controls. Source: Compiled by this book.*

| Control | Evidence |
| --- | --- |
| Tool permission | Policy decision and executor rejection records |
| Retrieval trust | Source metadata and chunk version |
| Output validation | Guardrails result and refusal reason |
| Audit | Run id, trace id, actor, tool call id |
| Incident response | Case record, affected scope, fix verification |

## 50.6 Prompt Injection offense-defense evaluation

Prompt Injection evaluation should include normal tasks, malicious tasks, and mixed tasks where the user question is valid but the retrieved evidence is hostile. The goal is to check whether the Agent follows platform policy while still completing safe parts of the task. Evaluation should measure attack success, false refusal, tool misuse, data exposure, and recovery quality. A defense that refuses every task is safe but unusable. A defense that completes every task without checking evidence source is unsafe.

The evaluation set should contain benign tasks near risky boundaries. For example, a legal-review Agent should be able to analyze a malicious clause as evidence without following it as an instruction. A customer-service Agent should summarize an abusive message without generating abusive content. These cases measure whether the platform can separate analysis from execution.

## 50.7 Offense-defense results entering platform governance

Attack results should not remain in a test report. They should update platform governance: policy rules, tool schemas, retrieval source weights, memory write rules, UI warnings, and evaluation samples. Each accepted fix should include a regression case. When the same issue appears in several applications, the fix belongs in the platform instead of the application. For example, approval bypass should be fixed in Runtime and Human-in-the-loop, while unsafe tool parameters should be fixed in Tool Registry and policy engine.

Governance also needs severity and ownership. A low-severity wording issue may be handled in the application prompt. A high-severity tool overreach belongs to platform policy and executor validation. A compliance-sensitive leakage issue should include compliance and data owners. Without severity and ownership, security findings accumulate without changing the platform.

## 50.8 Continuous verification of security baselines

Security baselines should be verified continuously. Scheduled jobs can replay attack samples against the current model, prompt, retrieval index, and tool schema. Release gates can block deployment when high-severity attacks succeed. Continuous verification needs stable datasets. Each sample should have a version, owner, severity, expected behavior, and last result. Without versioning, teams cannot tell whether improvement came from a model change, policy change, or test drift.

Verification should run in the same path as production where possible. Testing a standalone prompt is useful, but it misses retrieval, memory, tool, gateway, and UI behavior. A high-value regression sample should replay the whole chain: context assembly, model call, tool decision, Guardrails, output validation, and Trace recording.

## 50.9 Security layering for tool write operations

Write operations need stronger controls than read operations. The platform should layer intent classification, permission check, parameter validation, approval, idempotency, execution, audit, and compensation. The model proposes the action, but the platform decides whether it can run. The UI should reflect this layering. Users should see that a write action is a proposal until approval and execution complete. If an approval expires or a compensation action runs, the UI should show that state. Security controls that are invisible to users often get misinterpreted as product bugs.

*Table 50-4: Security layers for write operations. Source: Compiled by this book.*

| Layer | Purpose |
| --- | --- |
| Intent classification | Identify side-effect actions |
| Permission check | Bind action to user and tenant |
| Parameter validation | Reject unsafe or incomplete parameters |
| Approval | Pause high-risk actions |
| Idempotency | Prevent duplicate writes |
| Compensation | Define recovery when execution fails |

## 50.10 Version management for attack samples

Attack samples are platform assets. They should be versioned like evaluation data. A sample should record attack type, source, target capability, severity, expected behavior, owner, and fix history. When a policy changes, the platform should rerun the relevant sample set. Version management also helps communicate risk. Instead of saying "security has improved," the team can say which attack classes are blocked, which remain under review, and which applications are affected.

Attack samples should be treated carefully because they may contain working exploit instructions or sensitive internal details. Access should be limited, and exported reports should summarize the class of attack without publishing reusable payloads. The sample repository should preserve enough detail for regression while avoiding unnecessary spread of attack content.

## 50.11 Security release gates and ownership

Security governance needs a release gate. Every new tool, data source, system prompt change, model upgrade, or external user group can change the attack surface. The gate should require a threat model, tool risk level, data classification result, red-team regression result, policy version, and incident-response plan. If a high-severity sample fails, the release record should show where the failure happened, what temporary control is in place, when the waiver expires, and who will rerun the test. This turns security review into evidence that the release system can inspect.

The gate should also separate platform ownership from business ownership. The platform team owns Runtime, Tool Registry, Policy Engine, Trace, approval state, and rollback mechanisms. The security team owns threat models, red-team samples, severity assignment, and incident review. The data team owns asset classification, field policy, and exfiltration assessment. The business team owns scenario admission, risk acceptance, and user notice. SRE owns freeze actions, blast-radius control, and recovery windows. Clear ownership keeps an incident from being attributed only to model behavior when the actual failure may sit in a tool system, data policy, frontend state, or missing audit record.

Attack samples should be managed as platform assets. Each sample should store its source, attack type, applicable application, target tool, severity, expected blocking point, last result, and owner. Public samples are useful for general controls such as jailbreaks, system-prompt exposure, and hidden instructions. Internal samples are better for business boundaries, such as over-permission customer lookup, approval bypass for contract sending, or exporting an internal report to the wrong recipient. Both groups need version management. Model upgrades should run samples tied to model behavior. Tool additions should run privilege and write-action samples. Knowledge-base imports should run indirect-injection samples. New frontend components should run fake-button and unsafe-rendering samples.

Incident response should be part of the gate as well. The platform should be able to search affected scope by `run_id`, `trace_id`, user, tenant, tool, data asset, model version, and policy version. It should be able to freeze the related Agent, revoke scoped tokens, take a tool offline, isolate a Memory namespace, withdraw or mark generated artifacts, and convert the fix into a regression sample. The speed of incident handling depends on whether this evidence exists before the incident. Incomplete runtime records force the team to spend time reconstructing scope while the fix waits.

Security policy needs regular calibration. Too many false positives will push business teams around the platform. Too many misses make the platform unsafe to operate. Guardrails, policy engines, and approval workflows should record false positives, misses, user appeals, and final decisions, then replay those cases during policy review. The first platform version can start with a narrow set of high-risk gates: high-risk tools require a Policy Engine decision record, external documents require source labels, data export requires field policy and approval evidence, and failed red-team samples require an owner and retest date. The scope can be small at first, but the evidence chain should be complete.

## 50.12 Security incident review and sample return

Agent security incident review should decompose the attack path into platform components. Prompt Injection may enter context through web pages, document fragments, uploaded files, external tool results, or memory. An unauthorized write operation may come from broad tool descriptions, a Planner that skipped approval, a Runtime state machine without suspend semantics, or a frontend button without server-side confirmation. Data leakage may come from retrieval filtering, log masking, Trace display, or export policy. Review should record entry point, carrier, affected tool, policy judgment, human intervention, output destination, and user-visible result instead of reducing every incident to model induction.

Attack samples should return to evaluation and release gates. Injection text, unauthorized parameters, sensitive-field combinations, and task descriptions that bypass approval should enter the security regression set. The sample should preserve source context, user role, tool permission, expected refusal behavior, and correct recovery action. Chapter 39 Eval can then run security samples as regular release gates. Chapter 23 Tool Registry can inspect tool schema. Chapter 30 HITL can verify approval state. Chapter 38 Trace can check audit evidence. If security work stops at a post-incident report, the next release can repeat the same error pattern.

Incident review also needs repair ownership. Model-side fixes may adjust system prompts and refusal policy. Tool-side fixes should narrow parameters and permissions. Runtime needs state checks. Frontend needs confirmation for dangerous actions. Security teams update attack samples. Business owners decide which functions need degradation or suspension. Each repair should have a verification method and observation window. Security then becomes engineering governance: every risk has an entry point, detection, block, recovery action, and owner, and later versions can show that the same failure class is covered.

## 50.13 Runtime acceptance for security control points

Security control points need runtime acceptance after launch. Prompt Injection blocking, tool-permission checks, output masking, approval for dangerous actions, data-export limits, and external-link controls cannot remain only in design documents. The platform should prove that controls trigger, record evidence, and enter review during real Runs. When a security sample passes, reviewers should see input source, policy version, blocking point, user-visible message, whether the tool was stopped, and whether Trace preserved evidence.

Acceptance should also cover false positives. When policy is too strict, business users move tasks outside the platform or into personal tools. When policy is too loose, risk enters production actions. Review should inspect whether blocked tasks have an appeal path, whether human review can adjust policy, and whether policy changes enter version records. Security teams should not look only at block rate. Business teams should not look only at pass rate. Both sides need samples, evidence, and final disposition.

The first version can start with a small set of high-risk controls: external-document injection, sensitive-field export, high-risk write tools, approval bypass, and log leakage. Each control point should have samples, policy, Trace, owner, and retest date. A narrow scope is acceptable when evidence is complete, because the same pattern can later expand to more scenarios.

## 50.14 Security samples in release gates

Security samples should enter release gates instead of staying in a security-team report. Whenever a new tool, external knowledge source, model route, frontend component, or export capability is added, the platform should replay the relevant samples. Each sample should state the attack entry point, target capability, expected blocking point, allowed degradation, and owner. The release system can then locate the failed layer: retrieval source, context assembly, model call, tool policy, Runtime state, frontend rendering, or audit record.

The gate can allow limited waivers, but each waiver needs scope and expiry. A low-risk internal scenario may be released temporarily, but the record should state affected tenants, compensating controls, retest date, and owner. Without these fields, waivers become permanent exceptions and the attack surface becomes harder to manage. Security release records make the trade-off between business speed and risk control visible, and they give the next review something concrete to inspect.

Security samples should also share execution machinery with Chapter 39 Eval. Red-team samples, false-positive samples, privilege samples, and export samples can all run as evaluation sets, with results reviewed by security and compliance owners. This puts security capability into the normal engineering rhythm instead of treating it as a late manual test before launch.

## 50.15 Lifecycle of security exceptions

Enterprise Agent security governance needs an exception mechanism. Business work will create temporary allowances: an internal user may need to test a high-risk tool, a project may need restricted knowledge access, or a release may need compensating controls before a patch is finished. Exceptions can exist, but they need a lifecycle. An exception without expiry, scope, and owner slowly becomes a new default rule.

An exception record should state request reason, affected users, affected tools, data scope, compensating controls, approver, expiry time, and retest requirement. When it expires, the platform should remind the owner to review it instead of leaving the exception in configuration. If an exception contributes to a security incident, the incident sample should enter red-team and regression sets so the same risk does not return through another entry point.

A first version can require exception records for high-risk write tools, external export, cross-tenant data access, and external knowledge sources. This keeps security governance from losing boundaries under business pressure and helps the platform team discover exceptions before an incident.

## 50.16 Security evidence during rollout and rollback

Security governance should be visible during rollout and design review. When a new Agent, tool, model route, or knowledge source enters gray release, the rollout record should show which security controls are active, which samples were replayed, which tenants are included, and which waivers are in effect. A gray release without this evidence gives the team only traffic metrics and user feedback. It cannot show whether the platform is still enforcing the intended boundary.

Rollback needs the same evidence. If a security sample fails after release, the platform should know whether to roll back a policy version, disable a tool, freeze a tenant route, switch a model route, isolate a memory namespace, or withdraw an artifact. These actions have different business effects. A blanket rollback may stop safe work; a narrow rollback may leave risk active. The decision should use Trace, policy version, tool version, data category, and affected Run records.

Security evidence should also help communication. When a release is paused, business owners need to know what capability is affected, what users can still do, and what recovery path exists. Security teams need the failed sample, expected control point, and fix owner. Platform teams need the exact configuration to revert. A useful release record therefore combines operational data with governance data: scope, control, failure, action, owner, and retest date.

For a first version, the platform can attach a small security evidence block to every high-risk release. It records the red-team sample set, policy version, tool risk level, approval status, exception ids, and rollback action. This block is not paperwork. It is the material that lets teams move quickly without losing track of the safety boundary.

## 50.17 Online observation and response drills for security controls

After security controls are released, the platform needs evidence that each control still works in production. Input filtering, retrieval isolation, tool authorization, output masking, export approval, audit retention, and anomaly alerting should all emit observable events. The event does not need to store full sensitive content, but it should record the control point, policy version, risk category, action result, related Run, tenant scope, and review state. Security teams then see a reviewable control chain instead of scattered application logs. If a high-risk tool suddenly produces a surge of denials, the team can distinguish changing user behavior, policy false positives, new attack samples, and a tool schema that exposed a risky parameter.

Response drills should be designed around realistic attack paths. The platform can regularly replay samples for Prompt Injection, unauthorized tool use, sensitive-field export, cross-tenant retrieval, malicious file upload, and frontend artifact injection. The drill should inspect more than the final block decision. It should verify whether the user message is understandable, the Run state is correct, audit evidence is complete, the business owner receives notification, and any exception is bounded. A sample that is blocked without Trace still leaves incident review weak. A sample that enters human review without expiry still leaves risk sitting in the queue.

Security response also needs degradation paths. When a write tool becomes risky, the platform can freeze writes while preserving read-only access. When a knowledge source appears polluted, the platform can isolate the index while allowing users to view approved material. When a model route becomes vulnerable to a prompt pattern, the platform can switch to a conservative route or require human confirmation. The response goal is to keep low-risk work available while the risky path is being isolated. For a first version, the platform can define a small set of response actions: disable a tool, isolate a knowledge source, freeze a tenant, withdraw an artifact, require human review, and add red-team samples. Each action should have an owner, trigger condition, recovery condition, and retest requirement.

This operating discipline makes security a daily platform function. It also gives business teams clearer expectations during incidents. A release pause should not sound like a vague security objection; it should name the failed sample, the expected control point, the affected capability, and the available fallback path. Engineering teams can then repair the right link, while security teams keep the boundary testable. Over time, the security baseline becomes a set of versioned controls, samples, and response actions that can be rehearsed before a real incident.

## 50.18 Operating maintenance for security regression samples

Security samples should not appear only during red-team exercises. Every online block, human review, unauthorized attempt, export denial, Prompt Injection variant, and false-positive appeal can become a regression sample. The sample record should include entry point, attack or false-positive intent, target capability, context source, expected control point, actual action, human ruling, related policy version, and fix state. With these fields, the platform can distinguish risks already covered by policy, risks that need new rules, and allowed business cases that need clearer messages. Without this classification, security teams keep revisiting similar samples and platform teams cannot tell which repairs have entered release gates.

Sample maintenance needs quality control. Stale samples, duplicates, and samples without enough context slow releases and push policies toward overblocking. The platform can use a small lifecycle: candidate, confirmed, fixed, under observation, and retired. Candidate samples come from online events and human feedback. Confirmed samples enter release gates. Fixed samples guard against regression. Observation samples track unstable risk. Retired samples keep their retirement reason but no longer block release. After each policy or tool change, the owner should review affected samples instead of letting the sample library grow without curation.

Regression samples should include false positives. Enterprise security governance cannot measure success only by blocking more risky requests. If ordinary business analysis, compliance lookup, or report export is blocked incorrectly, users will move work outside the platform and weaken the security boundary. A false-positive sample should record user role, task purpose, blocked field, correct handling, and acceptable user message. Release gates should run risk samples and false-positive samples together, so the team can judge production suitability instead of treating stricter behavior as success.

A first version can run the security sample library on the same execution framework as Chapter 39 evaluation. The difference is ownership of the ruling: quality samples are judged by business and data owners, while security samples are judged by security, compliance, and platform owners. This keeps security governance out of a separate manual process. It becomes replayable, measurable, and accountable, using the same engineering rhythm as other platform quality controls.

## 50.19 Business recovery after security response

Security response should not stop at the blocking action. After a tool is frozen, a knowledge source is isolated, a model route is switched, or an artifact is withdrawn, the business team still needs a path back to controlled operation. The platform should define recovery conditions for each response type: whether samples have passed replay, whether the policy version has shipped, whether affected Runs have been marked, whether users received an explanation, and whether an approved fallback path exists. Without recovery conditions, a security response can become a long-term shutdown, and business teams may move work into informal channels.

Recovery should preserve evidence continuity. Before a write tool regains permission, reviewers should see the fix record, regression samples, approving owner, and retest result. Before an isolated knowledge source returns to retrieval, reviewers should see the pollution sample, cleanup scope, and index rebuild record. Before a withdrawn artifact is republished, reviewers should see the withdrawal reason, revised version, and review conclusion. Security, platform, and business owners can then decide from the same material, instead of relying on separate local notes.

A first version can put recovery actions into the security incident ledger. The ledger records response action, recovery action, owner, retest sample, and user communication status. Response and recovery should appear as a pair. This keeps security governance from becoming a one-way brake and gives business processes a way back into controlled operation.

Business recovery also needs a user-facing communication path. During a security response, users need to know which part of the task is suspended, which read-only or masked mode is still available, and when the next review will happen. A vague "security restriction" message often sends users to side channels, which reduces observability and makes the incident harder to contain. A better recovery design gives each response action an allowed fallback. A frozen write tool may still allow draft generation. An isolated knowledge source may still allow approved reference material. A blocked export may continue as an internal preview. These fallback paths should be designed before the incident, because the team will not have time to invent them while scope, evidence, and ownership are still being reconstructed.

The recovery record should connect user communication with technical evidence. If a report has been withdrawn, the record should state whether users saw the old report, whether a corrected artifact exists, which Trace entries justify the withdrawal, and which downstream systems need notification. If a tool call was blocked after execution, the record should show whether the side effect was compensated, whether a duplicate action may occur during retry, and whether the affected user received a clear status. Security teams care about containment, but business owners also need continuity. The platform has to support both views through the same incident ledger.

Recovery is also where many security programs lose their evidence chain. A team may fix a prompt, change a tool schema, update a classifier threshold, and reopen the scenario, but forget to link those changes to the original failed sample. Later, when the same class of risk appears again, the review team cannot tell whether the original repair was incomplete or whether a new path has opened. Each recovery should therefore produce a small closure package: failed sample, root cause, changed component, retest result, affected Runs, user communication, remaining exception, and next review date. The package should be short enough to maintain during real incidents, yet structured enough for later release gates.

For production Agent systems, recovery should also consider shared dependencies. A Prompt Injection incident in one scenario may involve a shared RAG ingestion path. A tool overreach incident may involve a shared tool schema used by several Agents. A leaked report may involve a shared export component. If recovery closes only the visible scenario, other routes can keep the same exposure. The incident ledger should let the platform search by component, policy, tool, data source, and frontend artifact. The response team can then decide whether the fix is local, shared, or platform-wide.

The most useful metric after a security incident is not the number of blocked requests. It is whether the same risk class can be replayed, whether the affected scope can be explained, and whether the platform can reopen service through defined controls. A first-version platform can start with a modest recovery standard: no high-risk capability returns to normal operation until the failed sample passes, the owner signs off, affected artifacts are marked, and the user communication path is complete. This standard gives security response an ending condition and gives business teams a visible route back to operation.

Security governance should also learn from near misses. A near miss occurs when the platform blocks the action, but the review shows that another layer almost failed: a tool schema allowed a risky parameter, a frontend component showed unsafe content before the final refusal, or Trace missed the evidence needed for review. Near misses are useful because they reveal weak links before harm occurs. Treating them as successes hides future risk. The sample library should include near misses with lower severity but clear repair actions, so they can influence tool design, policy routing, and frontend safety.

The final operating habit is to keep security decisions close to product decisions. When a capability is admitted, expanded, paused, or retired, security evidence should be part of the same release and operating material. Separate security spreadsheets become stale quickly. If security status lives in the same platform records as tool versions, model routes, Run states, evaluation results, and business owners, later teams can understand why a decision was made and whether the conditions still hold. That is the point of treating security as platform engineering: risk control becomes reviewable system behavior, not memory held by a few reviewers.

## 50.20 Near misses and shared-dependency review

Security governance should record near misses. A near miss means the platform eventually blocked the risky action, but review shows that an earlier control point nearly failed: a tool schema allowed a dangerous parameter, a frontend component briefly exposed sensitive content before refusal, retrieval sent polluted material into candidates, or Trace missed fields needed for review. If the team treats such cases only as successful blocks, the weak link remains and the next attempt may enter through another path.

Near-miss samples should enter the security sample library. Their severity can be lower than a real incident, but each one still needs a repair owner and retest condition. The sample records trigger path, weakened control point, final blocking point, potential impact, component to strengthen, and replay method. The security library then stores signals of platform fragility along with incidents that already caused impact. For enterprise Agents, these signals often matter because they point to shared capabilities.

Shared-dependency review is equally important. A Prompt Injection incident may expose a common RAG ingestion path. A tool overreach incident may come from a shared tool schema used by several Agents. A report leak may involve the common export component. If recovery closes only the visible scenario, other routes can keep the same exposure. The security incident ledger should support search by component, policy, tool, data source, frontend artifact, and model route. Response teams can then decide whether a fix is local, shared, or platform wide.

A first version can add two fields to the incident review template: whether a near-miss control point existed, and whether a shared dependency was involved. If either answer is yes, the repair cannot stop at the current Agent prompt or configuration. It has to return to the shared component contract, samples, and release gate. Security governance then shifts from after-the-fact blocking toward earlier platform improvement.

## 50.21 Business review for security exceptions

Security exceptions become long-term bypasses when they lack review. A tenant may temporarily receive relaxed export limits, a tool may temporarily allow writes, or a knowledge source may temporarily skip isolation, and those settings can remain after the incident ends. The platform should treat exceptions as lifecycle objects with reason, scope, expiry, approver, compensating control, and retest samples.

Business review should ask whether the exception is still needed. If the exception supports a high-value workflow, the platform may keep it with narrower scope. If it exists only to work around product defects, product or permission design should be repaired. If it has no recent use, it should retire and configuration should be removed. Security teams should not decide every exception alone. Business owners need to state value and risk acceptance.

A first version can publish a monthly security exception list. The list is sorted by tenant, tool, data domain, and expiry time, and each owner chooses keep, narrow, or delete. Once exception review enters release gates, security policy remains explainable instead of accumulating into a configuration layer that nobody wants to touch.

## 50.22 Business exception management for safety policy

When business exception management for safety policy reaches production, the platform needs a shared evidence standard for exception reason, scope, approver, expiry time, compensating control, retest sample, and withdrawal path. This standard is not paperwork for its own sake. It lets business, platform, data, security, and operations teams discuss the same facts. Without this material, incident review depends on memory and personal judgment. With it, the team can see which inputs were valid, which actions executed, which artifacts can still be used, and which results need correction or withdrawal.

This evidence should connect to Chapter 51 on Guardrails, Chapter 52 on compliance, and Chapter 53 on governance. The upstream chapters provide the capability base, downstream chapters consume the runtime result, and this chapter explains how the middle layer is verified. If a capability looks complete inside one chapter but cannot enter Trace, Eval, release records, or the compliance evidence package, the production system still has a break in the chain. Readers should treat cross-chapter interfaces as engineering contracts, not as a reading order.

Common risks include temporary approval becoming a permanent bypass, exception scope expanding, and business owners not knowing remaining risk. A successful demo rarely exposes these problems because demo samples are usually clean, short, and direct. Real business traffic brings stale data, abnormal input, permission changes, user withdrawal, budget limits, and long-running state. If the platform does not turn those situations into samples and ledgers, later scenarios will hit the same class of issues again.

Temporary approval becoming a permanent bypass should be turned into a tracked review item when it appears repeatedly. The operating record should at least state owner, version, sample, affected scope, action, and review time. It does not need to become a long process report, but it must be clear enough for a later maintainer to understand the decision. For high-risk capability, the record should also state which conditions allow wider use and which failures require degradation or withdrawal.

A first version can build this habit in a few representative scenarios. It is better to make high-traffic, high-risk, externally visible paths solid first, then copy the sample, ledger, and review method to related capabilities in other chapters. This makes the chapter read like engineering guidance: it explains how the capability is integrated, validated, operated, and retired.

## 50.23 Sample-based regression for security incidents

Security incidents should become regression samples. Unauthorized tool calls, prompt injection, sensitive-field leakage, unsafe external links, and high-risk write actions should not remain only in incident reports. The platform should turn each incident into repeatable samples for Guardrails, tool permission, model routing, frontend messages, and human approval.

Sample-based regression keeps security repair from stopping at a local patch. If the incident came from tool permission, the sample should cover the same boundary. If it came from retrieval contamination, the sample should cover knowledge source and citation. If it came from misleading UI, the sample should cover user-visible state. A first version can require a regression sample for every major security incident and retest it before the next release.

## Chapter Recap

Enterprise Agent security is an execution-chain problem. The platform must control inputs, retrieval content, memory, tools, UI artifacts, and logs. Prompt Injection and tool overreach require runtime enforcement, not prompt wording alone. Red teaming should feed policy, tool, retrieval, memory, UI, and evaluation changes. Security baselines need continuous verification and versioned attack samples.
