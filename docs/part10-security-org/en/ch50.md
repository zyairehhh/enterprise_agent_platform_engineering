# Chapter 50 Security and Offense-Defense

---

Enterprise Agent security starts from the execution chain. User input, retrieved documents, tool results, memory, screenshots, email, tickets, and generated artifacts may all enter the model context. The Agent may also hold tool permissions that can read data, write records, send messages, or trigger workflow steps. This makes the attack surface wider than a conventional chatbot.

Security work should therefore be connected to platform governance. Prompt Injection, indirect injection, tool overreach, data leakage, red teaming, baseline verification, and incident response need repeatable controls and evidence. This chapter explains how to turn offense-defense work into platform mechanisms instead of one-time testing.

---

## 50.1 Enterprise Agent attack surface

An Agent attack surface includes all inputs that can affect model behavior and all tools that can create side effects. The attacker may not call an internal API directly. They may place instructions in a document, webpage, ticket, email, or field description that later enters retrieval context.

*Table 50-1: Enterprise Agent attack surface. Source: Compiled by this book.*

| Surface | Example | Control |
| --- | --- | --- |
| User input | Direct malicious instruction | Input policy and intent classification |
| Retrieval content | Poisoned document or webpage | Source trust, chunk policy, citation review |
| Tool result | Malicious text returned by API | Tool output sanitization |
| Memory | Stored instruction pollution | Write admission and expiry |
| UI artifact | Fake approval or hidden action | Component allowlist and backend confirmation |

Security review should ask which surfaces can change the model context and which actions can create external effects. The higher the side effect, the stronger the policy gate should be.

The attack surface should be reviewed whenever a new input source, retrieval collection, tool, memory type, or UI artifact is added. A document collection that was safe for search may become unsafe when its text is inserted into model context. A read-only API may become risky when the Agent uses its result to decide a write action. This is why the attack surface inventory should be part of release evidence rather than a one-time architecture diagram.

## 50.2 Prompt Injection and indirect injection

Direct Prompt Injection comes from the current user input. Indirect injection comes from external content retrieved or loaded into context. Indirect injection is harder to see because the user may ask a benign question while the malicious instruction sits inside a document.

The platform should separate instruction hierarchy from evidence content. Retrieved text is evidence, not policy. Tool returns are data, not instructions. Memory is context, not authority. The model prompt should state these boundaries, but runtime controls must enforce them through tool permissions, policy checks, and review states.

Indirect injection should be tested with realistic content. A malicious sentence hidden in a policy document, spreadsheet comment, email footer, or webpage can be more dangerous than a direct jailbreak because the user may not know it exists. The defense should therefore mark source trust, strip executable instructions from retrieved content when possible, and require tool-layer checks before any side effect.

## 50.3 Tool overreach and data leakage

Tool overreach occurs when an Agent uses a tool outside the user's permission, outside the task boundary, or with unsafe parameters. Data leakage occurs when the Agent exposes data through an answer, artifact, tool call, log, or frontend event.

*Table 50-2: Tool and data risks. Source: Compiled by this book.*

| Risk | Common cause | Mitigation |
| --- | --- | --- |
| Unauthorized read | Tool trusts Agent intent only | Bind tool call to user and tenant permissions |
| Unsafe write | Model chooses side-effect action | Require approval and idempotency key |
| Excessive data return | Tool returns raw records | Return scoped views or references |
| Log leakage | Prompt or result logged in full | Store ids, hashes, and redacted summaries |

Least privilege should be implemented at the tool layer as well as in prompts. A prompt can tell the model to avoid unauthorized access, but the executor must reject unauthorized parameters.

Tool responses should also be scoped. If a tool returns more records than the task needs, the model may leak data even when the call was technically authorized. A safer executor returns a task-specific view, aggregates sensitive fields, or provides a reference that the frontend can inspect under permission. This keeps the model from becoming a second access-control layer.

## 50.4 AI Red Teaming method system

AI Red Teaming should become a repeatable engineering process. Test cases should cover direct injection, indirect injection, tool overreach, data exfiltration, jailbreak attempts, memory pollution, unsafe output handling, and approval bypass. Each test should define input, expected refusal or degradation, observed behavior, severity, and fix status.

Red teaming should run before release and after policy changes, model changes, retrieval changes, and tool additions. The test set should include successful attacks as regression cases so fixed issues do not reappear.

The red-team process should produce engineering work items. Each finding should map to a failing control: prompt hierarchy, retrieval source policy, tool schema, permission check, output validation, UI rendering, or incident response. Findings that cannot be mapped are hard to fix and tend to become generic warnings.

## 50.5 Security baseline and incident response

A security baseline defines the minimum controls that every Agent application must satisfy: identity binding, tenant isolation, tool permission, retrieval source control, output validation, audit retention, and incident escalation. Incident response defines what the team does after a security event: freeze risky tools, preserve Trace, identify affected runs, notify owners, patch policy, and replay tests.

Incident response should be rehearsed. Teams should know how to disable a tool, revoke a route, quarantine a memory namespace, roll back a policy, and search affected runs. A playbook that has never been tested is unreliable during a real event, especially when the incident involves generated artifacts that may have already been exported or shared.

*Table 50-3: Security baseline controls. Source: Compiled by this book.*

| Control | Evidence |
| --- | --- |
| Tool permission | Policy decision and executor rejection records |
| Retrieval trust | Source metadata and chunk version |
| Output validation | Guardrails result and refusal reason |
| Audit | Run id, trace id, actor, tool call id |
| Incident response | Case record, affected scope, fix verification |

## 50.6 Prompt Injection offense-defense evaluation

Prompt Injection evaluation should include normal tasks, malicious tasks, and mixed tasks where the user question is valid but the retrieved evidence is hostile. The goal is to check whether the Agent follows platform policy while still completing safe parts of the task.

Evaluation should measure attack success, false refusal, tool misuse, data exposure, and recovery quality. A defense that refuses every task is safe but unusable. A defense that completes every task without checking evidence source is unsafe.

The evaluation set should contain benign tasks near risky boundaries. For example, a legal-review Agent should be able to analyze a malicious clause as evidence without following it as an instruction. A customer-service Agent should summarize an abusive message without generating abusive content. These cases measure whether the platform can separate analysis from execution.

## 50.7 Offense-defense results entering platform governance

Attack results should not remain in a test report. They should update platform governance: policy rules, tool schemas, retrieval source weights, memory write rules, UI warnings, and evaluation samples. Each accepted fix should include a regression case.

When the same issue appears in several applications, the fix belongs in the platform rather than the application. For example, approval bypass should be fixed in Runtime and Human-in-the-loop, while unsafe tool parameters should be fixed in Tool Registry and policy engine.

Governance also needs severity and ownership. A low-severity wording issue may be handled in the application prompt. A high-severity tool overreach belongs to platform policy and executor validation. A compliance-sensitive leakage issue should include compliance and data owners. Without severity and ownership, security findings accumulate without changing the platform.

## 50.8 Continuous verification of security baselines

Security baselines should be verified continuously. Scheduled jobs can replay attack samples against the current model, prompt, retrieval index, and tool schema. Release gates can block deployment when high-severity attacks succeed.

Continuous verification needs stable datasets. Each sample should have a version, owner, severity, expected behavior, and last result. Without versioning, teams cannot tell whether improvement came from a model change, policy change, or test drift.

Verification should run in the same path as production where possible. Testing a standalone prompt is useful, but it misses retrieval, memory, tool, gateway, and UI behavior. A high-value regression sample should replay the whole chain: context assembly, model call, tool decision, Guardrails, output validation, and Trace recording.

## 50.9 Security layering for tool write operations

Write operations need stronger controls than read operations. The platform should layer intent classification, permission check, parameter validation, approval, idempotency, execution, audit, and compensation. The model proposes the action, but the platform decides whether it can run.

The UI should reflect this layering. Users should see that a write action is a proposal until approval and execution complete. If an approval expires or a compensation action runs, the UI should show that state. Security controls that are invisible to users often get misinterpreted as product bugs.

*Table 50-4: Security layers for write operations. Source: Compiled by this book.*

| Layer | Purpose |
| --- | --- |
| Intent classification | Identify side-effect actions |
| Permission check | Bind action to user and tenant |
| Parameter validation | Reject unsafe or incomplete parameters |
| Approval | Pause high-risk actions |
| Idempotency | Prevent duplicate writes |
| Compensation | Define recovery when execution fails |

## 50.10 Version management for attack samples

Attack samples are platform assets. They should be versioned like evaluation data. A sample should record attack type, source, target capability, severity, expected behavior, owner, and fix history. When a policy changes, the platform should rerun the relevant sample set.

Version management also helps communicate risk. Instead of saying "security has improved," the team can say which attack classes are blocked, which remain under review, and which applications are affected.

Attack samples should be treated carefully because they may contain working exploit instructions or sensitive internal details. Access should be limited, and exported reports should summarize the class of attack without publishing reusable payloads. The sample repository should preserve enough detail for regression while avoiding unnecessary spread of attack content.

## Chapter Recap

Enterprise Agent security is an execution-chain problem. The platform must control inputs, retrieval content, memory, tools, UI artifacts, and logs. Prompt Injection and tool overreach require runtime enforcement, not prompt wording alone. Red teaming should feed policy, tool, retrieval, memory, UI, and evaluation changes. Security baselines need continuous verification and versioned attack samples.
