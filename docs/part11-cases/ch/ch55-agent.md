# 第55章 案例复审与平台化收束

---

案例写完以后，还不能直接进入正式版本。企业 Agent 案例牵涉工具权限、业务口径、人工确认和审计证据，一处边界写错，就可能让读者误判系统能自动完成的范围。销售 Agent 看起来只是给客户生成报价草稿，实际会碰到折扣审批、库存承诺和合同责任；财务 Agent 看起来只是解释指标波动，实际会碰到口径版本、凭证异常和正式披露；法务 Agent 看起来只是抽取条款，实际会碰到法律判断、责任意见和签署建议。案例写作如果只展示“系统能给出结果”，就会把这些责任压扁成一个功能演示。

复审的第一项工作，是检查证据是否支撑结论。一个案例可以说系统生成了报价草稿，但需要说明草稿来自哪些数据、折扣规则从哪里读取、哪些动作停在审批前；可以说系统辅助财务分析，但要说明指标口径、异常定位和复核记录；可以说系统帮助法务审阅合同，但要说明条款证据、风险标签和最终签署责任。缺少这些材料时，书稿不能靠“保留人工确认”这类笼统说法带过，因为读者不知道确认发生在哪一步，也不知道系统失败后谁负责。

复审的第二项工作，是把案例重新收束到平台能力上。销售、财务、法务看起来是三类业务，底层却经常复用同一批能力：Runtime 负责运行态，Registry 管住工具，语义层管住口径，Guardrails 和审批链管住风险，Trace 负责事后复盘。一个案例如果只能说明某个应用做了一个功能，却看不出这些平台能力如何协作，就很难成为本书的正式案例。它可以是产品材料，也可以是内部复盘，但还不能证明企业级 AI 原生业务系统的设计。

这里还有一个容易忽略的问题：案例会反过来暴露主体章节的不足。多个案例都卡在审批超时，说明第30章的人机协同要补恢复策略；多个 DataAgent 案例都卡在指标口径，说明第33章和第34章要补版本和编译契约；多个工单案例都解释不了失败原因，说明第38章 Trace 和第39章评测样本还不够。案例用于检查前文架构能否落到真实流程，不能当作全书末尾的装饰。

中文第一版不编造具体企业故事、上线数据或收益结论。本章给出的是复审口径和平台化收束方法：先审证据，再审风险边界，然后把单点场景沉淀为可复用的平台能力。读者读完本章，应能判断一个候选案例能否发布、该降级成什么形式，以及它暴露的问题应该回写到哪些主体章节。

---

## 55.1 销售、财务与法务场景的共同边界

销售、财务和法务场景通常比普通知识问答风险更高，因为它们会影响报价、结算、合规判断或合同责任。案例补写要先交代任务边界。“保留人工确认”这句话本身不够，正文还要写清系统在哪一步停下、谁接手、确认什么、确认结果如何回到系统。

复审这些案例时，最容易出问题的是动词。作者写“生成报价”，读者可能理解为正式对外报价；作者写“完成财务分析”，读者可能理解为系统已经给出可披露结论；作者写“审阅合同”，读者可能理解为系统承担了法律判断。更稳的写法是把动作拆开：系统生成草稿、列出依据、标出风险、提交审批、等待确认、记录结果。每个动词都对应一个系统状态或人的责任，案例边界就清楚得多。

表 55-1 保留了三类场景的自动化范围和人工确认范围。它用来提醒作者，不是为了把业务复杂性压成三行：只要场景涉及外部承诺、正式披露或法律责任，系统输出就需要被写成“待确认材料”，不能写成已经生效的业务动作。

*表55-1：销售、财务与法务场景中可自动化部分与必须人工确认部分的边界。来源：本书整理。*

| 场景 | 可以自动化的部分 | 必须保留人工确认的部分 |
| --- | --- | --- |
| 销售 Agent | 汇总客户背景、检索历史报价、生成报价草稿 | 正式报价、折扣例外、客户承诺 |
| 财务 Agent | 解释指标变动、生成核对清单、定位异常凭证 | 财务结论确认、口径变更、正式披露 |
| 法务 Agent | 抽取条款、识别风险、生成审阅意见草稿 | 法律判断、合同签署建议、责任承诺 |

这三类场景的共同点，是系统帮助人更快接近判断，但责任仍由人和组织承担。复审时尤其要警惕两种写法：把“生成草稿”写成“完成业务动作”，把“辅助判断”写成“自动决策”。这两种表达都会把 Agent 的边界推过头。

把边界写清楚，并不会削弱案例价值。相反，读者会更容易相信这个系统接近生产，因为生产系统本来就要知道哪些动作可以自动执行，哪些动作需要等待，哪些动作应该拒绝。一个能停下来的 Agent，比一个什么都声称能做的 Agent 更适合企业场景。

### 55.1.1 边界要落到状态和动作

案例中写“需要人工确认”仍然太抽象。审稿者应该追问：确认发生在哪个状态？系统给人什么材料？人确认以后，系统继续做什么？如果人拒绝或超时，系统如何处理？

以销售报价为例，比较稳妥的状态设计是：Agent 先生成报价草稿和风险说明；若折扣低于常规阈值，系统只记录草稿；若折扣超过阈值，Run 状态进入 `waiting_approval`；经理确认后，系统才允许生成正式报价邮件。这个状态变化要能在 Trace 中看到，而不是只写在设计文档里。

*表55-2：高风险业务案例中的状态边界。来源：本书整理。*

| 状态 | 系统动作 | 人的动作 | 复审重点 |
| --- | --- | --- | --- |
| draft_ready | 生成草稿、列出证据和风险 | 阅读并修改 | 草稿是否明确标注未生效 |
| waiting_approval | 暂停高风险动作、发送审批请求 | 确认、驳回或要求补充 | 审批人是否有权限 |
| approved | 继续执行被批准的动作 | 承担业务确认责任 | 执行动作是否与审批一致 |
| rejected | 停止执行并记录原因 | 给出拒绝原因 | 系统是否会绕过拒绝 |
| escalated | 转人工或转工单 | 接管任务 | 上下文是否完整移交 |

### 55.1.2 复审要区分“能力可演示”和“案例可发布”

功能演示跑通，不等于案例可以发布。演示通常只证明系统在理想输入下能产出结果；案例还要说明它面对边界输入、权限不足、证据缺失和工具失败时怎样处理。参考书的实战项目能成立，是因为项目、代码和 smoke test 互相支撑。本书第一版暂不补实战项目，因此案例复审更要把证据要求写清楚。

正式案例至少要能回答三个追问。第一，用户换一个不完整输入时，系统如何澄清或降级；第二，工具返回错误或权限不足时，前端和 Trace 怎样记录；第三，人工复核发现结果不可靠时，系统如何保留反馈并进入下一轮评估。演示视频通常不会展示这些路径，但生产案例必须展示或说明这些路径。

*表55-3：功能演示、内部复盘与正式案例的差异。来源：本书整理。*

| 材料类型 | 可以展示 | 不足之处 | 是否适合进入正文 |
| --- | --- | --- | --- |
| 功能演示 | 理想路径和界面交互 | 缺失败样本、权限和评估 | 不适合单独进入 |
| 内部复盘 | 真实问题、失败原因、修复动作 | 可能含敏感信息 | 脱敏后可作为素材 |
| 离线评测 | 样本质量和模型边界 | 无法代表生产收益 | 可支撑质量结论 |
| 正式案例 | 任务链路、证据、风险、评估 | 需要复审成本 | 适合进入正文 |

## 55.2 平台化收束的写作方式

案例除了描述一个 Agent 的功能，还要说明平台能力如何沉淀。一个销售 Agent、一个财务 Agent 和一个法务 Agent 如果每个都从头实现工具权限、审批、Trace 和评估，就会变成重复建设。平台化收束要把案例中的共性能力抽出来，反写回前文的平台章节。

后续补写时，每个案例都要把业务应用和共享平台分开。业务应用负责场景规则、用户界面和任务语境，共享平台负责运行状态、工具注册、审批插入、审计记录、评估样本和成本控制。工具调用也要写清楚：哪些工具注册到 Tool Registry，哪些调用需要审批或审计，哪些调用只有只读权限。上下文来源同样要拆开，RAG 负责文本证据，Memory 负责会话和用户偏好，语义层或主数据系统负责指标、客户、合同和订单等权威对象。

结果交付也要分层。低风险摘要可以直接返回给用户，高风险报价、财务结论和法律意见只能作为草稿或建议，等待业务负责人确认。评估指标则要同时覆盖质量、成本、时延、复核负担和业务价值。这样写出来的案例，才能让读者看到业务规则和平台机制各自承担什么，而不是把所有责任都塞进一个“Agent 自动完成”的句子里。

平台化收束的难点在于保留业务差异，同时避免每个业务重新造一套基础能力。销售报价和财务分析的规则不同，但二者都需要工具权限、口径版本、审批状态和 Trace 证据；客服工单和运维告警的对象不同，但二者都需要任务状态机、失败恢复和人工接管。案例写作要把“业务规则”与“平台机制”分开。前者由业务团队维护，后者沉淀进平台章节。

这种写法也能帮助读者判断投入优先级。如果三个案例都需要审批链路，就应优先建设第30章的人机协同能力；如果多个案例都出现指标口径冲突，就说明第33章语义层和第34章 NL2SQL 需要补口径版本治理；如果多个案例都无法解释为什么失败，就说明第38章 Trace 和第39章评测样本还不够。案例不是正文末尾的装饰，它会反向检验前文架构是否完整。

### 55.2.1 从单点案例抽取共享能力

平台化收束不是把所有业务差异抹平。它要保留业务应用的差异，同时识别可复用的运行机制。销售场景中的折扣审批、财务场景中的口径确认、法务场景中的责任意见签署，表面不同，但都可以抽象为“高风险动作前的人类确认”。这类能力应沉淀到 Runtime 和 Human-in-the-loop，而不应由每个应用自己实现。

*表55-4：单点案例到共享平台能力的抽取。来源：本书整理。*

| 单点现象 | 可抽取的平台能力 | 对应章节 |
| --- | --- | --- |
| 折扣超过阈值要审批 | Run 状态机、审批节点、超时处理 | 第22章、第30章 |
| SQL 查询可能拖垮库 | sql_executor 资源限制、只读策略 | 第34章、第42章 |
| 回答必须引用政策原文 | RAG 证据覆盖、引用校验 | 第20章、第40章 |
| 工具参数可能越权 | Tool Registry、策略引擎 | 第23章、第51章 |
| 多角色同时参与任务 | 多 Agent 分工和交接协议 | 第28章、第30章 |

### 55.2.2 平台化收束要写清楚 ownership

共享能力一旦沉淀，就要有人负责。案例复审时要检查 ownership：业务团队负责规则解释，平台团队负责运行和工具治理，安全团队负责策略和审计，数据团队负责口径和血缘。如果正文只写“平台统一处理”，却没有说明谁维护规则、谁审批例外、谁处理事故，读者会误以为平台是一个没有组织成本的抽象层。

*表55-5：平台化能力的责任归属。来源：本书整理。*

| 能力 | 主要责任方 | 复审时要问 |
| --- | --- | --- |
| 工具注册与权限 | 平台团队、安全团队 | 谁批准工具上线，谁处理越权调用 |
| 指标口径与语义层 | 数据团队、业务 owner | 口径变更如何通知 Agent |
| 审批链路 | 业务 owner、平台团队 | 审批人缺席或拒绝时如何处理 |
| 评估样本 | 业务团队、评估团队 | 样本是否覆盖失败和边界场景 |
| Trace 与审计 | 平台团队、合规团队 | 证据保留多久，谁可以访问 |

ownership 还要写到变更流程里。业务规则变化时，谁通知 Agent？语义层字段下线时，谁触发评测回归？工具 API 改参数时，谁更新 Tool Registry？审批人组织架构变化时，谁维护审批链？这些问题如果不写，案例会默认“平台自动知道一切”。现实中，平台只能执行被声明和被维护的规则，不能替组织消化所有变更。

因此，案例复审时应同时检查“运行责任”和“维护责任”。运行责任回答线上出问题谁处理；维护责任回答规则变化谁更新。一个财务 Agent 如果依赖经营指标口径，数据团队和业务 owner 就必须对口径版本负责；平台团队只能保证版本进入 Runtime 后被正确引用和记录。把责任写清楚，案例才不会把平台描写成没有边界的万能系统。

## 55.3 与第一版主体章节的关系

案例平台化收束不能脱离主体章节。单点 Agent 如果要进入统一运行态，应回到第22章的 Agent Runtime；工具和权限治理要回到第23章 Tool Registry 与第24章 MCP；多角色任务的分工、交接和等待，要回到第28章多 Agent 协作和第30章 Human-in-the-loop；数据问题进入可信分析链路时，则要回到 Part III 的数据底座和 Part VI 的 DataAgent 主线；质量、成本、SLO 和评估要回到 Part VII；安全、合规和组织机制要回到 Part X。

这样写的好处，是让案例成为章节之间的连接器，而不是末尾附录。读者看到一个销售报价 Agent，不只知道“这个场景可以做”，还应知道它依赖哪些平台能力、哪些能力已经在前文讲过、哪些能力需要后续补齐。若案例无法映射到主体章节，通常意味着它只是一个孤立演示；若一个案例映射到太多章节却没有主线，说明正文可能需要重新收束。

案例复审时，主体章节不是“相关阅读”，它们是检查案例是否站得住的依据。比如一个 DataAgent 案例如果没有语义层，就很难支撑可信分析；一个客服 Agent 案例如果没有人工接管，就很难进入长任务和高风险场景；一个研发 Agent 案例如果没有工具权限隔离，只写“提升开发效率”也不够。

### 55.3.1 反向修订主体章节

好的案例会引用前文，也会反过来暴露前文的不足。如果多个案例都需要某种能力，而主体章节没有讲清楚，说明主体章节需要补。比如后续案例如果反复出现“审批超时后如何处理”，第30章就应补充长任务等待和恢复策略；如果多个 DataAgent 案例都出现“指标口径变更导致解释错误”，第33章和第34章就应补充口径版本和 SQL 编译缓存失效。

*表55-6：案例反向修订主体章节的触发条件。来源：本书整理。*

| 案例中反复出现的问题 | 应回补的章节 | 回补方向 |
| --- | --- | --- |
| 审批等待、超时、恢复 | 第22章、第30章 | Run 状态和人工接管机制 |
| 工具参数越权 | 第23章、第51章 | 工具 schema、策略和审计 |
| 指标口径冲突 | 第33章、第34章 | 语义层版本、SQL 编译契约 |
| 证据引用不足 | 第20章、第40章 | RAG 评估和引用校验 |
| 成本不可控 | 第41章、第45章 | 缓存、网关限流和预算 |

### 55.3.2 案例索引要服务读者路径

后续案例增多后，索引不能只按业务类型排序。读者可能想找“如何做审批”“如何做工具权限”“如何做 DataAgent 评估”，也可能想找“销售/财务/法务场景”。因此案例索引至少要有两套入口：按业务场景进入，按平台能力进入。第一版可以先在 Part XI 中保留这套索引方法，等正式案例补齐后再落到导航和索引页。

## 55.4 案例复审材料

后续补写案例时，需要同步补齐以下材料。

- 脱敏业务任务说明。
- 数据对象、工具和权限拓扑。
- Run 状态机或时序图。
- 人工确认和拒答边界。
- 评估样本与验收口径。
- 失败模式和回滚动作。

材料不齐时，案例可以保留为后续计划，不应进入正式正文。编造公司、收益和截图只会让案例看起来完整，却削弱全书的证据标准。

补写清单应服务于决策，而不是制造文档负担。脱敏任务说明用于判断场景是否真实；数据对象、工具和权限拓扑用于判断系统边界；Run 状态机用于判断任务能否复盘；人工确认和拒答边界用于判断风险是否被控制；评估样本和验收口径用于判断结论是否可信；失败模式和回滚动作用于判断它是否接近生产。每一项都对应正文中的一个判断点，缺失时就要降低结论强度。

作者准备材料时，可以先写“最小案例包”。它不需要完整项目报告，但要包含一个脱敏任务、一条可解释运行链路、一份工具和权限说明、一组评测或复核样本、一段失败处理记录。最小案例包通过复审以后，再决定是否扩写成完整章节。这样能避免先投入大量润色，最后又因为证据不足被整体退回。

### 55.4.1 案例补全优先级

后续案例不应平均铺开。优先选择能覆盖全书主线、又能复用 mini-platform 的场景。第一批案例建议从两类开始：一类是 DataAgent 可信分析，能串起语义层、NL2SQL、Trace 和评估；另一类是工单/客服 Agent，能串起结构化输出、工具调用、人工接管和 Guardrails。销售、法务、运维等案例可以排在第二批，等证据材料更充分后再写。

*表55-7：后续案例补写优先级。来源：本书整理。*

| 优先级 | 案例方向 | 为什么优先 | 准备材料 |
| --- | --- | --- | --- |
| 第一批 | DataAgent 经营分析 | 能贯穿 Part III、Part VI、Part VII | 语义层样本、SQL 执行日志、评估集 |
| 第一批 | 工单/客服 Agent | 能覆盖工具调用、人工接管、安全边界 | 工单样本、工具契约、接管记录 |
| 第二批 | 销售报价 Agent | 高业务价值，但审批和责任边界敏感 | 报价规则、折扣审批、脱敏流程 |
| 第二批 | 运维 Agent | 能展示 Trace、SLO 和 GitOps 边界 | 告警样本、变更流程、回滚记录 |
| 暂缓 | 法务审阅 Agent | 风险高，材料脱敏难度大 | 条款样本、复核口径、责任边界 |

### 55.4.2 复审记录要留下来

案例被退回也有价值。退回原因可以反映本书后续要补的工程材料：缺评估样本、缺权限拓扑、缺失败恢复、缺脱敏截图，或者收益数字无法验证。建议后续维护一张案例复审记录，把每个候选案例的状态记录下来，而不是让材料散落在文档和聊天记录里。

复审记录不需要很复杂，但字段要能支撑编辑决策。`case_id` 用候选编号替代客户名，避免素材阶段就暴露敏感信息；场景类型用于区分 DataAgent、客服、销售、运维、法务等业务方向；证据状态要说明样本、Trace、评估和截图是否齐全；脱敏状态要记录图片、日志、数据和业务名词是否完成敏感信息检查；风险边界要明确自动、确认、拒绝和降级是否写清；主体章节映射用于判断这个案例对应哪些章节，是否需要反向修订；复审结论则给出通过、退回、保留素材或暂不发布。

复审记录还可以用于规划后续版本。若多个候选案例都因“缺 Trace”被退回，说明平台需要先补运行记录能力；若都因“脱敏截图不可用”被退回，说明编辑流程需要统一截图重绘规范；若都因“收益数字无来源”被退回，说明作者培训要强调证据边界。复审表不是行政表格，它是一本书继续迭代的质量反馈系统。

当案例被接受后，复审记录也不应删除。它可以附在内部编辑材料中，记录哪些结论被允许写入公开版本，哪些材料只能内部保留，哪些说法被降级或删除。这样后续重印、英文版翻译或新增图表时，编辑团队不会重新踩敏感信息和过度外推的问题。

## 55.5 案例材料不足时的降级写法

案例复审时经常会遇到材料不足。真实项目可能没有完整 trace，没有可公开截图，或者业务数据不能披露。这种情况下不应硬写成完整成功案例，而应降级为方法案例、链路示例或问题复盘。降级用于诚实说明证据边界，让读者知道哪些结论来自真实链路、哪些只是抽象后的方法总结，并不降低质量要求。

材料不足时，优先保留任务链路。读者需要看到用户目标、系统拆解、工具调用、风险控制和复核方式。具体数字可以脱敏，截图可以重绘，字段名可以替换为通用名称，但链路不能断。如果链路也无法证明，就不要写成案例，只能写成设计建议或待验证方向。

第54章负责案例准入，第55章负责把案例反馈回主体章节。复审发现的缺口应反向修改正文：如果案例总是在语义层口径处失败，就回到第33章补指标治理；如果案例总是在报告发布处失败，就回到第36章补 EvidenceRef 和复核流程；如果案例总是在审批处卡住，就回到第30章补 HITL 状态恢复。案例章不应只做附录，而应成为全书质量闭环的一部分。

案例发布前还要确定读者收益。一个案例至少要让读者学到一种平台能力、一条工程边界或一个失败教训。只有业务背景、界面截图和效果描述，不足以支撑本书案例。案例写作应回到本书主线：企业 Agent 平台怎样把模型能力转成可治理、可复盘、可运营的业务系统。

## 55.6 案例复审的工作流

案例复审应当有明确工作流，而不是最后由作者凭印象判断能否发布。第一步是核对场景边界：用户是谁，任务是什么，系统做了哪些动作，哪些动作由人确认，哪些能力只是演示。第二步是核对证据链：输入、工具调用、数据来源、Trace、评测样本、人工复核和最终产物是否能对应上。第三步是核对平台收束：这个案例沉淀了哪些共享能力，哪些仍然是场景专用实现，哪些需要回写到主体章节。

复审过程中要区分材料缺失和能力缺失。材料缺失是指系统可能已经具备能力，但书稿没有展示证据，例如没有 Trace 截图、没有失败样本、没有评测结果。能力缺失是指系统确实还没有实现，例如没有审批恢复、没有权限裁剪、没有报告版本。前者可以补材料，后者不能在书里写成已完成能力，只能写成后续扩展方向或设计约束。

案例复审还要避免把故事写得过满。第一版书稿不需要每个案例都覆盖完整平台能力，但需要诚实说明覆盖范围。一个只读问数案例可以重点展示语义层、NL2SQL 和 EvidenceRef；一个写操作案例才需要重点展示 HITL、工具幂等和补偿；一个组织治理案例可以展示评测和运营节奏。案例边界越清楚，读者越容易相信书中的平台设计。

## 55.7 与第54章的收束关系

第54章更适合承担案例准入和写作方法，第55章则负责把案例复审结果收回到平台主线。两章不能重复写“案例应该怎么写”。更好的分工是：第54章告诉作者哪些案例可以进入书、需要准备哪些材料、材料不足时如何降级；第55章检查这些案例是否真的支撑了全书主张，并把复审结论映射回前面章节。

映射关系可以很具体。销售案例如果暴露工具权限和审批问题，就回到第23章和第30章；财务问数案例如果暴露指标口径问题，就回到第33章和第34章；报告案例如果暴露证据不足，就回到第36章和第38章；安全案例如果暴露越权或注入风险，就回到第50章和第51章。这样案例会成为验证平台架构的材料，而不只是正文后的附录。

第一版完成时，第55章可以作为全书的质量闸门。它提醒作者和读者：案例用于检验平台设计能否经受真实业务流程，而不是证明 Agent 很强。能通过复审的案例，可以进入正式版本；材料不足的案例，宁可降级为设计讨论，也不要写成已经上线的成功实践。

## 本章小结

业务案例和平台化收束需要明确写作标准。正式案例要有可审计的任务链路、脱敏样例和评估口径；材料不足时，第一版宁可保留规范和补写清单，也不能编造公司、收益或截图。

案例复审要把功能演示和可发布案例分开。可发布案例必须说明失败、权限、评估和脱敏证据，也要写清楚场景怎样抽取为共享模块，谁负责规则、审批、工具权限、语义层版本、引用评估和成本管理。

案例补写可以先从 DataAgent 和工单/客服两类高复用场景开始。它们覆盖主体章节的大部分平台内容，也更容易反向发现 Runtime、Registry、Trace、评估和安全章节中的缺口。
