# 第55章 案例复审与平台化收束

---

案例材料形成以后，还需要经过复审。企业 Agent 案例牵涉工具权限、业务口径、人工确认和审计证据，一处边界写错，就可能让读者误判系统能自动完成的范围。销售 Agent 看起来只是给客户生成报价草稿，实际会碰到折扣审批、库存承诺和合同责任；财务 Agent 看起来只是解释指标波动，实际会碰到口径版本、凭证异常和正式披露；法务 Agent 看起来只是抽取条款，实际会碰到法律判断、责任意见和签署建议。案例如果只展示“系统能给出结果”，就会把这些责任压扁成一个功能演示。

复审的第一项工作，是检查证据是否支撑结论。一个案例可以说系统生成了报价草稿，但需要说明草稿来自哪些数据、折扣规则从哪里读取、哪些动作停在审批前；可以说系统辅助财务分析，但要说明指标口径、异常定位和复核记录；可以说系统帮助法务审阅合同，但要说明条款证据、风险标签和最终签署责任。缺少这些材料时，案例不能靠“保留人工确认”这类笼统说法带过，因为读者不知道确认发生在哪一步，也不知道系统失败后谁负责。

复审的第二项工作，是把案例重新收束到平台能力上。销售、财务、法务看起来是三类业务，底层却经常复用同一批能力：Runtime 负责运行态，Registry 管住工具，语义层管住口径，Guardrails 和审批链管住风险，Trace 负责事后复盘。一个案例如果只能说明某个应用做了一个功能，却看不出这些平台能力如何协作，就很难支撑本书的工程主张。它可以是产品材料，也可以是复盘材料，但还不能证明企业级 AI 原生业务系统的设计。这里还有一个容易忽略的问题：案例会反过来暴露主体章节的不足。多个案例都卡在审批超时，说明第30章的人机协同要补恢复策略；多个 DataAgent 案例都卡在指标口径，说明第33章和第34章要补版本和编译契约；多个工单案例都解释不了失败原因，说明第38章 Trace 和第39章评测样本还不够。案例的价值在于检查前文架构能否落到真实流程，而不是给全书末尾增加几个看似完整的故事。本书不使用缺少证据的具体企业故事、上线数据或收益结论。本章给出的是复审口径和平台化收束方法：先审证据，再审风险边界，然后把单点场景沉淀为可复用的平台能力。读者读完本章，应能判断一个候选案例能否进入公开叙述、该降级成什么形式，以及它暴露的问题应该回到哪些主体章节。

## 55.1 销售、财务与法务场景的共同边界

销售、财务和法务场景通常比普通知识问答风险更高，因为它们会影响报价、结算、合规判断或合同责任。案例呈现要先交代任务边界。“保留人工确认”这句话本身不够，正文还要写清系统在哪一步停下、谁接手、确认什么、确认结果如何回到系统。复审这些案例时，最容易出问题的是动词。案例写“生成报价”，读者可能理解为正式对外报价；案例写“完成财务分析”，读者可能理解为系统已经给出可披露结论；案例写“审阅合同”，读者可能理解为系统承担了法律判断。更稳的写法是把动作拆开：系统生成草稿、列出依据、标出风险、提交审批、等待确认、记录结果。每个动词都对应一个系统状态或人的责任，案例边界就清楚得多。

表 55-1 保留了三类场景的自动化范围和人工确认范围。它提醒读者：只要场景涉及外部承诺、正式披露或法律责任，系统输出就需要被写成“待确认材料”，不能写成已经生效的业务动作；表格的作用是保留边界判断，而不是把业务复杂性压成三行。

*表55-1：销售、财务与法务场景中可自动化部分与必须人工确认部分的边界。来源：本书整理。*

| 场景 | 可以自动化的部分 | 必须保留人工确认的部分 |
| --- | --- | --- |
| 销售 Agent | 汇总客户背景、检索历史报价、生成报价草稿 | 正式报价、折扣例外、客户承诺 |
| 财务 Agent | 解释指标变动、生成核对清单、定位异常凭证 | 财务结论确认、口径变更、正式披露 |
| 法务 Agent | 抽取条款、识别风险、生成审阅意见草稿 | 法律判断、合同签署建议、责任承诺 |

这三类场景的共同点，是系统帮助人更快接近判断，但责任仍由人和组织承担。复审时尤其要警惕两种写法：把“生成草稿”写成“完成业务动作”，把“辅助判断”写成“自动决策”。这两种表达都会把 Agent 的边界推过头。把边界写清楚，并不会削弱案例价值。相反，读者会更容易相信这个系统接近生产，因为生产系统本来就要知道哪些动作可以自动执行，哪些动作需要等待，哪些动作应该拒绝。一个能停下来的 Agent，比一个什么都声称能做的 Agent 更适合企业场景。

### 55.1.1 边界要落到状态和动作

案例中写“需要人工确认”仍然太抽象。复核时应继续追问：确认发生在哪个状态？系统给人什么材料？人确认以后，系统继续做什么？如果人拒绝或超时，系统如何处理？以销售报价为例，比较稳妥的状态设计是：Agent 先生成报价草稿和风险说明；若折扣低于常规阈值，系统只记录草稿；若折扣超过阈值，Run 状态进入 `waiting_approval`；经理确认后，系统才允许生成正式报价邮件。这个状态变化要能在 Trace 中看到，而非只写在设计文档里。

*表55-2：高风险业务案例中的状态边界。来源：本书整理。*

| 状态 | 系统动作 | 人的动作 | 复审重点 |
| --- | --- | --- | --- |
| draft_ready | 生成草稿、列出证据和风险 | 阅读并修改 | 草稿是否明确标注未生效 |
| waiting_approval | 暂停高风险动作、发送审批请求 | 确认、驳回或要求补充 | 审批人是否有权限 |
| approved | 继续执行被批准的动作 | 承担业务确认责任 | 执行动作是否与审批一致 |
| rejected | 停止执行并记录原因 | 给出拒绝原因 | 系统是否会绕过拒绝 |
| escalated | 转人工或转工单 | 接管任务 | 上下文是否完整移交 |

### 55.1.2 复审要区分“能力可演示”和“案例可发布”

功能演示跑通，不等于案例可以发布。演示通常只证明系统在理想输入下能产出结果；案例还要说明它面对边界输入、权限不足、证据缺失和工具失败时怎样处理。参考书的实战项目能成立，是因为项目、代码和 smoke test 互相支撑；在缺少可运行项目支撑的案例中，复审更要把证据要求写清楚。正式案例至少要能回答三个追问。第一，用户换一个不完整输入时，系统如何澄清或降级；第二，工具返回错误或权限不足时，前端和 Trace 怎样记录；第三，人工复核发现结果不可靠时，系统如何保留反馈并进入下一轮评估。演示视频通常不会展示这些路径，但生产案例必须展示或说明这些路径。

*表55-3：功能演示、内部复盘与正式案例的差异。来源：本书整理。*

| 材料类型 | 可以展示 | 不足之处 | 是否适合进入正文 |
| --- | --- | --- | --- |
| 功能演示 | 理想路径和界面交互 | 缺失败样本、权限和评估 | 不适合单独进入 |
| 内部复盘 | 真实问题、失败原因、修复动作 | 可能含敏感信息 | 脱敏后可作为素材 |
| 离线评测 | 样本质量和模型边界 | 无法代表生产收益 | 可支撑质量结论 |
| 正式案例 | 任务链路、证据、风险、评估 | 需要复审成本 | 适合进入正文 |

## 55.2 平台化收束的写作方式

案例除了描述一个 Agent 的功能，还要说明平台能力如何沉淀。一个销售 Agent、一个财务 Agent 和一个法务 Agent 如果每个都从头实现工具权限、审批、Trace 和评估，就会变成重复建设。平台化收束要把案例中的共性能力抽出来，连接回前文的平台章节。案例进入正文时，要把业务应用和共享平台分开。业务应用负责场景规则、用户界面和任务语境，共享平台负责运行状态、工具注册、审批插入、审计记录、评估样本和成本控制。工具调用也要写清楚：哪些工具注册到 Tool Registry，哪些调用需要审批或审计，哪些调用只有只读权限。上下文来源同样要拆开，RAG 负责文本证据，Memory 负责会话和用户偏好，语义层或主数据系统负责指标、客户、合同和订单等权威对象。

结果交付也要分层。低风险摘要可以直接返回给用户，高风险报价、财务结论和法律意见只能作为草稿或建议，等待业务负责人确认。评估指标则要同时覆盖质量、成本、时延、复核负担和业务价值。这样写出来的案例，才能让读者看到业务规则和平台机制各自承担什么，而非把所有责任都塞进一个“Agent 自动完成”的句子里。平台化收束的难点在于保留业务差异，同时避免每个业务重新造一套基础能力。销售报价和财务分析的规则不同，但二者都需要工具权限、口径版本、审批状态和 Trace 证据；客服工单和运维告警的对象不同，但二者都需要任务状态机、失败恢复和人工接管。案例写作要把“业务规则”与“平台机制”分开。前者由业务团队维护，后者沉淀进平台章节。

这种写法也能帮助读者判断投入优先级。如果三个案例都需要审批链路，就应优先建设第30章的人机协同能力；如果多个案例都出现指标口径冲突，就说明第33章语义层和第34章 NL2SQL 需要补口径版本治理；如果多个案例都无法解释为什么失败，就说明第38章 Trace 和第39章评测样本还不够。案例不是正文末尾的装饰，它会反向检验前文架构是否完整。

### 55.2.1 从单点案例抽取共享能力

平台化收束不是把所有业务差异抹平。它要保留业务应用的差异，同时识别可复用的运行机制。销售场景中的折扣审批、财务场景中的口径确认、法务场景中的责任意见签署，表面不同，但都可以抽象为“高风险动作前的人类确认”。这类能力应沉淀到 Runtime 和 Human-in-the-loop，而不应由每个应用自己实现。

*表55-4：单点案例到共享平台能力的抽取。来源：本书整理。*

| 单点现象 | 可抽取的平台能力 | 对应章节 |
| --- | --- | --- |
| 折扣超过阈值要审批 | Run 状态机、审批节点、超时处理 | 第22章、第30章 |
| SQL 查询可能拖垮库 | sql_executor 资源限制、只读策略 | 第34章、第42章 |
| 回答必须引用政策原文 | RAG 证据覆盖、引用校验 | 第20章、第40章 |
| 工具参数可能越权 | Tool Registry、策略引擎 | 第23章、第51章 |
| 多角色同时参与任务 | 多 Agent 分工和交接协议 | 第28章、第30章 |

### 55.2.2 平台化收束要写清楚 ownership

共享能力一旦沉淀，就要有人负责。案例复审时要检查 ownership：业务团队负责规则解释，平台团队负责运行和工具治理，安全团队负责策略和审计，数据团队负责口径和血缘。如果正文只写“平台统一处理”，却没有说明谁维护规则、谁审批例外、谁处理事故，读者会误以为平台是一个没有组织成本的抽象层。

*表55-5：平台化能力的责任归属。来源：本书整理。*

| 能力 | 主要责任方 | 复审时要问 |
| --- | --- | --- |
| 工具注册与权限 | 平台团队、安全团队 | 谁批准工具上线，谁处理越权调用 |
| 指标口径与语义层 | 数据团队、业务 owner | 口径变更如何通知 Agent |
| 审批链路 | 业务 owner、平台团队 | 审批人缺席或拒绝时如何处理 |
| 评估样本 | 业务团队、评估团队 | 样本是否覆盖失败和边界场景 |
| Trace 与审计 | 平台团队、合规团队 | 证据保留多久，谁可以访问 |

ownership 还要写到变更流程里。业务规则变化时，谁通知 Agent？语义层字段下线时，谁触发评测回归？工具 API 改参数时，谁更新 Tool Registry？审批人组织架构变化时，谁维护审批链？这些问题如果不写，案例会默认“平台自动知道一切”。现实中，平台只能执行被声明和被维护的规则，不能替组织消化所有变更。因此，案例复审时应同时检查“运行责任”和“维护责任”。运行责任回答线上出问题谁处理；维护责任回答规则变化谁更新。一个财务 Agent 如果依赖经营指标口径，数据团队和业务 owner 就必须对口径版本负责；平台团队只能保证版本进入 Runtime 后被正确引用和记录。把责任写清楚，案例才不会把平台描写成没有边界的万能系统。

## 55.3 与主体章节的关系

案例平台化收束不能脱离主体章节。单点 Agent 如果要进入统一运行态，应回到第22章的 Agent Runtime；工具和权限治理要回到第23章 Tool Registry 与第24章 MCP；多角色任务的分工、交接和等待，要回到第28章多 Agent 协作和第30章 Human-in-the-loop；数据问题进入可信分析链路时，则要回到 Part III 的数据底座和 Part VI 的 DataAgent 主线；质量、成本、SLO 和评估要回到 Part VII；安全、合规和组织机制要回到 Part X。这样写的好处，是让案例成为章节之间的连接器，而非末尾附录。读者看到一个销售报价 Agent，不只知道“这个场景可以做”，还应知道它依赖哪些平台能力、哪些能力已经在前文讲过、哪些能力需要后续补齐。若案例无法映射到主体章节，通常意味着它只是一个孤立演示；若一个案例映射到太多章节却没有主线，说明正文可能需要重新收束。案例复审时，主体章节不是“相关阅读”，它们是检查案例是否站得住的依据。比如一个 DataAgent 案例如果没有语义层，就很难支撑可信分析；一个客服 Agent 案例如果没有人工接管，就很难进入长任务和高风险场景；一个研发 Agent 案例如果没有工具权限隔离，只写“提升开发效率”也不够。

### 55.3.1 反向修订主体章节

好的案例会引用前文，也会反过来暴露前文的不足。如果多个案例都需要某种能力，而主体章节没有讲清楚，说明主体章节需要补。比如案例如果反复出现“审批超时后如何处理”，第30章就应补充长任务等待和恢复策略；如果多个 DataAgent 案例都出现“指标口径变更导致解释错误”，第33章和第34章就应补充口径版本和 SQL 编译缓存失效。

*表55-6：案例反向修订主体章节的触发条件。来源：本书整理。*

| 案例中反复出现的问题 | 应回补的章节 | 回补方向 |
| --- | --- | --- |
| 审批等待、超时、恢复 | 第22章、第30章 | Run 状态和人工接管机制 |
| 工具参数越权 | 第23章、第51章 | 工具 schema、策略和审计 |
| 指标口径冲突 | 第33章、第34章 | 语义层版本、SQL 编译契约 |
| 证据引用不足 | 第20章、第40章 | RAG 评估和引用校验 |
| 成本不可控 | 第41章、第45章 | 缓存、网关限流和预算 |

### 55.3.2 案例索引要服务读者路径

案例增多后，索引不能只按业务类型排序。读者可能想找“如何做审批”“如何做工具权限”“如何做 DataAgent 评估”，也可能想找“销售/财务/法务场景”。因此案例索引至少要有两套入口：按业务场景进入，按平台能力进入。前期可以先在 Part XI 中保留这套索引方法，等案例材料成熟后再落到导航和索引页。

## 55.4 案例复审材料

准备案例时，下面这些材料要同步补齐。否则章节即使写完了主体叙述，也很难进入统一复审和平台化沉淀。

- 脱敏业务任务说明。
- 数据对象、工具和权限拓扑。
- Run 状态机或时序图。
- 人工确认和拒答边界。
- 评估样本与验收口径。
- 失败模式和回滚动作。

材料不齐时，案例可以保留为候选材料，不应进入正文。编造公司、收益和截图只会让案例看起来完整，却削弱全书的证据标准。补写清单应服务于决策，而非制造文档负担。脱敏任务说明用于判断场景是否真实；数据对象、工具和权限拓扑用于判断系统边界；Run 状态机用于判断任务能否复盘；人工确认和拒答边界用于判断风险是否被控制；评估样本和验收口径用于判断结论是否可信；失败模式和回滚动作用于判断它是否接近生产。每一项都对应正文中的一个判断点，缺失时就要降低结论强度。案例材料可以先收敛成“最小案例包”。它不需要完整项目报告，但要包含一个脱敏任务、一条可解释运行链路、一份工具和权限说明、一组评测或复核样本、一段失败处理记录。最小案例包通过复审以后，再决定是否展开成完整章节。

### 55.4.1 案例准备优先级

案例不应平均铺开。优先选择能覆盖全书主线、又能复用 mini-platform 的场景。第一批案例建议从两类开始：一类是 DataAgent 可信分析，能串起语义层、NL2SQL、Trace 和评估；另一类是工单/客服 Agent，能串起结构化输出、工具调用、人工接管和 Guardrails。销售、法务、运维等案例可以排在第二批，等证据材料更充分后再写。

| 优先级 | 案例方向 | 为什么优先 | 准备材料 |
| --- | --- | --- | --- |
| 第一批 | DataAgent 经营分析 | 能贯穿 Part III、Part VI、Part VII | 语义层样本、SQL 执行日志、评估集 |

### 55.4.2 复审记录要留下来

案例被退回也有价值。退回原因可以反映还需要补齐的工程材料：缺评估样本、缺权限拓扑、缺失败恢复、缺脱敏截图，或者收益数字无法验证。案例复审记录应把每个候选案例的状态记录下来，而非让材料散落在文档和聊天记录里。复审记录不需要很复杂，但字段要能支撑证据判断。`case_id` 用候选编号替代客户名，避免素材阶段就暴露敏感信息；场景类型用于区分 DataAgent、客服、销售、运维、法务等业务方向；证据状态要说明样本、Trace、评估和截图是否齐全；脱敏状态要记录图片、日志、数据和业务名词是否完成敏感信息检查；风险边界要明确自动、确认、拒绝和降级是否写清；主体章节映射用于判断这个案例对应哪些章节，是否需要反向修订；复审结论则给出通过、退回、保留素材或暂缓发布。

复审记录还可以用于判断平台能力缺口。若多个候选案例都因“缺 Trace”被退回，说明平台需要先补运行记录能力；若都因“脱敏截图不可用”被退回，说明截图重绘规范不够清楚；若都因“收益数字无来源”被退回，说明案例证据边界需要提前确认。复审表不是行政表格，它是全书质量反馈系统。当案例被接受后，复审记录也不应删除。它应记录哪些结论可以公开，哪些材料只能内部保留，哪些说法被降级或删除。这样后续重印、英文版翻译或新增图表时，团队不会重新踩敏感信息和过度外推的问题。

## 55.5 案例材料不足时的降级写法

案例复审时经常会遇到材料不足。真实项目可能没有完整 trace，没有可公开截图，或者业务数据不能披露。这种情况下不应硬写成完整成功案例，而应降级为方法案例、链路示例或问题复盘。降级不会降低质量标准；它要求案例说明证据边界，让读者知道哪些结论来自真实链路，哪些只是抽象后的方法总结。材料不足时，优先保留任务链路。读者需要看到用户目标、系统拆解、工具调用、风险控制和复核方式。具体数字可以脱敏，截图可以重绘，字段名可以替换为通用名称，但链路不能断。如果链路也无法证明，就不要写成案例，只能写成设计建议或待验证方向。

第54章负责案例准入，第55章负责把案例反馈回主体章节。复审发现的缺口应反向修改正文：如果案例总是在语义层口径处失败，就回到第33章补指标治理；如果案例总是在报告发布处失败，就回到第36章补 EvidenceRef 和复核流程；如果案例总是在审批处卡住，就回到第30章补 HITL 状态恢复。案例章不应只做附录，而应成为全书质量闭环的一部分。案例发布前还要确定读者收益。一个案例至少要让读者学到一种平台能力、一条工程边界或一个失败教训。只有业务背景、界面截图和效果描述，不足以支撑本书案例。案例写作应回到本书主线：企业 Agent 平台怎样把模型能力转成可治理、可复盘、可运营的业务系统。

## 55.6 案例复审的工作流

案例复审应当有明确工作流，而非最后凭印象判断能否发布。第一步是核对场景边界：用户是谁，任务是什么，系统做了哪些动作，哪些动作由人确认，哪些能力只是演示。第二步是核对证据链：输入、工具调用、数据来源、Trace、评测样本、人工复核和最终产物是否能对应上。第三步是核对平台收束：这个案例沉淀了哪些共享能力，哪些仍然是场景专用实现，哪些需要回到主体章节。

复审过程中要区分材料缺失和能力缺失。材料缺失是指系统可能已经具备能力，但案例材料没有展示证据，例如没有 Trace 截图、没有失败样本、没有评测结果。能力缺失是指系统确实还没有实现，例如没有审批恢复、没有权限裁剪、没有报告版本。前者可以补材料，后者不能在正文里写成已完成能力，只能写成设计约束或待验证能力。案例复审还要避免把故事写得过满。案例文本不需要每个案例都覆盖完整平台能力，但需要诚实说明覆盖范围。一个只读问数案例可以重点展示语义层、NL2SQL 和 EvidenceRef；一个写操作案例才需要重点展示 HITL、工具幂等和补偿；一个组织治理案例可以展示评测和运营节奏。案例边界越清楚，读者越容易相信书中的平台设计。

## 55.7 与第54章的收束关系

第54章更适合承担案例准入和写作方法，第55章则负责把案例复审结果收回到平台主线。两章不能重复写“案例应该怎么写”。更好的分工是：第54章说明哪些案例可以进入正文、需要准备哪些材料、材料不足时如何降级；第55章检查这些案例是否真的支撑了全书主张，并把复审结论映射回前面章节。映射关系可以很具体。销售案例如果暴露工具权限和审批问题，就回到第23章和第30章；财务问数案例如果暴露指标口径问题，就回到第33章和第34章；报告案例如果暴露证据不足，就回到第36章和第38章；安全案例如果暴露越权或注入风险，就回到第50章和第51章。这样案例重点落在验证平台架构的材料，正文后的附录只是其中一部分。第55章可以作为全书的质量闸门。它提醒读者：案例目的在于检验平台设计能否经受真实业务流程，证明 Agent 很强只是手段之一。能通过复审的案例，可以进入正文；材料不足的案例，宁可降级为设计讨论，也不要写成已经上线的成功实践。

## 55.8 案例复审台账与演进路线

案例复审需要一份台账，不能只靠复审意见散落在文档评论里。台账应记录候选案例编号、业务方向、材料状态、证据来源、脱敏状态、风险边界、主体章节映射、复审结论和下一步动作。候选案例编号不要使用客户名或项目代号，避免素材阶段就泄露敏感信息。材料状态要区分任务说明、Trace、工具调用、评估样本、截图和人工复核是否齐全。复审结论要写清楚：通过、退回补材料、降级为方法示例、保留复盘材料，还是暂缓进入正文。

这份台账也能形成案例演进路线。材料不足时，可以先保留方法和准入标准；材料成熟后，再根据台账选择证据最完整的 DataAgent、工单/客服、运维或销售案例进入正文。选择顺序应看材料质量和主线价值，而不是看场景听起来是否热门。一个证据完整的工单 Agent，往往比一个只有效果描述的高价值行业案例更适合放进书里。台账能帮助团队抵抗“先把故事写满”的冲动，把案例演进建立在证据成熟度上。

复审台账还应记录反向修订项。若某个案例暴露第30章没有讲清审批超时，就记录“回补 HITL 超时恢复”；若某个案例暴露第33章缺少指标版本迁移，就记录“回补语义层变更治理”；若某个案例暴露第51章安全策略无法解释，就记录“回补策略证据与拒答说明”。这些条目应进入主体章节维护计划，而不是只留在案例章节。案例的价值在于验证主体架构，不能只作为正文后的展示材料。

## 55.9 案例进入正文后的维护

案例进入正文后仍要维护。业务规则、协议版本、工具契约、图表素材和评估口径都会变化；如果正文不维护，案例很快会从工程证据变成历史快照。维护时应优先检查三类内容：是否仍能复盘任务链路，是否仍能证明风险边界，是否仍能映射到主体章节。若某个工具接口已经下线，正文可以保留历史案例，但要注明它是历史材料；若某个指标口径已迁移，应更新说明或移出默认案例；若某个图表来自旧版流程，应重新绘制或标注版本。

英文版维护也要同步。案例翻译不能只翻译正文，还要同步图题、表题、术语、证据边界和材料来源。中文里写“脱敏案例改绘”，英文也应对应说明；中文里把某个材料降级为方法示例，英文不能写成 production case。双语版本如果边界不一致，海外读者会得到不同的风险判断。新增案例时，应先确定中文证据和表达边界，再按同一逻辑重写英文，而非逐句翻译。

维护还包括删除。已经无法复盘、脱敏风险升高、收益数字失去来源、主体章节已经改变导致映射失效的案例，应从正文移出或降级为复盘材料。删除案例可以保护全书证据标准。一本工程书真正需要避免的情况，是案例看起来丰富却经不起追问。只要复审台账和演进路线存在，案例数量可以逐步增加，质量标准不需要降低。

## 55.10 从复审结论到修订任务

案例复审的结论要能转成修订任务。通过的案例进入正文排期；退回的案例要列出缺失材料；降级的案例要说明改成方法示例、链路示例还是复盘材料；暂缓发布的案例要说明是脱敏风险、证据不足、收益无来源，还是主体章节尚未准备好。这样团队知道下一步该补 Trace、补评测样本、补截图、补审批记录，还是直接降低表达强度。没有这一步，复审会变成一次主观判断，下一轮仍然不知道如何改。

修订任务还应分配到具体角色。业务负责人补任务背景、用户目标和复核意见；平台负责人补 Runtime、Registry、Trace、HITL 和协议证据；数据负责人补指标口径、语义层版本和数据权限；安全或合规负责人补策略、脱敏和审计要求；内容负责人统一行文、图表编号、引用和证据边界。案例经常失败在职责不清：业务侧写不出 Trace，平台侧写不出业务价值，内容侧又没有足够证据判断哪些句子要删。把修订任务拆给正确角色，案例才可能补齐。

复审结论也要进入案例演进计划。材料不足时，可以保留方法框架和少量可公开链路；材料成熟后，再补证据完整的行业案例。中文先确认材料边界，英文再按同一边界重写；图表先用自绘结构图，等真实脱敏素材通过后再替换。这样的演进计划比一次性追求“案例完整”更可靠。它允许本书先建立质量标准，再逐步增加材料厚度。

最后，复审任务要回到读者价值。每个案例进入正文前，都要回答读者能学到什么：一种平台能力、一条风险边界、一个失败修复方法，或一种组织协作方式。若答案只是“这个行业也可以用 Agent”，材料还不够。企业级 Agent 平台工程需要的案例，是能让读者在自己的项目里复用判断方法的案例。

这套任务化复审也能减少章节返工。先补证据，再展开正文；先确认边界，再润色语言；英文版先对齐材料来源，再重写表达。顺序一旦反过来，正文很容易出现已经写得很顺、但证据无法支撑的段落。到那时再删改，成本比前期复审高得多。案例章节尤其需要这种前置判断，因为它天然容易被业务故事、界面截图和效果描述带偏。

案例复审最终要形成一种运营习惯。每次新增案例、替换图表、更新英文版或调整主体章节时，都回到复审台账检查材料、证据和边界。这样案例章不会随着材料增加而变成松散附录。它会持续扮演质量闸门，把真实业务材料和平台工程主线连接起来。

这也是第55章的收束作用：把案例从“写完即可”变成“持续复审、持续回写、持续维护”的工程流程。

只有这样，案例章节才会随着材料增加变厚，而不是变散。

## 55.11 从案例复审到主体章节校准

案例复审还可以指导主体章节校准。若候选案例反复缺少 Trace，说明可观测章节需要更具体地说明 Run 回放、证据导出和事故复盘；若候选案例无法解释工具权限，说明工具注册和 MCP 章节要补权限继承、凭证轮换和 Server 准入；若案例材料总是写不清指标口径，说明 DataAgent 章节要补语义层版本、指标迁移和查询证据。补充优先级应来自这些复审缺口，而非只按章节字数排序。

这种做法能把案例章纳入全书质量循环。案例承担正文校验材料的角色：一个章节如果能指导团队准备案例材料、识别证据缺口、写出降级边界，就说明它已经接近可落地；如果读者看完仍不知道上线时应保存什么证据、失败时怎样复盘、哪些动作需要人工确认，章节就需要继续加厚。第55章把这些判断显式写出来，让主体章节的补充更有方向。

这个机制也能避免为了篇幅而增加内容。新增内容要能服务至少一种复审动作：帮助团队准备材料，帮助判断证据，帮助平台团队补运行记录，帮助英文版保持同样边界。不能服务这些动作的段落，即使读起来完整，也不应放进案例收束章。案例章节越克制，越能维护全书的证据标准。

## 55.12 案例章节的持续维护

案例章节发布后，也需要维护。平台能力会变化，术语会变化，图表会替换，英文版会更新，真实案例材料也可能补齐。若案例章没有维护规则，很容易出现正文已经改了平台边界，案例仍沿用旧说法；中文案例更新了证据，英文案例还停留在旧版本；图表替换了，正文引用仍指向旧图。

版本维护要记录每个案例的材料状态、脱敏状态、图表状态、英文同步状态、关联章节和最近复审时间。主体章节新增能力后，要检查案例是否需要补证据；案例新增材料后，也要反向检查主体章节是否需要补解释。案例章和正文不是单向关系，真实材料经常能暴露正文里的薄弱环节。

案例可以分为三类维护状态：可发布案例、方法样例、候选材料。可发布案例需要完整证据和脱敏确认；方法样例说明写法和准入标准；候选材料只记录证据缺口，不进入正式叙事。这样案例章节既能随着材料成熟而扩展，也能避免为了完整感编造内容。

## 55.13 案例章节的证据留白

案例章节允许留白，但留白要说清楚原因。缺少真实项目材料时，不应补虚构企业、虚构数字或虚构事故。更稳妥的写法，是把可公开的信息写成方法，把缺失的部分写成需要补充的证据类型，例如业务目标、原流程、平台改造点、上线指标、失败样本、人工复核和持续运营。这样读者能看到案例应如何成立，团队也知道该向项目侧补哪些材料。

证据留白还可以保护敏感信息。企业项目里的客户名、收入、合同、权限策略和事故细节，往往不适合直接写进公开文本。章节可以使用脱敏后的任务链路描述，但要保留判断依据：任务从哪里开始，平台调用哪些能力，风险在哪里暂停，人工如何确认，产物如何进入下游。脱敏不能把案例写成空泛故事，证据结构仍要完整。

补充案例时，应优先补能支撑平台判断的材料，而不是补故事性细节。读者需要知道为什么这个任务适合 Agent、哪些能力真正复用、哪些地方失败过、平台如何修正。案例章节的价值，在于把前文的工程方法放回真实组织环境中验证。

## 55.14 案例复审的发布节奏

案例复审应进入固定发布节奏。材料不足时，可以只保留方法和少量链路示例；材料成熟后，再加入证据完整的公开案例。每次发布都应说明哪些材料已经满足条件，哪些材料仍在复核。发布节奏能避免两种风险：一种是为了让书看起来完整，把未验证案例写成成功实践；另一种是因为真实材料暂时不便公开，案例章节长期停留在抽象方法。固定节奏让团队知道什么时候补材料、什么时候降级表达、什么时候删除不再可用的素材。

发布节奏可以和主体章节修订绑定。每次主体章节增加 Runtime、Registry、Trace、Eval、安全或合规内容，都检查案例台账中是否有对应材料可以升级；每次案例复审发现缺口，也把修订任务回写到相关章节。这样案例不会变成书末孤立内容，而会持续推动全书加厚。比如一个工单 Agent 案例缺少审批超时材料，就回写第30章；一个 DataAgent 案例缺少指标迁移材料，就回写第33章和第34章；一个安全案例缺少误杀申诉材料，就回写第51章。

发布前还要做双语同步复审。中文案例如果降级为方法示例，英文也必须降级；中文图题写“脱敏案例改绘”，英文图题也要保留相同来源；中文删除未经验证收益，英文不能留下同类表达。案例章节比普通技术章节更容易在翻译中发生边界漂移，因为英文读者看不到中文复审过程。双语复审应检查标题、图表、证据来源、结论强度和风险边界。只有两种语言的证据强度一致，案例才适合进入正文。

案例发布节奏也要允许退出。一个案例作为方法样例保留后，可能因为材料无法脱敏、项目停止、收益口径丢失或主体章节改变而退出。退出不代表浪费，复审记录仍能留下失败样本、工具契约、证据要求和写作教训。对一本工程书来说，维护案例标准比保留案例数量更重要。案例能稳定进入、升级、降级和退出，说明全书的证据体系在运行。

## 55.15 案例复审后的文本收束

案例复审进入生产后，平台需要把事实核对、样本回放、图表来源、责任边界、可公开信息、删改记录和复审结论放进统一证据口径。证据口径会减少事后解释成本，让业务、平台、数据、安全和运营团队能够围绕同一组事实讨论问题。没有这些材料，故障发生后只能凭经验判断；有了这些材料，团队可以知道哪些输入有效、哪些动作已经执行、哪些产物可以继续使用、哪些结果需要撤回。

这类证据应和第54章案例准入、第38章 Trace 和第52章合规证据连起来。上游章节提供能力基础，下游章节使用运行结果，本章则负责说明中间环节如何被验证。若某个能力只在本章看起来完整，却无法进入 Trace、Eval、发布记录或合规证据包，生产系统仍然会出现断点。读者在实现时应把章节之间的接口看成工程契约，而不是阅读顺序上的相邻关系。

常见风险包括案例写成宣传稿、问题被过度美化、平台能力和业务结果之间缺少证据、公开版本泄露内部信息。这些问题通常不会在一次成功演示中暴露，因为演示样本往往干净、短小、路径明确。真实业务会带来旧数据、异常输入、权限变化、用户撤回、预算限制和长时间运行状态。平台如果没有把这些情况纳入样本和台账，后续扩展场景时就会重复遇到同类问题。

复审后的修改应进入版本记录，让后续案例复用同一套证据口径。执行记录至少要说明 owner、版本、样本、影响范围、处置动作和复查时间。记录不需要写成流程报告，但要足够让后来者理解当时的判断。对于高风险能力，还应说明哪些条件满足后才能扩大使用，哪些条件失败时必须降级或撤回。

落地时可以先选择少量代表场景建立这种习惯。实践上，应先把高频、高风险、外部可见的路径做扎实，再把样本、台账和复盘方式复制到其他能力中。这样做能让能力说明落到接入、验证、运营和退出，而不是停留在概念描述。

## 55.16 案例复审对正文的反向修订

案例复审不应只决定案例能否发布，还应反向修订前文。真实案例常会暴露正文中过于理想化的说法：某个工具接入流程在企业网络里需要额外审批，某个评测样本在业务变化后失效，某个 Trace 字段在事故复盘时不够用，某个 UI 提示让用户误解了系统状态。复审时要把这些发现写回对应章节，而不是只在案例章末尾列一条经验。

反向修订需要有明确入口。工具问题回到第23章和第24章，Planner 问题回到第25章，DataAgent 问题回到第32章到第36章，观测问题回到第38章，合规问题回到第52章，组织责任问题回到第53章。这样案例章节才会成为全书质量控制的一部分。读者也能看到案例承担校准平台工程判断的作用，而非只承担装饰作用。

复审记录可以增加“正文回写”字段：对应章节、需要改的判断、已有证据、是否已经修改、补充样本。这个字段很小，但能避免案例和理论分离。随着更多案例进入案例库，正文会逐步减少模板化表述，留下更贴近生产现场的判断。

## 55.17 案例发布后的反馈归档

案例发布后仍然需要反馈归档。读者、作者和业务团队可能会指出表述不准确、证据不足、术语不一致或流程已经变化。若这些反馈只停留在聊天记录中，下一次修订会重新寻找上下文。平台化写作应把反馈归档到案例编号、章节编号、证据材料和处理状态上。

反馈归档还可以服务持续维护。某个案例被多次质疑同一处指标口径，说明正文需要补充定义；某个案例总被问到安全审批，说明第50章或第52章需要增加连接；某个案例读者无法理解平台边界，说明总论章节的预期管理还不够清楚。案例反馈因此会成为全书修订输入，而不是孤立评论。

## 55.18 案例版本之间的差异说明

案例章节会随证据变化而调整。版本之间应说明差异：新增了哪些证据，删掉了哪些无法证明的判断，哪些图表替换为公开版本，哪些业务指标改为区间或相对描述。没有差异说明，读者很难判断文本质量提升在哪里。

差异说明也能保护写作边界。某些删改来自保密要求，某些来自证据不足，某些来自业务流程变化。把原因写清楚，可以避免下一轮修订把已经删除的内容重新加回来。可以维护简短差异记录，公开正文只保留最终文本。

## 本章小结

业务案例和平台化收束需要明确写作标准。正式案例要有可审计的任务链路、脱敏样例和评估口径；材料不足时，宁可保留规范和证据清单，也不能编造公司、收益或截图。案例复审要把功能演示和可发布案例分开。可发布案例必须说明失败、权限、评估和脱敏证据，也要写清楚场景怎样抽取为共享模块，谁负责规则、审批、工具权限、语义层版本、引用评估和成本管理。新增案例可以先从 DataAgent 和工单/客服两类高复用场景开始。它们覆盖主体章节的大部分平台内容，也更容易反向发现 Runtime、Registry、Trace、评估和安全章节中的缺口。
