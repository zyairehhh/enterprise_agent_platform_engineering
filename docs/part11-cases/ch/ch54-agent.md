# 第54章 业务案例准入与写作方法

---

企业级 Agent 的案例很容易写成“功能演示加效果口号”。一个团队做出知识助手原型，能在演示环境里回答制度问题，就会有人想把它写成“企业知识管理效率提升”；一个工单 Agent 能把投诉摘要得很顺，就会被写成“客服流程自动化”；一个研发 Agent 能在仓库里跑通一次代码修改，也容易被包装成“研发效率大幅提升”。这些说法听起来完整，却经不起复盘。读者真正需要知道的是：这个任务发生在什么业务流程里，系统用了哪些数据和工具，哪些动作真的自动执行，哪些地方停下来等人确认，失败后还能不能回放。

案例章节的难点不在写得热闹，而在证据是否够。企业里很多材料只能证明“系统演示过”，还不能证明“系统可上线”。一次离线实验可以说明模型在某组样本上的表现，却不能说明生产环境里的权限、审计、成本和人工接管都已经就绪；一张界面截图可以说明交互形态，却不能说明背后的工具调用是否受控；一次业务用户试用反馈可以说明方向有价值，却不能替代评测集、Trace 和复核记录。案例如果跳过这些限制，就会给读者制造错觉：似乎只要接上模型和知识库，业务系统就自然进入了可信自动化。

本章先建立案例准入方法，原因就在这里。企业 Agent 的案例必须接受比普通产品案例更严格的证据检验：它不只讲“做了什么”，还要讲“凭什么相信它”。材料进入正文前，需要先说明真实任务，再拿出数据、工具、权限、评估和人工确认的证据，并回扣前文的平台能力。缺少这些证据时，案例只能作为候选材料保留，不能用合成截图或未经验证的收益数字填补空白。

企业知识助手、工单/客服 Agent、研发 Agent 和运维 Agent 都适合说明平台能力，也最容易被写成泛泛的功能故事。知识助手要证明证据引用、权限过滤和知识更新；工单 Agent 要证明工具调用、人工接管和内容安全；研发 Agent 要证明代码工具权限、长任务执行和评测；运维 Agent 要证明 Trace、SLO 和变更边界。每个方向都能写成案例，但只有当任务、链路、证据和边界都能被复原时，才适合进入正式正文。本章的作用，是给案例写作立一条证据线。团队可以沿着这条线判断材料是否成熟，也可以用同一套口径退回夸大、缺证据或边界不清的案例。这样做会让案例章节推进得慢一些，却能保护全书的可信度：读者看到案例时，能把案例材料和前文的平台能力对应起来，而不是只看到一段宣传素材。

---

## 54.1 案例纳入标准

案例进入正文之前，要回答一个朴素问题：读者沿着材料，是否复原得出这件事是怎样做成的。如果材料只写“用了 Agent 提效”“接入知识库”“自动生成报告”，却没有任务输入、系统边界、失败处理和评估证据，那么它还不是案例，只是宣传素材。一个合格案例至少要满足四个条件。它们对应企业 Agent 上线后最容易发生争议的四个位置：任务写不清，读者不知道系统到底替谁解决了什么问题；平台能力对不上，案例就会变成孤立功能；数据与工具拓扑缺失，风险动作和权限来源都无法复盘；效果写得过满，后续任何一次失败都会反过来削弱案例可信度。

审核案例时，可以把自己放在事故复盘现场。业务负责人追问为什么系统给出这个建议，平台团队要能拿出运行记录；安全团队追问有没有无权数据进入模型，案例要能说明过滤发生在哪一步；若出现“效率提升”这类结论，材料必须指向样本、工时记录或复核口径。如果这些问题只能靠口头解释，材料还没有准备好。

表 54-1 保留为准入清单，因为它承担的是验收作用。正文中的叙述负责解释为什么这些条件重要，表格负责让后续作者逐项自检。两者配合使用，比把所有要求写成长段落更利于复审。

*表54-1：案例纳入正式正文的四个条件与验收口径。来源：本书整理。*

| 条件 | 验收口径 |
| --- | --- |
| 业务任务明确 | 能写清楚用户角色、输入、输出、决策动作和风险边界 |
| 平台能力可对应 | 能映射到 Runtime、Tool Registry、MCP、Planner、Memory、RAG、Eval、Trace 或 Guardrails |
| 数据与工具拓扑清楚 | 能说明涉及哪些系统、数据对象、权限、审计和失败恢复动作 |
| 不夸大效果 | 不写未验证的降本、提效、准确率或上线规模 |

这四个条件不是形式审查。它们分别拦住四类常见问题：任务不清、能力悬空、链路缺证据、效果被夸大。材料可以先按表 54-1 自检，再按同一口径复核。任何一项说不清，材料只能进入候选库，暂不进入正式正文。准入判断还要看材料之间能否互相印证。一个案例可能有界面截图，却没有工具调用记录；可能有评测分数，却没有样本来源；可能有一段漂亮的业务故事，却没有失败样例。单个材料看起来都合理，放在一起却无法证明同一条运行链路。正式案例需要的是同一任务从输入到输出的连续证据，而非几张彼此独立的展示材料。

准入审查还要留下可执行的退回意见。材料不成熟时，退回意见应指出缺的是哪一段链路：任务输入缺失、工具调用缺失、权限拓扑缺失、评测样本缺失，还是人工确认记录缺失。这样团队才能补齐材料，而不是继续追加更长的背景故事，让正文看起来更完整，却没有解决证据问题。案例章的判断标准，应从“故事是否好看”转向“证据是否连得上”。

### 54.1.1 任务要能被复原

任务明确，不等于写一句“销售需要自动报价”或“运营需要分析下滑原因”。可复原的任务至少包含五个要素：谁发起、带着什么输入、希望得到什么输出、系统允许做哪些动作、哪些动作必须停下来交给人。以报价场景为例，“生成报价建议”仍然太粗。更合格的写法是：销售在 CRM 中选择客户和产品包，系统读取历史合同、库存状态和折扣策略，生成报价草稿与风险提示；正式报价、折扣例外和客户承诺由销售经理确认。这样读者才能判断它是 Copilot、Workflow，还是具有 Agent 闭环的任务执行系统。

### 54.1.2 全链路证据覆盖

很多案例会展示一张漂亮的结果截图，却跳过系统如何得到这个结果。对本书来说，截图只能证明“界面出现过”，不足以证明链路可信。案例证据要覆盖输入、检索、工具调用、权限判断、输出、复核和反馈。Trace、Tool Call 记录、评测样本、脱敏日志和人工复核表，往往比最终截图更重要。

*表54-2：案例证据材料与可支持的写作结论。来源：本书整理。*

| 证据材料 | 可以支持的结论 | 不能支持的结论 |
| --- | --- | --- |
| 脱敏任务样本 | 任务类型、输入复杂度、用户角色 | 上线规模、业务收益 |
| Tool Call 记录 | 调用了哪些系统、是否产生副作用 | 工具返回结果一定正确 |
| Trace 与状态机 | 任务如何推进、在哪里等待人工确认 | 模型能力优于其他方案 |
| 评测集与人工复核 | 质量边界、常见失败类型 | ROI、组织效率提升 |
| 脱敏界面截图 | 用户如何看到结果和证据 | 生产链路已经稳定运行 |

写作时要控制证据的外推范围。一次离线评测适合说明质量趋势，不足以说明上线收益；一段脱敏日志适合说明链路形态，不能证明系统已达到生产 SLO。

### 54.1.3 脱敏范围与因果关系保留

案例脱敏不能停留在把公司名改成“某企业”。需要处理的对象包括业务身份、客户信息、系统域名、字段口径、审批角色、工单编号、日志时间和截图中的元数据。对于 Agent 案例，还要特别注意工具调用参数和检索片段，因为它们常常暴露内部系统结构。脱敏后的材料还要保留因果关系。合同编号可以替换，审批人可以改成角色，字段名可以泛化，但“哪个输入触发了哪个工具”“哪个证据支撑了哪个判断”“哪个风险动作被人确认”这些关系不能断。脱敏如果只剩下一段通用描述，案例就失去了工程价值；脱敏如果保留了真实系统细节，又会带来安全风险。复核时需要在两者之间反复校准。

*表54-3：Agent 案例脱敏检查项。来源：本书整理。*

| 对象 | 常见泄露点 | 写作处理方式 |
| --- | --- | --- |
| 业务实体 | 客户名、供应商名、门店、合同编号 | 改为角色或类别，保留任务关系 |
| 数据字段 | 内部字段名、指标口径、权限标签 | 保留语义，替换实现细节 |
| 工具调用 | 内网域名、真实 API、账号 ID | 写成接口类型和必要参数 |
| 日志与 Trace | 时间戳、请求 ID、用户 ID | 使用合成 ID，保留状态转移 |
| 截图 | 水印、导航栏、组织架构、头像 | 重新绘制或裁剪，只保留说明所需区域 |

如果脱敏会破坏案例的关键因果关系，就不要强行发布。更稳妥的做法是把材料沉淀为内部复盘，等匿名化样本足够后再进入公开版本。

## 54.2 推荐案例结构

每个案例建议按同一结构展开，便于与前文能力章节互相引用。统一结构的目的，是让读者比较不同场景：哪些能力来自业务差异，哪些能力应该沉淀为平台共性。案例应从业务任务写起，先交代谁在什么场景下发起任务，为什么固定流程或普通问答不够用。读者只有看清任务摩擦，才知道后面的 Agent 设计是否必要。随后写数据与工具，说明任务需要访问哪些系统、文档、表、API 或人工审批节点；这一段要把权限、数据来源和副作用讲清楚，避免把“接了几个工具”写成万能能力。

第三部分写 Agent 设计，解释 Planner、工具调用、记忆、检索、人工确认和结果交付怎样分工。第四部分写运行链路，描述一次任务从输入到输出经历哪些状态，在哪里记录 trace 和审计证据。风险控制和评估放在后半段：前者说明哪些动作需要拒绝、降级、复核或转人工，后者说明如何用离线样本、人工复核、线上反馈和业务指标判断是否可用。这个顺序很重要。技术名词先行，案例会变成架构目录复述；评估放得过早，案例又容易变成指标宣传。先写任务，再写数据与工具，接着写运行链路和风险控制，读者才能顺着材料判断：系统到底解决了什么问题，依赖哪些平台能力，哪些结论有证据支撑，哪些地方仍然需要人工判断。

每一部分都应该经得起复核追问。业务任务要说明“用户为什么不能用固定流程解决”；数据与工具要说明“哪些字段只能读、哪些动作会产生副作用”；Agent 设计要说明“为什么需要规划而非一次问答”；运行链路要说明“Trace 怎样还原当时发生了什么”；风险控制要说明“系统在哪些条件下停下”；评估方式要说明“样本和复核口径是否足以支撑结论”。这些问题答不上来，案例就不该进入正文。

### 54.2.1 任务优先的叙述顺序

差的案例常常这样开头：“本案例基于 RAG、Planner、Tool Registry 和 Guardrails 构建了一个智能客服 Agent。”这句话看似完整，其实没有告诉读者为什么需要这个系统。更好的写法是先写任务摩擦：客服需要在几分钟内处理退款争议，但订单系统、物流系统和政策知识库分散在不同系统；固定流程只能覆盖标准退款，遇到促销叠加、部分发货和人工承诺时就会失效。任务写清楚以后，平台能力才有位置。RAG 用来找政策，Tool Registry 用来约束订单查询和退款试算，Human-in-the-loop 用来处理例外退款，Trace 用来保留证据。这种顺序更接近参考书的行文方式：先让读者看到问题，再给出工程结构。

### 54.2.2 写清楚“不自动化”的部分

Agent 案例容易把自动化写得过满。成熟的企业案例往往会清楚标出系统停在什么地方。比如销售报价案例中，Agent 可以汇总信息、生成草稿、检查折扣策略，正式客户承诺仍由销售确认；法务审阅案例中，Agent 可以抽取条款和标注风险，最终法律意见仍由法务负责人签署；财务分析案例中，Agent 可以解释指标变化，披露口径仍由负责人确认。这些边界是生产系统可信的前提。写作时应把“自动完成”“需要确认”“拒绝执行”分开列出，读者才能判断案例是否可落地。

*表54-4：案例写作中的自动化边界表达。来源：本书整理。*

| 动作类别 | 写作时应说明 | 示例 |
| --- | --- | --- |
| 自动完成 | 输入、输出、成功条件和审计记录 | 查询历史订单并生成退款原因摘要 |
| 需要确认 | 谁确认、确认什么、超时如何处理 | 经理确认超额折扣后才生成正式报价 |
| 拒绝执行 | 拒绝条件、用户提示和记录方式 | 用户要求绕过审批或访问无权限合同 |
| 降级处理 | 降级触发条件和备用路径 | 检索证据不足时转为人工知识库工单 |

### 54.2.3 把评估写成证据，不写成口号

“准确率提升”“效率提升”只有在评估口径清楚时才有意义。案例至少要说明样本来自哪里，标注或复核由谁完成，失败类型如何分类，以及哪些指标只是过程指标。对 Agent 案例来说，最终答案准确率只是其中一项，还要检查工具调用是否正确、权限是否被遵守、人工确认是否按预期触发、失败时是否留下可复盘证据。

*表54-5：案例评估指标与适用边界。来源：本书整理。*

| 指标 | 适合回答的问题 | 风险 |
| --- | --- | --- |
| 任务完成率 | 系统能否走完整个任务链路 | 容易掩盖高风险动作被人工修正 |
| 证据命中率 | 回答是否引用到正确文档或数据 | 不能证明结论推理正确 |
| 工具调用成功率 | Tool 参数和返回是否符合契约 | 不能说明业务结果被接受 |
| 人工接管率 | 系统在边界处是否知道停下 | 过低可能代表越权，过高可能代表不可用 |
| 复核通过率 | 结果是否能通过业务人员检查 | 依赖复核标准的一致性 |

评估段落还要避免把所有指标写成“越高越好”。人工接管率就是典型例子。对于高风险报价、法务审阅或财务披露场景，接管率过低未必是好事，可能说明系统越过了该停下的边界；对于低风险 FAQ，接管率过高则说明系统无法独立处理常见问题。任务完成率也一样，如果系统把本该拒绝的请求也算作完成，指标会很好看，但风险很高。案例写作中要说明指标的适用边界，不能把它们当作统一评分。

评估结果最好能对应到失败分类。比如 DataAgent 案例可以把失败分成语义层口径错误、SQL 生成错误、查询超时、证据解释错误和权限拒绝；客服 Agent 可以分成政策检索失败、订单工具失败、退款规则冲突和人工接管超时。失败分类比单一准确率更能帮助读者理解平台该如何改进，也能回扣第38章至第42章的观测、评测、成本和 SLO 章节。如果案例只有小样本，不要强行写成“上线效果”。可以写“在这组脱敏样本中，系统暴露出三类主要失败”，也可以写“当前材料只能支撑离线质量判断，尚不能支撑业务收益结论”。这种写法比编造 ROI 更可信，也更接近参考书的工程写作风格：先把条件和证据说清楚，再给出有限结论。

表 54-6 的用途是防止案例与正文脱节，而非要求每个案例覆盖所有章节。工单 Agent 若缺少结构化输出和人工接管，便很难支撑第8章和第30章的主张；运维 Agent 若缺少 Trace、SLO 和变更边界，也只能算工具演示，不能算生产案例。

### 54.3.1 用能力地图筛掉“只有功能没有平台”的案例

一个案例能否进入本书，还要看它是否反映了平台能力。如果材料只说明“调用大模型生成了一个结果”，却看不出 Runtime、Registry、Gateway、Memory、Eval、Trace 或 Guardrails 的作用，那么它更适合放在博客或产品介绍中，而非放进本书正文。复核时可以使用一个简单判断：案例中至少应有三类平台能力真实出现，并且每一类都有证据。知识助手案例通常需要 RAG、Memory 和 Eval；工单 Agent 通常需要 Registry、Human-in-the-loop 和 Guardrails；DataAgent 案例通常需要语义层、sql_executor、Trace 和评估样本。

### 54.3.2 案例材料与工程证据的对应关系

案例应优先选择能和 mini-platform 工程路径对应的场景，而非另起一套无法验证的系统。案例不一定把所有代码写进正文，但至少要说明每类材料落到哪类工程证据上。

*表54-7：案例材料与 mini-platform 最小对应关系。来源：本书整理。*

| 案例材料 | mini-platform 对应能力 | 正文应说明 |
| --- | --- | --- |
| 任务状态流转 | Runtime / Run 状态 | 状态如何从输入推进到完成或失败 |
| 工具调用记录 | Tool Registry / executor | 工具契约、权限和失败返回 |
| 指标口径与查询 | semantic layer / sql_executor | 口径绑定、SQL 校验和只读限制 |
| 证据引用 | RAG / retrieval | 文档版本、引用片段和证据覆盖 |
| 评估样本 | eval / benchmark | 样本来源、标签和失败分类 |

## 54.4 案例材料发布门槛

准备公开案例时，可以使用匿名化复合场景，但要明确这是工程常见场景而非真实公司故事。正文不得写入客户名称、生产数据、内部域名、真实工单、真实账号或未经验证的收益数字。若案例需要展示界面、日志或报表，应使用脱敏样例，并把图表来源写入图题。案例发布前至少经过素材登记、证据复核和正文成稿三步。素材登记只确认任务说明、脱敏样本、系统拓扑、风险边界和评估材料是否齐备；证据复核检查材料是否支撑正文中的每一个结论；凡是涉及收益、准确率、时延、成本和上线规模的句子，都要有来源，没有来源的内容只能写成“待验证假设”或删除。通过这两步后，材料才适合进入公开案例。写作时优先保留因果链和失败边界，少写产品话术。

*表54-8：案例发布流程与退出条件。来源：本书整理。*

| 阶段 | 输入 | 产出 | 退出条件 |
| --- | --- | --- | --- |
| 素材登记 | 任务说明、样本、拓扑、截图 | 素材清单 | 材料缺关键链路时退回 |
| 证据复核 | 素材清单、评估记录 | 可写结论列表 | 结论无证据时删除或降级 |
| 正文改写 | 可写结论、章节映射 | 案例正文 | 通过 tone、图表、引用检查 |
| 发布复核 | 正文、图片、数据来源 | 合入版本 | 无敏感信息、无夸大收益 |

案例未通过复核通常说明材料还缺关键证据，不等于场景没有价值。缺少工具调用记录，正文就只能写“设计上需要受控工具调用”，而不能写成已经安全接入业务系统；缺少评测样本，正文只能描述评估计划，不能写“质量达到可用水平”；缺少人工确认记录，正文只能说明风险动作需要确认，不能写“风险动作已被控制”。把退回原因记录下来，可以反向推动材料补齐，也能避免相同问题在多个案例中重复出现。图表也要有证据约束。案例中的流程图、状态机、工具拓扑和评测表都应来自脱敏材料或可解释的合成样例。图题要说明来源，正文引用要能对应到图表编号。若图中展示的是通用模式，可以写“来源：本书整理”；若图中来自真实项目脱敏，应写清楚“来源：脱敏案例改绘”。真实案例进入正文前，建议先从单个场景写到可复审，再考虑扩展数量。三个证据扎实的案例，价值高于十个看似完整但无法复盘的故事。案例章节要让读者看到企业级 Agent 如何在数据、工具、权限、评估和组织责任之间落地，而非证明所有行业都能用 Agent。

## 54.5 最小案例包

准备业务案例前，团队应先整理最小案例包。最小案例包不追求材料完整漂亮，而是确认案例能否被复审。它至少包含六类材料：脱敏任务说明、输入样本、工具和数据边界、运行链路、风险处理、评估或复核证据。脱敏任务说明回答“谁在什么流程中遇到什么问题”；输入样本回答“系统实际面对什么复杂度”；工具和数据边界回答“系统读了什么、调用了什么、哪些动作有副作用”；运行链路回答“任务如何从发起走到完成、拒绝或转人工”；风险处理回答“哪些条件下系统停下”；评估或复核证据回答“凭什么判断这个方案可用”。

最小案例包要保留失败材料。很多案例只提交成功路径，读起来顺畅，却无法说明平台边界。企业级 Agent 的价值经常体现在失败路径：权限不足时是否拒绝，检索证据不足时是否降级，SQL 超时时是否保留 Trace，审批超时时是否转派，工具返回冲突时是否请求人工确认。如果案例没有失败样本，正文最多只能说明设计意图，不能写成生产经验。材料准备时，应至少提供一条成功链路、一条拒绝链路和一条降级或人工接管链路。

最小案例包还应区分真实材料和合成材料。真实材料经过脱敏后可以支撑“这个任务链路在真实流程中出现过”；合成材料可以支撑“这种模式可解释、可复审”，但不能支撑上线规模、收益或稳定性结论。二者都可以进入写作，只是结论强度不同。复核时要看每一句结论对应哪类材料：来自真实 Trace、脱敏样本、离线评测、专家复核，还是工程经验整理出的通用模式。来源不同，写法就要不同。

案例包的形式可以很轻。一个目录、几份脱敏 JSON、两张流程图、一次 Trace 导出、一组评测样本和一份复审记录，就足以支撑初稿判断。重点是让材料可追溯，而非把它包装成完整项目报告。若素材来自多个真实项目的共性抽象，正文要说明“复合场景”或“通用模式”，避免读者误以为它对应某一家企业的完整上线实践。

## 54.6 公开版本的表达边界

公开版本写作要把“能证明的内容”和“合理推断的内容”分开。能证明的内容包括任务链路、工具调用、状态迁移、审批记录、证据引用、样本复核和失败分类；合理推断的内容包括后续扩展方向、可能的组织收益、适合沉淀的平台能力。第一类可以写成案例事实，第二类只能写成分析或建议。尤其是收益数字、效率提升、成本下降和上线规模，如果没有来源，不能用口号补齐。

表达边界也影响标题和小结。案例标题不宜写成“某行业 Agent 成功实践”，除非材料能支撑真实上线和可复盘收益。更稳妥的标题是任务类型或方法边界，例如“经营分析任务中的证据链组织”“工单 Agent 的工具权限与人工接管”“销售报价场景的审批边界”。这样的标题降低宣传味，也让读者一开始就知道章节要解释的是工程问题。

公开案例还要控制细节粒度。写得太粗，读者只看到抽象流程；写得太细，可能泄露系统结构、业务规则或客户信息。比较合适的粒度，是保留输入、工具、状态、证据和责任之间的关系，替换掉真实系统名、字段名、账号、时间戳和业务身份。比如可以写“订单查询工具返回三类状态”，不写真实 API 路径；可以写“审批人角色为财务 Controller”，不写真实姓名和组织层级；可以写“报告引用了语义层指标版本”，不写内部指标表名。

当材料不足时，公开版本应主动降级。缺 Trace 时，写成设计方法或待补证据；缺评测时，写成链路示例；缺公开截图时，使用自绘图说明流程；缺真实收益时，删除收益判断。降级不是退步，它让章节的证据和结论匹配。读者看到明确边界，比看到一段完整但无法复盘的成功故事更容易建立信任。

## 54.7 案例材料自检

作者提交案例前，可以用一组问题自检。这个案例是否能用一句话说明业务任务，而不先列技术名词？读者能否看出用户输入、系统拆解、工具调用、人工确认和最终交付之间的因果关系？每个结论是否能对应到 Trace、样本、人工复核或公开文档？自动化边界是否写清楚，哪些动作由系统完成，哪些动作需要人确认，哪些动作必须拒绝？失败样本是否出现，还是全文只展示了顺利路径？

还要检查案例是否真的支持本书的平台主线。一个案例如果只说明模型回答很好，却没有 Runtime、Registry、Policy、Trace、Eval、Memory、语义层或 HITL 参与，就很难放进本书主体。它可能适合做产品短文，但不能支撑企业级 Agent 平台工程。相反，一个案例即使业务场景很小，只要能展示任务状态、工具治理、证据引用、人工边界和评估复盘，也值得进入正文，因为它能帮助读者理解平台能力怎样落到真实流程。

最后检查语言。案例写作应少用“成功打造”“全面提升”“深度融合”这类宣传句。更合适的写法，是把条件、动作、证据和限制写清楚：在什么输入下，系统调用了什么能力，留下了什么证据，在哪些边界停下，复核人员如何判断结果。案例章节的可信度来自这些细节，而不是来自形容词。

## 54.8 复审输出与正文收束

案例复审结束后，不应只给出“通过”或“退回”。更有用的输出，是一份可执行的正文处理意见：哪些材料可以写成事实，哪些材料只能写成方法，哪些句子需要降级，哪些图表需要重绘，哪些风险边界需要补证据。比如 Trace 齐全但评估样本不足时，正文可以写清楚运行链路，却不能写质量结论；工具调用记录完整但审批记录缺失时，正文可以说明受控工具调用，却不能写高风险动作已经被完整治理；脱敏截图可用但数据来源不明时，图可以保留，涉及指标的解释要删除或改成待验证。

复审输出还应标注主体章节回写点。案例材料如果反复暴露工具权限、审批恢复、指标口径、证据引用或成本控制问题，应把这些问题带回第23章、第30章、第33章、第36章、第41章等主体章节。这样案例会从正文末尾的补充材料，转成帮助主体章节变扎实的复审工具。即使暂时没有完整案例，也可以用复审规则反向检查前文：每个工程模块是否能产生案例所需证据，每个上线判断是否能被 Trace、Eval、HITL 或发布台账支撑。

正文收束时，应保留复核过程中的限制条件。很多案例在复核后会变得更克制：收益判断被删掉，自动化范围缩小，失败路径补进去，图表从真实截图改成脱敏改绘。这样的变化不应被视为削弱案例，而是让案例从宣传稿回到工程书。读者真正需要的是可迁移的判断方法：如何准备材料，如何识别证据缺口，如何降级表达，如何把单点经验沉淀为平台能力。只要这些判断写清楚，案例即使没有客户名称和漂亮数字，也能服务本书主题。

案例收束还要考虑章节节奏。案例方法章不能堆满表格和清单，否则读者会觉得是在读审稿制度。表格适合承载材料类型、评审状态和流程节点；判断逻辑、证据边界和写作取舍则更适合用正文展开。新增内容应优先说明“为什么这样写”“证据不足时怎样处理”“这个案例怎样反向影响平台设计”，而不是再增加更多 checklist。这样第54章才能从规则集合变成真正可执行的写作方法。

案例方法还要服务协作。业务、平台、数据和安全合规团队看到同一份最小案例包时，关注点不同：业务看任务是否真实，平台看状态和工具是否可复盘，数据看口径和权限是否清楚，安全合规看结论是否有证据、脱敏是否充分。第54章的规则要让这些角色能对同一份材料给出一致判断。若每个人只按自己的经验补充内容，案例会变成多个视角的拼贴；若复审问题和证据标准一致，案例才能形成统一叙述。

因此，案例写作不应等所有素材完美后才开始。可以先用最小案例包判断材料强弱，再决定写成正式案例、方法示例、链路示例或暂缓发布。这个过程让团队知道缺口在哪里，也让读者看到本书对案例证据的要求。案例章节的质量来自这种取舍，而不是来自案例数量。

新增案例时，也应保留这条顺序：先有材料包，再有复审结论，最后才进入正文。顺序清楚，案例才不会被叙事完整性推着走。英文版也要沿用同一套边界，先确认材料来源、脱敏范围和结论强度，再重写叙述。证据顺序清楚，正文才更稳，也更可复审。

## 54.9 案例方法对正文质量的约束

案例方法章的作用不止是告诉作者怎样补案例。它还应反向约束前面章节的写法。若一个章节提出了某项平台能力，就要能说清这项能力在案例材料中会留下什么证据：Runtime 留下状态和事件，Registry 留下工具版本和执行结果，语义层留下指标版本和查询口径，Trace 留下运行路径，Eval 留下样本和失败分类，HITL 留下审批责任和恢复记录。正文如果只写设计原则，却无法说明案例中如何验证，就说明这一段还停留在概念层。

这套约束能帮助团队判断内容深度。正文不应只是多解释概念，而要补足读者在真实项目里会追问的问题：上线时看哪些证据，失败时谁负责，材料不足时怎么降级，哪些结论不能写成事实。第54章因此可以作为全书的案例证据标尺。每次新增章节内容时，都可以问一句：如果要用案例证明这段内容，最小案例包里需要哪些材料？如果回答不出来，说明正文的工程落点还不够清楚。

案例方法还会影响图表使用。能用一段话解释清楚的写作边界，不应强行做成表格；需要横向比较材料状态、复审阶段和退出条件时，表格才有价值。这样可以减少案例章的清单感，让读者沿着证据逻辑理解写作取舍，而不是面对一串审稿规则。真实案例材料不足时，方法章节更要写得克制：它提供准入标准、降级方式和复审路径，为案例进入正文保留质量门槛。

## 54.10 案例准入与真实材料保护

案例准入要保护真实材料。企业案例往往包含客户名称、业务流程、指标口径、内部系统截图、成本数据和失败记录。可以使用脱敏链路和抽象任务，但不能把缺失材料补成虚构公司故事。更稳妥的写法，是说明任务类型、平台能力、证据结构和风险边界，保留企业身份、金额和敏感流程的脱敏处理。

真实材料保护还要影响图表和截图。截图如果不能公开，就不要用仿真图冒充真实系统；可以改用结构图、流程图或字段示意，并明确来源为本书自绘。指标结果如果不能披露，就写口径和验证方法，不写未经授权的收益数字。失败案例如果涉及安全或客户风险，就写问题类型和修复动作，不公开可复用攻击细节。

案例准入的目标，是让读者相信方法可复用，而不是相信某个夸张结果。一个合格案例应能说明用户任务、Agent 拆解、平台能力调用、证据留存、人工节点、失败恢复和后续复盘。材料不足时，宁可写成方法样例或待补案例，也不要把故事写满。

## 54.11 案例材料的采访与复核流程

案例材料最好通过访谈和复核获得。业务 owner、平台工程师、数据负责人和安全或合规负责人分别确认业务目标、平台改造、数据证据和风险边界。访谈记录不要直接写进正文，而要整理成可核对的材料包：任务链路、上线证据、失败样本、复盘结论和可公开范围。

复核流程要保护真实项目。项目团队确认技术事实，业务 owner 确认场景描述，安全和合规负责人确认脱敏范围，再决定哪些内容进入公开版本。这样案例不会变成宣传稿，也不会因为担心敏感信息而写得过空。即使案例材料暂不完整，也可以先保留材料结构，为后续补充真实细节留下清晰入口。

## 54.12 材料不足时的降级写法

案例材料不足时，正文要降级表达，而不是补齐故事。缺少 Trace 的材料，可以写成“任务链路设计”或“待复核链路”，说明系统应留下哪些状态和事件；缺少评测样本的材料，可以写成“评估计划”或“样本构造方法”，不能写成质量已经可用；缺少真实截图的材料，可以使用自绘流程图，但图题要说明来源为本书自绘；缺少收益数据的材料，应删除收益判断，改写为“若要证明收益，需要比较人工处理时长、复核通过率和返工率”。这种降级会让案例看起来更克制，但能保护文本可信度。

降级写法要保留读者能复用的工程判断。比如一个客服 Agent 只有工具接口设计，没有真实工单 Trace，正文仍然可以讨论工具白名单、幂等键、错误码和人工接管；但不能说它已经稳定处理客服请求。一个 DataAgent 只有语义层样例和 SQL 执行器，没有业务复核记录，正文可以说明口径绑定和只读校验，却不能写经营分析结论已经被业务接受。材料强度决定结论强度，写作时应主动标注这一点。

降级也要避免把章节写成空洞模板。读者不需要看到“后续可以完善材料”这种泛句，而需要知道下一步该补什么证据。缺工具调用记录，就补 Registry 调用日志和执行结果；缺人工审批记录，就补审批人角色、审批依据、超时处理和恢复状态；缺评测样本，就补样本来源、标签规则、失败分类和复核人；缺脱敏图，就补自绘图和来源说明。每个缺口都要对应具体补证据动作。

这种处理方式也能服务第55章的案例复审。第54章给出降级原则，第55章再把复审意见收束成版本决策：正式进入正文、作为方法示例保留、转为待补案例，或删除。案例可以少放，但不能降低证据标准。真实材料补齐后，再把降级段落升级为正式案例事实，这样整本书的案例部分会随着材料增长而增强，而不是靠缺乏证据的内容撑满。

## 54.13 案例准入材料的完整性审查

案例准入进入生产后，平台需要把业务目标、原始样本、系统边界、数据来源、人工复核、失败样本、上线证据和可公开范围放进统一证据口径。证据口径会减少事后解释成本，让业务、平台、数据、安全和运营团队能够围绕同一组事实讨论问题。没有这些材料，故障发生后只能凭经验判断；有了这些材料，团队可以知道哪些输入有效、哪些动作已经执行、哪些产物可以继续使用、哪些结果需要撤回。

这类证据应和第32章 DataAgent 主线、第50章安全和第53章组织治理连起来。上游章节提供能力基础，下游章节使用运行结果，本章则负责说明中间环节如何被验证。若某个能力只在本章看起来完整，却无法进入 Trace、Eval、发布记录或合规证据包，生产系统仍然会出现断点。读者在实现时应把章节之间的接口看成工程契约，而不是阅读顺序上的相邻关系。

常见风险包括材料只来自访谈、缺少失败样本、业务效果无法和平台能力对应、敏感细节不能公开。这些问题通常不会在一次成功演示中暴露，因为演示样本往往干净、短小、路径明确。真实业务会带来旧数据、异常输入、权限变化、用户撤回、预算限制和长时间运行状态。平台如果没有把这些情况纳入样本和台账，后续扩展场景时就会重复遇到同类问题。

案例 owner 应先补样本和证据，再决定是否进入正式章节；材料不足时，应降级为方法说明或匿名片段。执行记录至少要说明 owner、版本、样本、影响范围、处置动作和复查时间。记录不需要写成流程报告，但要足够让后来者理解当时的判断。对于高风险能力，还应说明哪些条件满足后才能扩大使用，哪些条件失败时必须降级或撤回。

落地时可以先选择少量代表场景建立这种习惯。实践上，应先把高频、高风险、外部可见的路径做扎实，再把样本、台账和复盘方式复制到其他能力中。这样做能让能力说明落到接入、验证、运营和退出，而不是停留在概念描述。

## 54.14 材料不足时的案例降级写法

案例准入章节还要说明材料不足时的处理方式。很多企业案例并非不能写，而是证据不够支撑完整叙事：访谈能说明需求背景，却不能证明平台能力；截图能说明交互形态，却不能说明运行质量；业务结果能说明价值，却不能说明 Agent 在其中承担了哪一段责任。遇到这类材料，正文不应强行写成完整案例。更合适的做法是降级为问题片段、方法说明或边界提醒，把能证明的内容写清楚，把不能证明的部分留给证据补充。

降级写法也要保留工程价值。若只有用户问题和人工修订记录，可以写成“样本如何进入评测集”；若只有一次事故复盘，可以写成“案例怎样暴露平台缺口”；若只有工具接入材料，可以写成“案例中的工具契约和审批边界”。这种写法比编造完整业务故事更可靠，也更接近参考书中先讲方法、再讲工程判断的组织方式。读者看到的是可复用的判断路径，而不是一个无法验证的企业故事。

可以为每个候选案例设置状态：可成章、可成片段、待补证据、暂不公开。状态变化需要记录缺少哪些材料、由谁补充、下次复审时间和公开限制。这样案例章节不会因为材料不足而空洞，也不会因为追求完整性而写出过度包装的内容。

## 54.15 案例素材的证据等级

案例素材可以按证据等级管理。最高等级是可回放运行证据，包括 Trace、评测样本、发布记录和事故复盘；第二等级是可核对业务材料，包括原始需求、用户反馈、人工修订和验收结论；第三等级是访谈和口头说明，只能用于背景铺垫。不同等级可以共同使用，但正文要让读者看出哪些判断有运行证据支撑，哪些只是场景说明。

证据等级能避免案例写作失真。只有访谈材料时，不应写成平台效果已经被验证；只有业务结果时，不应把结果全部归因给 Agent；只有系统截图时，不应推断后端治理已经完善。写作时可以给每个案例段落标注证据等级，最终正文不必显示标注，但团队应知道哪些句子可以写得确定，哪些句子需要保守表达。

## 54.16 案例匿名化后的可读性保留

案例匿名化后仍要保留可读性。企业名称、部门名称、真实指标和内部系统可以替换，但任务链路、责任边界、证据类型和失败处理方式应尽量保留。若匿名化只剩下一组抽象名词，读者无法判断这个案例为什么成立，也无法把经验迁移到自己的平台建设中。

可读性保留可以通过结构完成：先说明用户任务，再说明 Agent 调用了哪些平台能力，接着说明风险点和修正动作，最后说明哪些证据支持结论。业务细节可以模糊，工程链路不能模糊。匿名案例应检查是否保留任务背景、是否保留平台动作、是否保留失败样本、是否说明删改原因。这样案例能满足公开要求，也能继续承担工程说明作用。

匿名化还要保留时间顺序。读者需要知道问题先出现在哪里，平台采取了哪些动作，人工复核在什么位置介入，最终产物如何发布或撤回。若时间顺序被打散，案例会变成能力清单，无法说明工程判断是怎样形成的。公开版本可以隐藏具体日期，但应保留“试点、灰度、生产、复审”这样的阶段信息。

复核时还要检查匿名化是否改变了责任关系。若把业务 owner、平台 owner、数据 owner 全部改成笼统的“团队”，案例会失去组织治理意义。可以隐藏姓名和部门，但应保留角色分工。对于涉及审批、事故或退役的案例，还要保留谁提出问题、谁批准变更、谁承担维护这些角色线索。若公开材料不能保留角色线索，这个案例就更适合写成方法片段，而不适合写成完整案例。必要时可以把案例拆成两个段落：一个交代业务背景，一个只讨论平台工程动作。这样读者既能理解场景，也能看清可复用的工程判断。案例写作应让删改痕迹服务可信度，而不是削弱可读性，并且保留足够的上下文，帮助团队继续补充证据和复核材料，减少反复解释，便于长期维护和持续复查。

## 本章小结

本章保留案例写作框架，不提供未经验证的业务案例。案例进入正文前，应先能复原任务、链路、证据和边界，再把平台能力放进去解释。案例写作不能用故事包装功能清单，也不能用截图或收益数字填补材料缺口。脱敏范围要覆盖系统名称、数据样例、日志、工具参数、截图元数据和人员角色，替换公司名只是第一步。评估指标也要说明样本来源、复核口径和失败分类，不能把离线结果直接写成上线收益。只有这些材料齐备，案例才适合进入正文。
