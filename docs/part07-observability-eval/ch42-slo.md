## 1. 从周一早高峰事故开始

假设某企业每周一上午 9 点，销售经理都会打开 DataAgent 生成周报。上线初期，系统把每个请求都当作普通同步请求处理：接收问题，调用大模型，查数据库，生成图表，再生成 PPT。平时运行正常，可一到周一早高峰，模型开始排队，SQL 引擎持续超时，用户以为系统没反应，于是反复点击“重新生成”。入口请求被放大，下游工具更慢，最后平台虽然没有完全宕机，但用户拿不到可用报告。

这个场景说明了 Agent SLO 的特殊性。SLO 是 Service Level Objective，可以理解为“服务等级目标”。它不是一句“系统要稳定”，而是一组可衡量目标，例如“99% 的交互式请求在 10 秒内返回可用进度”“财务 DataAgent 每周任务成功率不低于 98%”“高风险任务安全违规率必须为 0”。

传统 Web 服务的 SLO 通常围绕可用性、延迟和错误率。Agent 平台要复杂得多。接口返回 200，不代表 Run 成功；Run 成功，不代表答案质量合格；答案生成很快，也可能是因为系统返回了低质量摘要；服务没有宕机，也可能因为长任务丢失、重复执行、越权访问或成本失控而让业务不可用。

所以 Agent SLO 的一句话定义可以是：用户在可接受时间和成本内，安全地完成有质量的任务。这里面同时包含可用性、延迟、任务成功率、质量、安全和成本。本章后面的所有机制，都是从这句话推导出来的。

## 2. SLO 是对用户预期的工程化表达

不是所有 Agent 都应该套同一套 SLO。交互式问答、长任务分析、后台批处理、审批流、报表生成的用户预期完全不同。如果用同一套指标管理所有场景，最后往往既管不住风险，也解释不了用户体验。

交互式问答强调首响应速度。用户不一定要求 3 秒内拿到完整答案，但希望几秒内看到系统开始理解问题、检索资料或执行工具。这里的 SLO 应拆成首响应延迟和最终答案延迟。首响应让用户知道系统还活着，最终答案才衡量任务完成。

长任务分析强调可恢复和可追踪。生成 CFO 报告可能需要几分钟，用户可以接受等待，但不能接受页面断开后任务丢失，也不能接受失败时没有解释。这里的 SLO 应该包含进度可查询、断点恢复、结果通知和失败可解释。

后台批处理强调吞吐和成本。比如每天夜间批量生成 500 份门店经营摘要，不需要秒级响应，但要在早上 8 点前完成，并且总成本不能超过预算。审批流则强调安全和审计，Agent 如果建议退款、发客户邮件或修改数据库字段，人工确认、权限检查和审计完整性比延迟更重要。

一个 SLO 定义可以这样写：

```yaml
slo:
  name: finance_dataagent_interactive
  scope:
    agent_id: dataagent_finance
    task_type: interactive_analysis
  objectives:
    availability: "99.9% monthly"
    first_response_p95_ms: 5000
    final_answer_p95_ms: 60000
    run_success_rate: ">= 98%"
    safety_violation_rate: "0"
    average_cost_per_successful_run_usd: "<= 0.20"
  guardrails:
    judge_score_p50: ">= 0.80"
    human_handoff_rate: "<= 5%"
```

这里有两个关键词：`objectives` 和 `guardrails`。前者是目标，后者是护栏。比如为了降低延迟切小模型，如果 Judge 分下降或人工接管率上升，就不能算 SLO 改善。SLO 不是单指标冲刺，而是多目标约束下的稳定性承诺。

## 3. 指标要沿着任务链路展开

设计 SLO 后，下一步是把它变成可观测指标。Agent 平台不能只看入口 QPS 和 HTTP 500，因为真正的问题常常藏在 Run 和 Step 里。周一早高峰事故里，入口服务可能一直返回 200，但模型队列、SQL 工具、图表渲染和 PPT 生成都已经在积压。

延迟指标要拆开看。首响应延迟影响用户是否觉得系统有反应；最终完成时长影响任务是否可用；模型调用延迟、工具调用延迟、队列等待时间、人工审批等待时间分别定位不同瓶颈。只看一个总延迟，排障时会不知道慢在模型、数据库还是队列。

成功率也要分层。入口请求成功不等于 Run 成功，Run 成功也不等于每个 Step 健康。一个 Run 可能最终成功，但中间 SQL 重试 5 次，成本和延迟都被放大。平台应该同时观察任务成功率、Step 成功率、工具成功率、重试率、超时率和降级率。

质量指标来自 Ch.39 和 Ch.40。离线 benchmark 给出回归质量，在线 Judge 和用户反馈给出真实场景质量。SLO 不一定要求每个任务都有 Judge 分，但关键任务至少要有质量抽样。成本指标来自 Ch.41，包括每个成功 Run 的平均成本、失败成本占比、缓存命中节省、每租户成本。安全指标则包括越权拦截、脱敏正确率、租户隔离、拒答正确率和审计完整率。

管理层视角可以用一个合成分表示整体状态：

$$
SLOScore =
w_a Availability
+ w_l LatencyScore
+ w_s SuccessRate
+ w_q QualityScore
+ w_c CostScore
+ w_p PolicyScore
$$

但工程执行时不能只看总分。总分适合概览，排障必须下钻。比如总分没变，但安全指标从 100% 变成 99.9%，在高风险场景里仍然是严重退化；又比如质量分没变，但失败成本上升，可能说明系统靠大量重试维持了表面成功率。

## 4. 错误预算决定发布节奏

错误预算是 SRE，也就是 Site Reliability Engineering，站点可靠性工程里的核心概念。简单说，如果 SLO 要求一个月 99.9% 可用，那么剩下 0.1% 就是允许失败的预算。它不是鼓励失败，而是帮助团队在快速迭代和稳定可靠之间做取舍。

Agent 平台也需要错误预算，但预算对象不止可用性。可以有延迟错误预算、任务失败预算、质量退化预算、成本超支预算和安全预算。安全预算通常应为 0，也就是不允许越权和泄漏。

假设某 DataAgent 每周有 10 万次交互式任务，SLO 要求任务成功率 98%。那么一周最多允许 2000 次任务失败。如果周三已经消耗了 1800 次失败预算，本周后续发布就应该收紧，除非是修复稳定性的变更。这个规则比“感觉最近不稳定，先别发版”更可执行，也能避免另一个极端：为了绝对稳定而停止所有迭代。

错误预算还能反过来约束实验。预算充足时，可以允许正常发布和小流量实验；预算消耗过快时，暂停非必要变更，优先修复失败原因；预算耗尽时，进入变更冻结，只允许稳定性修复。这样，SLO 就不只是看板上的数字，而是发布治理的一部分。

## 5. 限流、熔断与降级是三道防线

回到周一早高峰。真正的问题不是单个请求慢，而是请求在入口和下游之间不断放大。要让系统不被拖垮，通常需要三道防线：限流、熔断和降级。

限流控制进入系统的请求量。比如每个用户最多同时 2 个长任务，每个租户有并发上限，低优先级 Agent 在高峰期最多占用 10% 的模型吞吐。限流不是惩罚用户，而是防止少数高流量任务拖垮整体服务。对于重复点击同一个报告生成按钮，平台也不应该创建新任务，而应该返回已有 `run_id`。

熔断用于隔离不健康下游。如果 SQL 引擎持续超时，Agent 不应该继续把请求打过去；如果某个模型服务错误率过高，网关应该短时间内停止路由到它。熔断可以理解为：下游已经明显不健康，先保护系统，不要继续加压。

降级是在完整能力不可用时提供可接受替代方案。高峰期可以先返回文字摘要和核心图表，PPT 详细版稍后生成；可以减少上下文，只保留最近 Turn 和关键摘要；可以关闭非关键工具；可以把同步请求改成异步任务；高风险动作可以转人工或只读执行。

一个网关保护策略可以这样写：

```yaml
protection_policy:
  rate_limit:
    per_user_concurrent_runs: 3
    per_tenant_qpm: 500
  circuit_breaker:
    sql_executor:
      open_when_error_rate_gt: 0.30
      window: 2m
      cool_down: 5m
    model_gpt5:
      open_when_p95_latency_gt_ms: 30000
      window: 5m
      cool_down: 3m
  degradation:
    high_load:
      disable_tools: ["chart_render"]
      max_context_tokens: 12000
      prefer_async: true
    policy_sensitive:
      mode: read_only
      require_human_approval: true
```

降级有一个底线：不能绕过安全。主 SQL 引擎超时，不能自动切到一个没有权限策略的备用连接；大模型不可用，也不能让小模型执行本来需要人工审批的高风险动作。稳定性机制如果绕过安全，本身就会变成事故来源。

## 6. 长任务要离开单个 HTTP 请求生命周期

很多 Agent 任务不是几秒钟完成的。生成报告、分析一批工单、跑多轮数据查询、等待人工审批，都可能持续几分钟到数小时。如果这些任务绑定在单个 HTTP 连接上，浏览器刷新、网络断开、服务重启都会导致状态混乱。

长任务应该有独立的 `run_id`、状态机、检查点和进度查询接口。用户提交任务后，前端拿到 `run_id`，后续通过轮询或消息推送查询进度。Runtime 在每个关键 Step 后写入 Checkpoint，失败后可以从最近安全点恢复。

一个长任务状态机可以这样设计：

```text
submitted -> queued -> running -> waiting_approval -> running -> succeeded
                              \-> failed_retryable -> queued
                              \-> failed_terminal
                              \-> cancelled
```

`failed_retryable` 表示可重试失败，例如模型超时、临时网络错误；`failed_terminal` 表示不可重试失败，例如权限拒绝、参数不合法、数据不存在。Agent 必须区分这两类，否则会出现无效重试。

长任务还需要幂等。幂等的意思是同一个操作重复执行，不会造成重复副作用。发送邮件、写数据库、创建工单都不是天然幂等，必须带上 `idempotency_key`。如果任务恢复时无法判断上一步是否已经执行，就可能重复发邮件或重复写入。

```json
{
  "run_id": "run_report_042",
  "step_id": "step_send_report",
  "tool": "email.send",
  "idempotency_key": "run_report_042:step_send_report:v1",
  "status": "succeeded",
  "artifact_refs": ["ppt_report_042"],
  "checkpoint_id": "ckpt_042_006"
}
```

长任务可靠性来自多个机制共同作用：队列负责缓冲，心跳负责判断 worker 是否存活，Checkpoint 负责恢复，幂等 key 防止重复副作用，超时策略防止任务无限挂起，人工审批状态防止高风险动作自动继续。它们共同把“页面等着结果”改造成“任务可排队、可恢复、可解释”。

## 7. 容量规划要看到步骤放大效应

Agent 平台的容量规划不能只看入口 QPS。一次入口请求会放大成多次模型调用、工具调用、检索调用和存储写入，这叫步骤放大效应。周一早高峰事故里，入口只有一批用户点击生成周报，但每个报告背后都有 SQL、图表、PPT、Judge 和 trace 写入，下游压力远高于入口流量。

可以用一个粗略公式估算：

$$
Workload =
QPS_{entry}
\times AvgStepsPerRun
\times AvgCallsPerStep
$$

如果入口是 10 QPS，每个 Run 平均 6 个 Step，每个 Step 平均 1.5 次下游调用，那么下游调用可能是 90 QPS。高峰期再叠加重试，实际压力会更高。

容量规划要分别看 LLM 网关、模型服务、工具执行器、检索系统、队列系统、存储系统和评测系统。LLM 网关关注连接池、路由、超时、限流和重试；模型服务关注 GPU 显存、token 吞吐、并发和 KV Cache；工具执行器关注 SQL、代码执行、浏览器和外部 API；队列关注长任务积压和优先级；评测系统则要避免 Judge 抢占线上用户流量。

缓存命中率也会影响容量。Prompt Cache、Prefix Cache、Semantic Cache 命中率下降，可能导致模型吞吐突然不足。因此缓存命中率不只是成本指标，也是容量指标。弹性伸缩也要分层：模型服务按 token 吞吐和队列长度扩容，工具执行器按任务类型扩容，长任务 worker 按队列积压扩容，评测任务低优先级排队。

## 8. 稳定性建设要形成运营闭环

企业级稳定性不是“服务不宕机”这么简单。对 Agent 平台来说，稳定性包括服务可用、延迟可控、任务能完成、质量不退化、成本不失控、风险可拦截、事故可复盘。

这需要把 Ch.38 的观测、Ch.39 的离线评测、Ch.40 的在线评测、Ch.41 的成本治理和本章 SLO 放进同一个运营闭环。没有 trace，事故无法复盘；没有 benchmark，修复无法回归；没有在线评测，真实退化无法发现；没有成本治理，高峰期系统可能自我放大；没有 SLO，团队不知道什么时候该发布、什么时候该冻结。

日常运营也要围绕 SLO 展开。告警不应只看 CPU 或 HTTP 500，而要看任务成功率、P95 延迟、质量抽样、失败成本和安全违规。事故分级要区分普通失败、租户级影响、全局不可用和安全事故。模型、Prompt、工具、语义层、权限策略都要支持灰度和回滚。压测演练要模拟高峰流量、下游超时、缓存失效和模型服务降级。故障注入可以主动制造受控失败，验证熔断和降级是否生效。

一次稳定性复盘不应该只写“模型不稳定”。更好的复盘要写清楚：哪个租户、哪个 Agent、哪个版本、哪个 Step、哪个下游、哪个 SLO 受影响，错误预算消耗多少，是否触发降级，为什么没有提前发现，哪些样本要加入 Regression Set 或 Stress Set。这样复盘才会变成系统能力，而不是一次性文档。

## 9. 案例回放：高峰期报表生成保护

最后再回到周一早高峰的周报事故。改造后的平台首先重新定义了场景 SLO：交互式预览要在 10 秒内返回进度和摘要，完整 PPT 可以异步完成，并在 30 分钟内通知用户；安全违规率必须为 0，单份报告平均成本不得超过预算。

入口层增加了限流和去重。每个用户最多同时 2 个报告任务，每个租户有并发上限，重复点击同一任务返回已有 `run_id`，而不是创建新任务。这样用户的焦虑点击不会继续放大下游压力。

执行层把报告生成改成长任务。任务进入队列，worker 异步执行，前端展示进度，每个阶段写入 Checkpoint。高峰期先生成文字摘要和核心图表，PPT 详细版稍后生成；非关键美化步骤延后；低优先级租户排队。常用周报数据提前物化，指标解释和 schema 使用 Prompt Cache，图表模板复用。

下游层增加熔断。如果 SQL 引擎超时率超过阈值，平台暂停新报表查询，返回“数据处理中”的可解释状态，而不是让 Agent 无限重试。失败 trace 进入 Regression Set 和 Stress Set，下次发布前必须通过压力测试。

改造后，用户不一定更快拿到完整 PPT，但平台从“同步阻塞、重复请求、级联超时”变成了“可排队、可恢复、可降级、可解释”。这就是系统韧性：不是保证每个下游永远不出问题，而是在下游出问题时，业务仍然能以可控方式继续运行。

## 本章小结

### 关键结论

1. Agent SLO 要同时覆盖延迟、成功率、质量、成本和安全，不能只看服务可用性。
2. 不同 Agent 场景需要不同 SLO，交互式问答、长任务、批处理、审批流不能套同一套目标。
3. 限流、熔断和降级要按任务风险和业务价值分层设计，降级不能绕过安全。
4. 长任务必须具备队列、检查点、幂等、断点恢复和进度查询能力。
5. 容量规划要考虑 Agent 的步骤放大效应，而不是只看入口 QPS。
6. 企业级稳定性建设需要把观测、评测、成本和 SLO 放在同一个运营闭环中。

### 上线检查清单

- [ ] 是否为不同 Agent 场景定义了不同 SLO 和错误预算？
- [ ] 是否同时监控延迟、任务成功率、质量、安全和成本？
- [ ] 下游模型、工具、数据库异常时是否有熔断和降级路径？
- [ ] 降级路径是否仍然遵守权限、审计和安全策略？
- [ ] 长任务是否能断点恢复、查询进度并避免重复副作用？
- [ ] 容量规划是否考虑平均 Step 数、工具调用放大、token 吞吐和缓存命中率？
- [ ] 事故复盘是否能沉淀到 trace、benchmark 和回归样本中？
