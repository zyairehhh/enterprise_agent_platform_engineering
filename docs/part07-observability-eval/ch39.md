## 1. 什么是 Benchmark，以及它如何演进到企业 DataAgent

Benchmark 直译是“基准测试”，但在 AI 系统里，它不只是一个题库。一个合格的 benchmark 至少包含四件事：明确的任务定义、可复现的数据集、统一的评测流程和可解释的指标。换句话说，它要回答“测什么、用什么测、怎么测、分数怎么解释”。如果只有一批问题和答案，却没有任务边界、数据版本、评测脚本和指标口径，那更像练习题，不是工程上可用于回归和上线决策的 benchmark。

在传统机器学习和早期 NLP（Natural Language Processing，自然语言处理）阶段，benchmark 多用于单点能力评测。例如分类任务看准确率，也就是模型把样本分到正确类别的比例；机器翻译看 BLEU（Bilingual Evaluation Understudy，一种用词语重合度衡量译文接近参考译文的指标）；阅读理解看 EM（Exact Match，答案完全匹配率）和 F1（同时考虑答案词语命中率和覆盖率的指标）；检索任务看 Recall（召回率，相关内容被找回的比例）、MRR（Mean Reciprocal Rank，正确结果排得越靠前分数越高）和 NDCG（Normalized Discounted Cumulative Gain，衡量排序结果质量的指标）。这些 benchmark 的共同特点是输入输出相对固定，评测流程容易复现。它们适合比较模型基础能力，但很难回答“模型能不能完成一个真实业务任务”。

LLM 出现后，benchmark 开始覆盖更广的通用能力。MMLU、BIG-Bench、HELM、C-Eval、CMMLU 等代表性评测，把知识、推理、数学、代码、常识、安全等能力放进统一框架里。这个阶段解决了一个重要问题：同一个模型不能只在单一任务上看起来强，而要在多种能力上接受横向比较。但它仍然主要评估“模型本身”，而不是一个带工具、带数据、带权限、带运行轨迹的 Agent 系统。

当 LLM 被用于数据分析，评测进一步演化到 Text-to-SQL。WikiSQL、Spider 等 benchmark 把自然语言问题、数据库 schema 和 SQL 生成连接起来。Spider 的重要意义在于跨数据库泛化：模型不能只记住一套表结构，而要理解新的 schema。BIRD、Spider 2.0 又把任务推向更真实的数据分析环境：数据库更大，schema 更复杂，执行结果更重要，模型要面对更接近生产的数据连接和查询难度。

但是，企业 DataAgent 不是单纯的 Text-to-SQL 模型。它面对的是私有数仓、语义层、BI 资产、权限策略、历史对话、工具调用和产物交付。用户问“本月经营性现金流为什么下降”，真正要评的不只是 SQL 是否生成正确，还包括指标口径是否理解、数据源是否选对、工具执行是否成功、分析结论是否可复核、图表和报告是否合规，以及整个过程是否能通过 trace 回放。

这就把评测推向 Agent benchmark 和 Deep Research benchmark 的方向。AgentBench、WebArena、OSWorld 等强调工具使用和环境交互；DeepResearch Bench、ResearchRubrics 等强调多轮检索、证据整合、报告质量和引用可信度。Deep Research 评测研究中的 RACE、FACT、多维 rubric，本质上都在说明一件事：复杂 Agent 的质量不只在最终文本里，也在研究路径、证据引用和上下文整合里。

最近值得企业 DataAgent 特别关注的两类 benchmark，一类是 BEAVER，另一类是 Workspace-Bench。BEAVER 由 MIT 等机构团队发布，作者包括数据库领域图灵奖得主 Michael Stonebraker。它把 Text-to-SQL 拉回企业环境：私有企业数据仓库、真实查询日志、复杂 schema、领域知识和可诊断子任务。Workspace-Bench 1.0 则关注 Agent 在真实工作区中的文件依赖、跨文件检索、上下文推理和多步执行。它不是 DataAgent benchmark，但它提醒我们：企业 Agent 面对的常常不是孤立问题，而是有历史版本、隐式依赖和执行轨迹的工作空间。

所以，本章讨论的“企业级 DataAgent benchmark”不是公开排行榜意义上的分数游戏，而是一套生产质量系统。它吸收 LLM benchmark 的标准化思想，继承 Text-to-SQL benchmark 的执行评测，借鉴 Agent benchmark 的工具和环境交互评测，再结合 Ch.38 的 trace 回放，把结果、过程、轨迹、安全和持续回归放在同一个框架里。

```text
论文图片占位 Prompt 1：
生成一张“从 LLM Benchmark 到企业 DataAgent Benchmark 的演进时间线”横向图。时间线从左到右依次是：NLP / IR Benchmark、LLM Benchmark（MMLU / BIG-Bench / HELM）、Text-to-SQL（WikiSQL / Spider）、真实数据分析（BIRD / Spider 2.0）、Agent Benchmark（AgentBench / WebArena / OSWorld）、Deep Research Benchmark（DeepResearch Bench / ResearchRubrics）、企业信息 Benchmark（BEAVER）、工作区依赖 Benchmark（Workspace-Bench）、Enterprise DataAgent Benchmark。每个阶段下方用简短中文标注核心变化：从固定输入输出，到通用模型能力，到 SQL 泛化，到真实数据库执行，到多步工具使用，到证据与报告，到企业私有 schema，到工作区依赖，到企业内部持续回归。白底，学术综述图风格，蓝灰主色，关键节点用橙色强调。

这条时间线的结论很直接：企业 DataAgent benchmark 应该吸收公开 benchmark 的方法，但最终必须回到自己的生产问题。它要同时评结果、过程、轨迹、安全和运营成本。

## 2. 为什么 DataAgent 不能只测最终答案

DataAgent 的输出看起来是一句话、一个 SQL、一张图表或一份报告，但真实能力并不只发生在最后一步。一次企业数据分析任务通常要经历：理解业务问题、识别指标口径、定位表和字段、生成查询、执行与纠错、分析结果、选择图表、组织解释、处理权限和不确定性。最终答案正确，只能说明这条链路在这一次样本上走通；最终答案错误，也不一定说明模型本身不行，可能是上下文包漏了约束，语义层缺字段说明，工具返回超时，或者权限策略没有把拒绝原因讲清楚。

所以评测至少要回答三层问题。

第一，结果是否正确。SQL 是否能执行，结果数值是否与参考答案一致，报告里的关键结论是否被数据支持，图表是否表达了正确维度和指标。

第二，过程是否可靠。Agent 是否选对表和字段，是否理解指标口径，是否在工具报错后有效修复，是否在多轮追问中正确引用上一轮上下文。

第三，运行是否适合生产。它有没有越权，有没有泄漏敏感字段，成本是否可控，时延是否可接受，trace 是否能回放，线上失败能否沉淀到回归样本。

可以用一个简化公式表达企业 DataAgent 的综合评测思想：

$$
Score_{agent} =
w_r \cdot Score_{result}
+ w_p \cdot Score_{process}
+ w_t \cdot Score_{trajectory}
+ w_s \cdot Score_{safety}
- w_c \cdot CostPenalty
$$

这里的权重不是固定常数，而应该由任务类型决定。财务报表生成更重视安全、口径和可复核；临时探索分析可以适当容忍格式不完美；权限敏感任务则应把 `Score_safety` 设为门禁项，只要越权就直接失败。

```

## 3. DataAgent 的能力边界与指标设计

设计 benchmark 前要先定义能力边界。否则题库很容易变成“SQL 考试”：模型只要把自然语言翻译成 SQL 就算完成，实际却漏掉企业分析最难的部分。

企业 DataAgent 的第一类能力是理解业务意图。用户说“本月现金流为什么下降”，并不是让系统简单查询一个数，而是在要求系统确定指标、时间范围、对比基线和归因维度。这里常见错误是把“本月”理解成自然月，把“下降”理解成环比而不是同比，或者把“为什么”简化成“查一下是多少”。

第二类能力是理解指标和语义层。企业里的“收入”“GMV”“有效订单”“经营性现金流”都有口径，不同部门甚至可能有版本差异。DataAgent 如果只看字段名，很容易把相似字段混用。评测样本必须覆盖显式口径、隐式口径和冲突口径。

第三类能力是 Schema Linking，也就是把业务概念映射到表、字段、维度、指标和 Join 路径。BEAVER 之所以值得关注，正在于它把企业 Text-to-SQL 拆成多表检索、Join Key 检测、Column Mapping、领域知识抽取和查询分解等子任务。企业内部 benchmark 也应该记录这些过程标签，否则最终 SQL 错了以后很难定位是选表错、字段错、Join 错还是业务知识缺失。

第四类能力是执行和纠错。生产系统里工具会超时，SQL 会报错，权限会拒绝，数据会为空。一个可上线的 DataAgent 不能只会生成第一版 SQL，还要能读取错误、判断是否可重试、修复参数，并在不确定时请求澄清。

第五类能力是分析和产物生成。DataAgent 不只是数据库客户端，它还要解释变化原因、组织证据、生成图表、报告或 PPT。这里的评测不能只靠精确答案，往往需要语义断言、引用检查和专家 rubric。

第六类能力是安全合规。企业 DataAgent 的失败不只是“答错”，也可能是“答了不该答的内容”。权限、脱敏、租户隔离、审计和拒答质量，都应该进入 benchmark。

一个更可操作的指标设计是把能力映射到四个分数：

$$
Score_{result} = \alpha_1 ExecAcc + \alpha_2 AnsAcc + \alpha_3 AssertHit
$$

$$
Score_{process} = \beta_1 SchemaAcc + \beta_2 ToolAcc + \beta_3 ParamAcc + \beta_4 RepairAcc
$$

$$
Score_{trajectory} = \gamma_1 Converge - \gamma_2 RedundantCall - \gamma_3 ContextLeak - \gamma_4 UnboundedRetry
$$

$$
Score_{safety} =
\begin{cases}
1, & \text{权限、脱敏、租户隔离全部通过} \\
0, & \text{出现越权、泄漏或违规执行}
\end{cases}
$$

这些公式只是示意，关键是把评测指标和工程动作连起来。`SchemaAcc` 下降，优先看语义层和字段描述；`ToolAcc` 下降，优先看工具描述和 Planner 约束；`RepairAcc` 下降，优先看错误反馈和重试策略；`RedundantCall` 上升，说明任务规划或停止条件有问题。

## 4. Benchmark 不是题库堆砌，而是任务空间设计

企业 benchmark 的价值不在于题目数量最大，而在于覆盖真实任务分布和关键风险。一个 500 条高质量、可回放、可归因、版本冻结的企业 benchmark，通常比 5 万条自动生成但无业务约束的题库更有用。

任务空间要从真实业务出发。先看用户问什么：查询、对比、归因、预测、解释、报告。再看数据复杂度：单表、多表 Join、多事实表、跨域数据、历史快照。然后叠加业务口径、交互轮次、工具产物和风险等级。这样构造出来的样本，才像企业 DataAgent 每天会遇到的问题。

例如“查询本月收入”是低复杂度样本；“解释本月经营性现金流下降，并按区域拆解主要贡献因素”是中等复杂度样本；“基于上一轮图表，按华东区拆到城市级，生成给 CFO 的摘要，并隐藏客户明细”则同时覆盖多轮上下文、指标口径、产物生成和权限约束。

可以用覆盖率公式约束任务空间，而不是只统计样本数量：

$$
Coverage =
\frac{\sum_{d \in D} I(count(d) \ge min(d)) \cdot weight(d)}
{\sum_{d \in D} weight(d)}
$$

其中 `D` 是任务空间中的维度或分桶，比如财务归因、销售对比、多轮追问、安全拒答、报告生成。`min(d)` 表示这个分桶至少要有多少可用样本。这样可以避免 benchmark 被大量简单查询刷高数量。

```text
论文图片占位 Prompt 3：
生成一张“企业 DataAgent Benchmark 任务空间矩阵”示意图。横轴是任务复杂度：单表查询、多表 Join、指标口径、异常归因、多轮追问、报告产物；纵轴是企业约束：业务语义、权限、数据质量、时延成本、可回放。矩阵中用不同颜色标注样本覆盖密度，右侧有“Golden Set”“Regression Set”“Stress Set”“Safety Set”四个样本桶。白底，简洁学术论文图风格，中文标签。
```

样本集建议拆成四类。Golden Set 是核心业务场景的稳定回归集，标注严格、版本冻结；Regression Set 来自线上失败和版本变更，用于持续防退化；Stress Set 覆盖长上下文、多表、多工具、高成本和边界条件；Safety Set 专门覆盖权限、脱敏、租户隔离、拒答和审计。四类样本的更新节奏不同，不应混成一个大题库。

## 5. 企业数据分析场景如何抽象

企业场景抽象的目标是保留真实复杂度，同时去掉敏感细节。抽象得太浅，benchmark 会退化成玩具题；抽象得太真实，又可能暴露业务机密、客户数据和内部流程。

一个可复用场景要包含七类信息。首先是业务目标，也就是用户为什么要问这个问题。其次是指标体系，包括指标定义、计算公式、口径版本和例外规则。第三是数据结构，包括事实表、维表、主键、Join 路径和数据质量说明。第四是常见问题，也就是真实用户会怎么问、怎么追问。第五是权限边界，哪些字段、行、产物受限。第六是产物形式，最终要交付 SQL、图表、Excel、摘要还是 PPT。最后是失败风险，哪些地方最容易错。

以财务场景为例，不能只构造“查询 4 月收入是多少”这种单步问题。真实任务通常会这样展开：用户先问“本月经营性现金流为什么下降”，Agent 需要确定下降对象、对比基线和时间范围；然后识别口径，例如是否剔除一次性项目、是否按财务月；接着从区域、客户类型、产品线、回款周期等维度做归因；最后生成图表和管理层摘要，并确保图表和结论都能追溯到 SQL、数据结果和口径说明。

采购、供应链、客服、营销、风控、人力等场景也可以用同样方式抽象。不同场景的表结构和指标不同，但评测骨架相同：用户意图、业务口径、数据定位、执行过程、解释质量、权限边界和产物交付。

## 6. QA 与任务集如何生成

企业 benchmark 的样本来源不应该只依赖人工从零造题。更可行的方式是从已有业务资产中生成，再用规则、执行器和专家审核过滤。

语义层和指标字典适合生成指标解释、口径问答和字段映射题；历史 SQL 与查询日志适合生成 NL2SQL、Schema Linking 和 Join 路径样本；BI 看板和报表适合生成图表解释、指标拆解和管理层摘要；用户对话、客服工单和 Ch.38 的 Trace 则适合生成多轮追问、歧义澄清、工具失败、上下文遗漏和真实失败样本。专家设计仍然必要，尤其适合构造高价值边界样本、安全样本和压力样本。

生成流程可以分为七步：抽取资产、结构化任务、生成变体、自动校验、去重分级、专家抽检、版本冻结。自动生成不是把“造题”交给 LLM 就结束，而是要把候选样本转成稳定结构，记录 `question`、`context`、`expected_output`、`allowed_tools`、`rubric` 和版本信息。

一个样本不应只保存问题和答案，还应保存评测所需的上下文：

```json
{
  "case_id": "fin_cashflow_042",
  "task_type": "root_cause_analysis",
  "question": "本月经营性现金流为什么下降？按区域拆一下主要原因。",
  "context_refs": {
    "semantic_layer_version": "finance_semantic:v12",
    "schema_version": "warehouse:v2026_04",
    "policy_version": "finance_policy:v7"
  },
  "expected": {
    "sql_refs": ["gold_sql_042"],
    "answer_assertions": [
      "必须说明下降主要来自应收账款回款延迟",
      "必须按区域给出贡献度",
      "不得暴露客户明细名称"
    ],
    "artifact_type": "chart_plus_summary"
  },
  "rubric": {
    "schema_linking": 0.25,
    "execution_result": 0.30,
    "analysis_quality": 0.25,
    "safety": 0.20
  }
}
```

自动生成样本时要特别警惕“泄漏”。如果问题、参考 SQL 和答案都由同一个模型从同一段上下文生成，评测可能只是在奖励模型复述自己生成的数据。企业 benchmark 至少要用执行校验、专家抽检和线上失败样本来打破这种闭环。

## 7. Ground Truth 与标注体系构建

DataAgent 的 Ground Truth 不是单一答案。简单查询可以有精确答案，NL2SQL 可以有参考 SQL，SQL 可执行任务可以有结果表，分析报告可以有语义断言，安全任务可以有拒答或脱敏标签，多步 Agent 任务还需要过程标签。开放式分析和管理层报告则往往需要专家 rubric。

硬标签适合自动回归，软标签适合质量评审。硬标签可重复、成本低、适合 CI 门禁，但覆盖不了洞察质量和报告表达。软标签能评估深度、完整性、引用可信度和可读性，但需要专家或 LLM-as-a-Judge，并且要控制主观性。

标注流程建议采用“自动求解 + 专家审核 + 争议处理 + 版本冻结”。先自动生成参考 SQL、执行结果和候选断言，再由领域专家审核口径、SQL、结果和业务解释。对于争议样本，不要强行指定唯一答案，而是记录多个可接受答案和不可接受边界。每次发布 benchmark 版本，都要冻结数据版本、语义层版本、工具版本、权限版本和标注版本。

可以用下面的断言评分表示开放式分析任务：

$$
AssertHit =
\frac{\sum_{i=1}^{n} w_i \cdot hit(a_i)}
{\sum_{i=1}^{n} w_i}
$$

其中 `a_i` 是专家定义的关键断言，例如“必须说明下降主要来自应收账款回款延迟”“必须按区域给出贡献度”“不得暴露客户明细名称”。这类断言比完整参考答案更灵活，也更适合评估多种合理表达。

## 8. 结果评测、过程评测与轨迹评测

DataAgent 评测可以分成三层。

第一层是结果评测。它回答“最后交付物对不对”。对于 SQL 查询，可以看执行成功率、结果准确率、SQL 等价率；对于图表，可以看图表类型、轴字段、筛选条件和数据一致性；对于报告，可以看关键事实是否覆盖、结论是否被证据支持。

第二层是过程评测。它回答“Agent 是怎么得到结果的”。比如是否选对表，是否生成正确 Join，是否读取了正确 Memory，是否在工具失败后有效修复。过程评测通常依赖 Ch.38 的 Trace。

第三层是轨迹评测。它回答“这条执行轨迹是否适合生产”。它不仅看步骤对不对，还看步骤是否收敛、工具调用是否过度、是否重复检索、是否越权、是否把无关上下文塞进模型、是否在不确定时请求澄清。

Deep Research 领域的 RACE、FACT 和多维 rubric 可以迁移到 DataAgent 的报告评测。比如报告质量可以拆成全面性、深度、指令遵循、可读性；引用质量可以拆成事实丰度和引用可信度。DataAgent 还要额外绑定执行结果、指标口径和权限边界。

一个示意性的报告评分可以写成：

$$
ReportScore =
\lambda_1 Completeness
+ \lambda_2 Depth
+ \lambda_3 Instruction
+ \lambda_4 Readability
+ \lambda_5 CitationTrust
$$

如果采用参考答案相对评分，可以借鉴 DeepResearch Bench 中 RACE 的思想：

$$
RelativeScore(R_{agent}) =
\frac{S(R_{agent})}
{S(R_{agent}) + S(R_{ref})}
$$

这里的 `R_ref` 不是为了要求 Agent 写出同一篇报告，而是作为质量锚点，降低 Judge 给所有报告打高分的倾向。

```text
论文图片占位 Prompt 4：
生成一张“结果评测、过程评测、轨迹评测”三层金字塔图。底层是 Trajectory Eval，包含 Trace、Step、Tool Call、Cost、Safety；中层是 Process Eval，包含 Schema Linking、SQL Generation、Tool Selection、Evidence Use；顶层是 Result Eval，包含 Answer、SQL Result、Chart、Report。右侧画一条箭头表示从线上 Trace 沉淀到 Benchmark，再回归到 Prompt/Tool/Schema 修复。白底，论文插图风格，中文标签。
```

轨迹评测是企业 Agent benchmark 相比传统 DataAgent benchmark 最需要补上的部分。一个答案可能数值正确，但如果它通过越权字段、错误口径或不可复现的中间步骤得到，就不能算生产可接受。

## 9. 不同 Agent 轨迹如何半标准化评测

这里需要和 Ch.38 明确联动。不同 Agent 框架的轨迹格式通常不一样：LangGraph 可能记录节点状态和边转移，AutoGen 可能记录多角色消息，OpenAI Agents SDK 可能记录 tool call 和 handoff，企业自研 Runtime 可能记录 Step、Span、Event、Artifact。即使都叫 trace，字段、粒度、命名和父子关系也可能不同。如果评测平台直接依赖某一种原始格式，benchmark 很快就会被框架锁死。

更稳妥的做法是做“半标准化”。所谓半标准化，不是要求所有 Agent 都产生完全相同的 trace，而是在评测入口把不同轨迹归一到一组最小可比较对象：

```json
{
  "run_id": "run_fin_042",
  "trace_id": "trace_fin_042",
  "steps": [
    {
      "step_id": "s1",
      "type": "context_pack",
      "inputs": ["turn_001", "summary_003", "schema_finance_v12"],
      "outputs": ["ctxpkg_042"]
    },
    {
      "step_id": "s2",
      "type": "tool_call",
      "tool": "sql_executor",
      "inputs": ["ctxpkg_042", "schema_finance_v12"],
      "outputs": ["sql_result_042"],
      "status": "succeeded"
    },
    {
      "step_id": "s3",
      "type": "artifact_write",
      "inputs": ["sql_result_042"],
      "outputs": ["chart_042", "summary_042"]
    }
  ],
  "sources": [
    {"source_id": "schema_finance_v12", "kind": "schema"},
    {"source_id": "sql_result_042", "kind": "tool_result"},
    {"source_id": "chart_042", "kind": "artifact"}
  ]
}
```

这个格式只保留评测需要的公共骨架：步骤、类型、输入、输出、状态、工具、产物和 source 引用。原始 trace 仍然可以保留在 Ch.38 的观测存储里；评测平台读取的是归一化后的 `eval_trace`。

半标准化以后，可以从轨迹中抽取一个 source graph，也就是“这次答案到底依赖了哪些来源”。这点可以借鉴 Workspace-Bench 的思路。Workspace-Bench 关注工作区中文件之间的依赖关系：任务答案往往不是来自单个文件，而是来自多个文件、目录、历史版本和跨文件线索。DataAgent 里也有类似结构，只是 source 不一定是文件，而可能是原始 Turn、Context Summary、Memory、Schema、SQL、查询结果、BI 看板、指标字典、Artifact 和权限策略。

可以把一次 Run 抽象成有向图：

$$
G_{trace} = (V, E)
$$

其中 `V` 是轨迹中的 source 和 step，`E` 表示“读取、生成、引用、派生”的关系。比如 `schema_finance_v12 -> sql_generation` 表示 SQL 生成读取了财务 schema，`sql_result_042 -> chart_042` 表示图表由查询结果生成，`turn_001 -> ctxpkg_042` 表示用户原始问题进入了本次上下文包。

如果 benchmark 有参考轨迹图 `G_ref`，可以用一个简化的图覆盖率评估 source 依赖是否合理：

$$
SourceGraphScore =
\eta_v \cdot \frac{|V_{pred} \cap V_{ref}|}{|V_{ref}|}
+ \eta_e \cdot \frac{|E_{pred} \cap E_{ref}|}{|E_{ref}|}
- \eta_n \cdot Noise(G_{pred})
$$

这里 `V_pred` 和 `E_pred` 来自 Agent 的实际轨迹，`V_ref` 和 `E_ref` 来自标注或参考执行。`Noise(G_pred)` 用来惩罚明显无关的 source，例如为回答现金流问题读取了无关的人事表、重复检索无关文件、把未进入上下文的 Artifact 当作证据引用。

这种评测比“最终答案对不对”更有诊断价值。假设两个 Agent 都答对了现金流下降原因，但 A 的轨迹引用了正确指标字典、财务 schema 和 SQL 结果，B 的轨迹没有读取指标口径，只是根据字段名猜测。结果分可能相同，source graph 分应该不同。反过来，如果答案错了，source graph 能帮助定位是缺少了关键 source，还是读了 source 但没有正确使用。

半标准化轨迹还可以支持跨 Agent 对比。不同 Agent 的原始日志可能完全不同，但只要都能映射到 `context_pack`、`model_call`、`tool_call`、`artifact_write`、`policy_check`、`memory_read`、`memory_write` 等公共 step 类型，就可以比较以下指标：关键 source 覆盖率、无关 source 比例、工具调用冗余、失败恢复路径、权限检查是否发生、产物是否可追溯。

```text
论文图片占位 Prompt 5：
生成一张“Agent 轨迹半标准化与 Source Graph 评测”示意图。左侧展示三种不同 Agent 原始轨迹：LangGraph Trace、AutoGen Messages、Enterprise Runtime Spans；中间是 Normalized Eval Trace，包含 context_pack、model_call、tool_call、policy_check、artifact_write；右侧是 Source Graph，节点包括 Turn、Summary、Memory、Schema、SQL Result、BI Dashboard、Artifact，边表示 reads、generates、references、derives。底部显示 SourceGraphScore 公式。白底，论文技术图风格，中文标签。
```

需要注意，企业内部不一定能为每个样本标注完整参考轨迹图。可以分三档做：核心 Golden Set 标注完整 `G_ref`；普通 Regression Set 只标注关键 source 和禁止 source；线上失败样本先自动抽图，再由专家在复盘时补充关键边。这样既能借鉴 Workspace-Bench 的依赖图思想，又不会把标注成本推到不可执行。

## 10. 公开 Benchmark 能告诉我们什么，不能告诉我们什么

公开 benchmark 的价值在于提供横向比较和方法论参考，但企业不能直接把公开分数当作上线依据。

| Benchmark 类型 | 代表 | 能评什么 | 企业缺口 |
| --- | --- | --- | --- |
| 经典 Text-to-SQL | WikiSQL、Spider | 基础 SQL 生成、跨 schema 泛化 | 企业表结构、隐式口径、权限、日志来源不足。 |
| 大规模数据分析 SQL | BIRD、Spider 2.0 | 更复杂 schema、执行准确率、真实数据库连接 | 仍难覆盖私有数据仓库和企业语义层。 |
| 企业 Text-to-SQL | BEAVER | 私有企业数仓、复杂 schema、领域知识、子任务诊断 | 对多轮分析、产物生成和完整 Agent 轨迹覆盖有限。 |
| Deep Research | DeepResearch Bench、ResearchRubrics | 多步检索、证据整合、报告质量、引用可信度 | 偏开放研究任务，不等同于企业数据分析。 |
| 工作区任务 | Workspace-Bench 1.0 | 大规模文件依赖、跨文件检索、上下文推理、多步任务 | 更偏文件工作区，不直接评 SQL 和语义层，但适合借鉴轨迹与依赖评测。 |
| 通用 Agent | AgentBench、WebArena、OSWorld | 工具使用、环境交互、长链路执行 | 企业数据、权限、指标口径和产物审计不足。 |

BEAVER 特别值得企业 DataAgent 团队关注。相比 Spider 这类公开数据库 benchmark，BEAVER 来自私有企业数据仓库和真实查询日志，包含 9128 个 question-SQL pair、812 张表和 19 个领域，并提供多表检索、Join Key、Column Mapping、领域知识和查询分解等子任务标注。它的启发不是“拿 BEAVER 分数替代内部评测”，而是告诉我们：企业 SQL 的难点不只在模型生成 SQL，而在复杂 schema、隐式领域知识和多子任务组合。

Workspace-Bench 1.0 也值得关注。它评测 Agent 在真实工作区中处理大规模文件依赖的能力，包含多种工作者画像、大量文件类型、文件依赖图和多步任务。它与 DataAgent 的任务表面不同，但方法论很相似：企业 Agent 面对的不是孤立输入，而是带历史版本、隐式依赖、跨文件证据和多步执行的工作环境。对 DataAgent 来说，这对应 BI 看板、历史报表、SQL 文件、指标字典、数据血缘和用户对话之间的依赖关系。

Deep Research benchmark 的启发在于评测开放式产出。RACE、FACT、多维 rubric 等方法，可以迁移到 DataAgent 的报告评测：不仅看报告是否“写得好”，还要看全面性、深度、指令遵循、可读性、事实丰度和引用可信度。区别是 DataAgent 还要额外绑定数据执行结果、指标口径和权限边界。

```text
论文图片占位 Prompt 6：
生成一张“公开 Benchmark 与企业 DataAgent Benchmark 对照图”。左侧按类别列出 Spider/BIRD、BEAVER、DeepResearch Bench、Workspace-Bench、AgentBench；右侧是企业 DataAgent 需要补充的约束：Private Warehouse、Semantic Layer、Permission、Trace Replay、Artifact、Online Feedback。中间用箭头表示“可借鉴的方法论”和“不能直接替代内部 benchmark”。白底，论文综述图风格，中文标签，清晰简洁。
```

结论是：公开 benchmark 用来校准方法，内部 benchmark 用来决定上线。企业最终要评的是自己的表、自己的指标、自己的权限、自己的用户任务和自己的运行轨迹。

## 11. 企业级持续评测平台建设

Benchmark 不是一次性项目，而是一套持续运行的平台。每次模型、Prompt、工具、语义层、权限策略、数据版本变化，都可能让 DataAgent 的行为发生变化。持续评测平台要把这些变化纳入回归。

先看公开榜单怎么做。Spider、BIRD 这类长期维护的 Text-to-SQL benchmark，并不是发布一个数据集就结束。它们通常会提供固定的数据切分、统一提交入口、统一执行脚本、公开 leaderboard、模型提交记录和论文/系统链接。参赛者不能只说“我的模型更好”，而要在同一套评测协议下提交结果，由榜单按固定指标排序。这种机制带来三个价值：第一，任务定义稳定，便于长期横向比较；第二，评测脚本统一，减少“各测各的”；第三，榜单持续更新，能记录模型和方法的代际变化。

但公开榜单也有边界。Spider 和 BIRD 适合比较 Text-to-SQL 能力，却不知道企业内部的语义层版本、权限策略、BI 看板、历史 trace 和用户反馈。企业持续评测平台要借鉴它们的“固定协议 + 统一执行 + 可追溯榜单”，但评测对象要换成自己的生产系统。也就是说，企业内部也应该有一个私有 leaderboard：每个模型版本、Prompt 版本、工具版本、语义层版本都要在同一批 Golden Set、Regression Set、Safety Set 上留下可追溯结果。

企业级产品的做法会更接近“评测控制塔”。以 Eldros、Openlayer、EdgeVerve 这类面向企业 AI/Agent 质量治理的产品形态为例，它们不会只提供一个离线分数，而是把数据集管理、实验运行、Judge、线上监控、回归门禁、风险告警和审计记录连在一起。放到 DataAgent 平台里，对应的能力就是：从线上 trace 和用户反馈自动沉淀样本；在私有数据快照上重放 Agent；同时记录模型、Prompt、工具、语义层和权限版本；把结果分、过程分、轨迹分、安全分和成本指标放进同一个看板；当核心指标退化或安全样本失败时，阻断发布。

如果参考 EverOS 这类把 Agent 轨迹沉淀为 Case、再从重复模式中抽取 Skill 的产品思路，持续评测平台还可以多做一步：不只是保存失败样本，还要把失败轨迹转成可复用资产。比如一次现金流分析失败，不只记录“答案错了”，还要保存当时的 Context Package、source graph、错误 SQL、修复后的 SQL、专家解释和最终正确报告。下一次类似任务回归时，这个 Case 既是评测样本，也是改进 Planner、工具描述和语义层的训练材料。

平台需要有样本库、执行器、Judge、Trace 采集、指标看板、回归门禁和样本沉淀。样本库管理 benchmark 版本、难度、任务类型、来源和标签；执行器调用 Agent Runtime，并固定模型、Prompt、工具、数据和权限版本；Judge 运行规则评测、执行评测、LLM-as-a-Judge 和人工评审；Trace 采集保存每次评测 Run 的 Context Package、Step、Tool Call、Artifact 和成本；指标看板展示准确率、过程指标、安全指标、成本和延迟；回归门禁在 CI/CD、Prompt 发布、模型路由、语义层发布前阻断明显退化；样本沉淀则把线上失败、用户反馈和事故复盘转成新的回归样本。

持续评测的关键是版本绑定：

```text
eval_run_id
  benchmark_version
  model_version
  prompt_version
  tool_version
  semantic_layer_version
  policy_version
  data_snapshot_version
  runtime_version
  trace_id
```

没有这些版本，回归分数无法解释。比如准确率下降，到底是模型变了、Prompt 变了、语义层字段描述变了，还是数据快照变了？持续评测平台必须能回答。

一个推荐闭环是：线上 DataAgent 每次运行都写入 Trace 和用户反馈；失败、点踩、高成本、人工接管样本进入候选池；评测团队按任务类型和失败类型聚类；高价值样本补充 Ground Truth 和过程标签；样本进入 Regression Set 或 Safety Set；团队修复 Prompt、工具描述、语义层、模型路由或权限策略；回归通过后灰度上线；Ch.40 的在线评测继续观察真实用户分布。

举一个内部榜单例子。假设团队同时维护 `gpt-4.1 + prompt:v12`、`gpt-5-mini + prompt:v3` 和一个本地模型路由方案。每晚评测平台在同一份 `benchmark:v2026_06` 上运行三套配置，生成一张私有榜单：结果准确率、Schema Linking 分、SourceGraphScore、安全通过率、P95 延迟、平均 token 成本。如果新方案结果分高 2%，但 Safety Set 有越权失败，发布门禁应直接失败；如果结果分持平但成本下降 40%，可以进入灰度；如果 Regression Set 中现金流场景下降，就把失败 trace 加入复盘队列，而不是只看总分。

```text
论文图片占位 Prompt 7：
生成一张“企业 DataAgent 持续评测平台架构图”。从左到右：Online Trace/User Feedback 进入 Sample Mining；进入 Benchmark Store，分为 Golden Set、Regression Set、Stress Set、Safety Set；中间 Eval Runner 调用 Agent Runtime、Model Gateway、Tool Registry、Semantic Layer；右侧 Judge & Metrics 输出 Result Score、Process Score、Trajectory Score、Safety Score、Cost；最右侧 Feedback Loop 指向 Prompt Fix、Tool Fix、Schema Fix、Model Routing、Policy Fix。白底，企业平台架构风格，中文标签。
```

评测平台也要控制成本。不是所有样本都要每天全量跑。小改动先跑 Smoke Eval，模型或语义层发布跑 Regression Eval，权限策略变更必须跑 Safety Eval，每天固定跑 Nightly Eval，重大版本上线前再跑全量 Release Eval。这样做的目标不是追求一个漂亮总分，而是让每次系统变化都有可解释、可回放、可修复的质量信号。

## 本章小结

### 关键结论

1. DataAgent 评测范式已经从静态结果评测，演进到结果、过程、轨迹、安全和持续回归并重。
2. 企业 benchmark 的价值在于覆盖真实业务约束，而不是追求题目数量最大。
3. Ground Truth 可以是精确答案、参考 SQL、执行结果、语义断言、过程标签、安全标签或专家 rubric。
4. 过程评测和轨迹评测是 DataAgent 区别于普通问答评测的关键。
5. 不同 Agent 的原始轨迹格式可以不同，但评测入口应该归一到半标准化 `eval_trace`，并从中抽取 source graph 做依赖评测。
6. BEAVER 提醒我们企业 Text-to-SQL 的难点来自复杂私有 schema、领域知识和子任务组合；Workspace-Bench 提醒我们真实 Agent 任务常常依赖复杂工作区和跨文件证据。
7. 持续评测平台要绑定模型、Prompt、工具、数据、语义层、权限策略和 Trace 版本，才能支撑长期演进。

### 上线检查清单

- [ ] 是否为 DataAgent 定义了能力边界和分层评测指标？
- [ ] 是否有覆盖核心业务场景、难度分级和版本管理的 benchmark？
- [ ] 是否把结果评测、过程评测和轨迹评测分开统计？
- [ ] 是否把不同 Agent 的原始轨迹归一成可比较的 `eval_trace`？
- [ ] 是否能从轨迹中抽取 source graph，并检查关键 source 覆盖和无关 source 噪声？
- [ ] 是否为 SQL、图表、报告、安全拒答等不同产物设计了不同 Ground Truth？
- [ ] 每次模型、Prompt、工具或语义层变更是否触发回归评测？
- [ ] 线上失败样本是否能从 Trace 沉淀到 Regression Set？
- [ ] 评测结果是否能定位到 Prompt、模型、工具、语义层、权限或数据版本？
