## 1. 在线评测到底评什么

Ch.39 讨论的是离线 benchmark：先固定样本、数据版本、工具版本和评测脚本，再比较不同模型、Prompt、工具策略或语义层版本。在线评测面对的是另一类问题：系统已经在真实用户流量里运行，用户问题、业务上下文、数据状态、权限约束和网络环境都在变化。此时不能只问“在固定测试集上得了多少分”，还要问“真实用户是否真的完成了任务”“质量退化有没有被及时发现”“修复上线后有没有产生新的副作用”。

在线评测不是离线评测的替代品，而是补充。离线 benchmark 适合做回归门禁，防止已知问题反复出现；在线评测适合发现未知问题，尤其是 benchmark 没覆盖到的新问法、新业务、新数据状态和新用户预期。企业 Agent 平台需要两者联动：线上发现问题，沉淀到离线回归集；离线验证修复，再通过在线灰度确认真实效果。

可以把评测链路理解成三层：

1. **离线评测**：固定样本、固定数据版本、固定评测脚本，适合比较版本和做发布门禁。
2. **在线评测**：观察真实流量中的用户反馈、行为指标、业务结果和系统指标，适合发现分布变化。
3. **人工复核**：对高风险、争议或自动评测不确定的样本做专家判断，适合校准 Judge 和处理事故。

```text
图片占位 Prompt 1：
生成一张“离线评测、在线评测、人工复核三层闭环”架构图。左侧是 Offline Benchmark，包括 Golden Set、Regression Set、Safety Set；中间是 Online Evaluation，包括 User Feedback、Behavior Signals、Business Metrics、Trace Sampling；右侧是 Human Review，包括 Expert Audit、Dispute Resolution、Rubric Calibration。底部有闭环箭头：Online Failure -> Benchmark Update -> Regression -> Gray Release -> Online Monitor。白底，企业技术书插图风格，中文标签，蓝灰主色，橙色强调闭环。
```

在线评测最容易犯的错误，是把用户反馈直接当成质量标签。用户点踩可能是答案错了，也可能是等待太久、表达太啰嗦、权限被拒、用户本来就问错了，或者用户想要的是 Excel 而不是自然语言解释。因此，在线评测必须结合 Ch.38 的 trace、任务类型、上下文包、工具调用、产物和权限结果来解释。

## 2. 线上反馈信号如何采集

线上反馈信号可以分为三类：显式反馈、隐式行为和业务结果。

显式反馈是用户主动告诉系统的信号，比如点赞、点踩、评分、问题标签、文本评论、人工举报。它的优点是直观，缺点是覆盖率低，而且容易受情绪影响。一个用户点踩“答案没用”，可能是 SQL 结果错，也可能是回答没有给出图表。

隐式行为是用户没有直接评分，但行为可以反映结果是否有用。比如用户是否复制答案、下载报表、继续追问、点击重新生成、放弃会话、请求人工接管、在短时间内重问同一问题。隐式行为覆盖率更高，但解释更困难。用户继续追问可能代表答案有价值，也可能代表答案没说清楚。

业务结果是 Agent 输出之后在业务系统里发生的动作。例如销售分析报告是否被发送给管理层，生成的 SQL 是否被保存为看板，客服回复是否降低工单升级率，采购分析是否触发审批动作。业务结果更接近真实价值，但滞后性强，且容易受到外部因素影响。

为了避免误读，反馈信号必须带上上下文：

```json
{
  "feedback_id": "fb_20260609_001",
  "session_id": "ses_fin_042",
  "run_id": "run_fin_042",
  "trace_id": "trace_fin_042",
  "task_type": "cashflow_root_cause",
  "feedback_type": "thumb_down",
  "user_comment": "数字不对，华东区口径有问题",
  "artifact_refs": ["chart_cashflow_042"],
  "latency_ms": 84200,
  "model_version": "gpt-5-mini-2026-06",
  "prompt_version": "finance_agent:v12",
  "semantic_layer_version": "finance_semantic:v18"
}
```

这个记录的重点不是保存一句“点踩”，而是把反馈和 `session_id`、`run_id`、`trace_id`、模型版本、Prompt 版本、语义层版本、产物引用关联起来。后续复盘才能判断问题来自模型、工具、语义层、数据口径、权限策略还是用户预期。

可以用一个简单公式表达反馈强度：

$$
FeedbackScore =
w_e \cdot Explicit
+ w_b \cdot Behavior
+ w_o \cdot Outcome
- w_r \cdot Risk
$$

其中 `Explicit` 表示显式反馈，`Behavior` 表示隐式行为，`Outcome` 表示业务结果，`Risk` 表示安全、越权、成本异常、超时等风险惩罚。这个分数不应直接作为“质量真相”，而是用于筛选需要自动评测或人工复核的样本。

## 3. 用户行为与业务指标怎么建模

在线评测要把零散行为转成可观察指标。指标不宜太多，但必须覆盖质量、效率、成本和安全。

质量指标回答“用户是否拿到了可用结果”。常见指标包括任务完成率、首次可用答案率、用户点踩率、重新生成率、人工接管率、同问题重问率。对于 DataAgent，还可以看 SQL 保存率、图表下载率、报告采纳率、结果引用率。

效率指标回答“用户花了多少成本完成任务”。常见指标包括平均交互轮数、P50 / P95 延迟、平均工具调用次数、平均 token 成本、失败重试次数。这里的 P95 是百分位指标，表示 95% 的请求都比这个时间更快，它比平均值更能反映长尾体验。

安全指标回答“系统有没有做不该做的事”。包括越权查询拦截率、敏感字段暴露率、拒答正确率、脱敏正确率、人工审批触发率和审计完整率。

业务指标回答“Agent 是否真的产生业务价值”。比如财务报告是否被下载，BI 看板是否被创建，客服工单是否减少升级，销售分析是否带来后续行动。这类指标不能完全归因给 Agent，但可以作为长期趋势观察。

在线指标要区分领先指标和滞后指标。领先指标能较早发现问题，例如点踩率、重新生成率、延迟、工具错误率；滞后指标更接近业务价值，例如报告采纳率、工单升级率、决策动作转化率。上线灰度时先看领先指标，长期运营时再看滞后指标。

一个 DataAgent 质量看板至少要能下钻到 trace。比如“现金流归因任务点踩率上升”，看板不应该停在一个百分比，而应该能继续展开：哪些租户上升、哪些模型版本上升、哪些语义层版本上升、失败集中在哪些 Step、是否同一类 Schema Linking 错误。

```text

生成一张“在线评测指标看板”示意图。左侧是质量指标：任务完成率、首次可用答案率、点踩率、重新生成率；中间是效率指标：P95 延迟、工具调用次数、token 成本；右侧是安全指标：越权拦截、脱敏正确、审计完整；底部有 Drill Down 箭头连接到 Trace Replay、Step、Tool Call、Artifact。白底，企业仪表盘风格，中文标签。
```

## 4. LLM-as-Judge 是什么，以及适合评什么

LLM-as-Judge 指用一个大语言模型充当评审器，对候选答案、报告、解释、引用或执行轨迹进行评分、比较或打标签。它的价值在于覆盖传统规则难以判断的语义质量，例如解释是否完整、报告是否有洞察、回答是否遵循指令、引用是否支持结论。

LLM-as-Judge 不等于真理。它是一种可扩展的评测工具，必须配合 rubric、参考答案、校准样本、人工抽检和版本管理使用。企业场景尤其不能让 Judge 自由发挥，因为评审标准一旦漂移，评测结果就不可比较。

常见的 Judge 模式有三类。

第一类是单答案打分。输入问题、上下文、候选答案和 rubric，让 Judge 给出分数和解释。它适合快速评估大量样本，但容易出现分数虚高、尺度不稳定的问题。

第二类是成对比较。输入同一个问题下的两个候选答案，让 Judge 判断 A 更好、B 更好或打平。它比绝对打分更稳定，适合比较两个模型版本或 Prompt 版本。但要随机化答案顺序，否则容易出现位置偏差。

第三类是多维 rubric 评分。把质量拆成多个维度，例如正确性、完整性、证据引用、指令遵循、可读性、安全合规。它更适合企业 DataAgent，因为可以把“报告写得好”拆成可诊断的子分数。

一个企业 DataAgent 的 Judge 输入可以这样组织：

```json
{
  "question": "本月经营性现金流为什么下降？",
  "task_type": "root_cause_analysis",
  "context_summary": "用户要求按区域分析本月经营性现金流下降原因。",
  "candidate_answer": "...",
  "evidence": {
    "sql_result_summary": "华东区贡献下降 62%，主要来自应收账款回款延迟。",
    "artifact_refs": ["chart_cashflow_042"],
    "source_graph_summary": "使用 finance_semantic:v18, cashflow_fact, org_dim"
  },
  "rubric": {
    "correctness": "结论必须与 SQL 结果一致。",
    "grounding": "关键结论必须能被证据支持。",
    "actionability": "必须说明可采取的下一步分析或动作。",
    "safety": "不得暴露客户明细和无权限字段。"
  }
}
```

评分输出也要结构化：

```json
{
  "overall_score": 0.82,
  "dimension_scores": {
    "correctness": 0.90,
    "grounding": 0.85,
    "actionability": 0.70,
    "safety": 1.00
  },
  "confidence": 0.76,
  "failure_tags": ["missing_next_step"],
  "rationale": "结论与 SQL 摘要一致，但行动建议较弱。"
}
```

对 PM 来说，最重要的不是 Judge 模型叫什么，而是 Judge 输出能不能指导动作。`correctness` 低，优先看数据和工具；`grounding` 低，优先看引用和 source graph；`actionability` 低，优先看产品输出模板；`safety` 失败，直接阻断发布。

可以用多维评分表达 Judge 结果：

$$
JudgeScore =
\sum_{d \in D} w_d \cdot score_d
$$

其中 `D` 是 rubric 维度集合，`w_d` 是每个维度的权重。不同任务权重不同：财务分析要提高正确性和安全权重，研究报告要提高证据和完整性权重，客服回复要提高可执行性和语气一致性权重。

## 5. Judge 的偏差与一致性控制

LLM-as-Judge 很有用，但风险也很明确。它可能偏好更长的回答，偏好更像自己风格的回答，偏好排在前面的答案，或者被华丽表达掩盖事实错误。它还可能缺少领域知识，对企业指标口径、权限边界和数据事实判断不准。

常见偏差包括：

1. **位置偏差**：成对比较时，Judge 更偏好第一个或第二个答案。
2. **长度偏差**：更长的回答看起来更完整，实际可能没有更多有效信息。
3. **风格偏差**：表达更流畅的答案获得更高分，即使事实不更准确。
4. **自偏好**：Judge 可能偏好与自身模型风格相近的答案。
5. **参考泄漏**：如果参考答案写法过强，Judge 可能奖励表述相似，而不是语义正确。
6. **领域盲区**：Judge 不理解企业口径，无法识别指标混用。

控制方法也要工程化。

成对比较要随机化答案顺序，并记录 `order_seed`。如果同一对答案换顺序后判断反转，就降低置信度。

Rubric 要版本化。每次修改评分标准，都要产生 `rubric_version`，旧分数不能和新分数直接混用。

Judge 要用黄金样本校准。黄金样本来自专家标注，覆盖正确、部分正确、错误、越权、引用不足、表达啰嗦等典型情况。每次更换 Judge 模型或 Prompt，都先跑黄金样本。

高风险样本要人工抽检。安全、财务、合规、人事、客户数据类任务，不应完全依赖自动 Judge。

多个 Judge 可以交叉验证。比如一个模型评正确性，一个模型评安全，规则引擎评权限，专家抽检高风险样本。不要让单个 Judge 成为唯一裁判。

可以用一致性指标监控 Judge 稳定性：

$$
JudgeAgreement =
\frac{\text{Judge 与专家一致的样本数}}
{\text{抽检样本总数}}
$$

如果 `JudgeAgreement` 持续下降，说明 Judge Prompt、模型版本、任务分布或 rubric 可能发生漂移。此时应先修评测系统，而不是直接根据 Judge 分数优化 Agent。

```text
图片占位 Prompt 3：
生成一张“LLM-as-Judge 偏差与控制”示意图。左侧列出偏差：位置偏差、长度偏差、风格偏差、自偏好、领域盲区；中间是控制手段：顺序随机化、Rubric 版本、黄金样本校准、多 Judge 交叉、人工抽检；右侧是输出：可信分数、置信度、失败标签、人工复核队列。白底，流程图风格，中文标签。
```

## 6. A/B 实验与能力验证

离线评测和 Judge 分数只能说明“在样本上可能更好”，不能直接证明真实用户会受益。线上能力变化需要 A/B 实验验证。A/B 实验是把真实流量按规则分成两组或多组，一组使用旧方案，一组使用新方案，然后比较主指标和护栏指标。

Agent 场景下，实验对象可以是模型、Prompt、工具策略、语义层版本、检索策略、缓存策略、模型路由或安全策略。比如把 `prompt:v12` 升级到 `prompt:v13`，离线 benchmark 通过后，只给 5% 流量灰度，看点踩率、任务完成率、P95 延迟、token 成本和安全拦截是否变化。

实验设计要先确定主指标。主指标是本次实验真正想改善的目标，例如首次可用答案率、任务完成率、报告采纳率。然后设置护栏指标。护栏指标是不允许明显变差的指标，例如安全失败率、P95 延迟、平均成本、人工接管率。一个实验不能只因为主指标提升就上线，如果安全或成本明显恶化，应回滚。

多轮 Agent 会让归因更复杂。一次用户会话可能跨多个 Run，用户第 3 轮的满意度可能受第 1 轮影响。因此实验分桶最好按用户或会话隔离，而不是按单次请求随机。否则同一个用户在同一段会话里同时遇到 A 方案和 B 方案，会污染体验和数据。

长任务也要注意延迟归因。一个报告生成任务可能几分钟后才完成，用户是否下载报告可能更晚发生。实验窗口要足够长，不能只看请求完成瞬间的指标。

可以用一个简化实验判定规则：

$$
Launch =
\begin{cases}
1, & \Delta Primary > \tau_p \ \text{且所有 Guardrail 通过} \\
0, & \text{否则继续灰度、修复或回滚}
\end{cases}
$$

其中 `Primary` 是主指标，`Guardrail` 是护栏指标，`\tau_p` 是最小有效提升阈值。这个公式的意思很朴素：上线不是看有没有一点提升，而是看提升是否足够大，并且没有破坏安全、成本和稳定性。

除了传统 A/B，也可以使用 interleaving。Interleaving 可以理解为“交错比较”：把两个候选结果以混合方式展示，让用户行为更直接地反映偏好。它更常见于搜索和推荐场景。对 DataAgent 来说，interleaving 不适合所有任务，但可以用于候选图表、候选摘要、候选解释的内部评测或受控用户研究。

## 7. 评测驱动的持续优化闭环

在线评测的最终目标不是追分，而是让每次优化都有证据、有边界、有回归保护。

一个完整闭环如下：

1. 线上运行产生 Trace、反馈、行为指标、业务结果和成本指标。
2. 系统自动筛选异常样本，例如点踩、高成本、超时、人工接管、安全拦截。
3. LLM-as-Judge 对低风险样本做自动评测，高风险样本进入人工复核。
4. 失败样本按根因聚类，例如 Schema Linking、工具参数、上下文遗漏、权限拒绝、报告质量。
5. 高价值样本沉淀到 Ch.39 的 Regression Set 或 Safety Set。
6. 团队修复 Prompt、工具描述、语义层、模型路由、权限策略或产品交互。
7. 离线回归通过后，进入小流量灰度。
8. 在线指标稳定后扩大流量，并持续观察漂移。

```text
图片占位 Prompt 4：
生成一张“在线评测驱动的持续优化闭环”图。节点包括 Online Trace、User Feedback、Behavior Metrics、LLM Judge、Human Review、Failure Clustering、Regression Set、Fix Prompt/Tool/Semantic Layer、Offline Regression、A/B Gray Release、Online Monitor。画成闭环箭头，白底，企业工程风格，中文标签。
```

这套闭环里有一个关键边界：Judge 负责发现和分类问题，不负责自动决定所有修复。比如 Judge 发现报告缺少引用，修复动作可能是 Prompt 模板调整，也可能是工具返回缺少 source，也可能是前端没有展示引用。工程团队需要结合 trace 判断真正责任域。

还要避免“为了 Judge 得分而优化”。如果团队只追求 JudgeScore，很容易把回答写得更长、更像 rubric、更讨 Judge 喜欢，但真实用户并不更满意。在线评测必须同时看用户行为、业务结果、安全护栏和成本指标。

## 8. 企业落地示例：DataAgent 报告质量优化

假设财务 DataAgent 上线后，现金流归因报告的点踩率从 6% 上升到 13%。如果只看点踩，团队不知道问题在哪。正确做法是把在线反馈和 trace 结合起来。

第一步，看指标分布。点踩集中在“经营性现金流归因”任务，主要出现在 `prompt:v14` 发布后，P95 延迟没有明显变化，工具错误率也没有明显上升。

第二步，抽样回放 trace。发现新 Prompt 让 Agent 更积极生成管理层摘要，但没有强制引用 SQL 结果中的区域贡献度。报告写得更完整，却经常漏掉“华东区贡献最大”的关键结论。

第三步，用 LLM-as-Judge 做批量评测。Rubric 拆成正确性、证据引用、行动建议和安全四项。Judge 显示正确性下降不明显，但证据引用分明显下降。人工抽检确认问题成立。

第四步，沉淀样本。把 30 个典型失败 trace 加入 Regression Set，标注关键断言：“必须按区域说明贡献度”“必须引用 SQL 结果摘要”“不得只给泛化建议”。

第五步，修复和回归。团队修改 Prompt 和报告模板，要求每个归因结论绑定 source。离线回归通过后，小流量灰度。

第六步，在线观察。灰度组点踩率下降到 7%，报告下载率上升，P95 延迟和 token 成本小幅增加但在护栏范围内。最终扩大流量。

这个例子说明，在线评测的价值不在于多一个分数，而在于把“用户觉得不好”拆成可定位、可修复、可回归的工程问题。

## 本章小结

### 关键结论

1. 在线评测补充离线 benchmark，用来发现真实流量中的未知问题和分布漂移。
2. 线上反馈不是天然标签，必须结合 trace、任务类型、业务上下文和版本信息解释。
3. LLM-as-Judge 适合补充语义评测，但必须有 rubric、校准样本、一致性控制和人工抽检。
4. Judge 分数不能单独决定上线，A/B 实验要同时设置主指标和护栏指标。
5. 在线评测的价值在于把线上失败样本重新喂回离线评测、回归集和持续优化流程。
