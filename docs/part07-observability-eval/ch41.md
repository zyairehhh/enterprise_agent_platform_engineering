## 1. 从一张异常账单开始

很多团队第一次认真讨论 Agent 成本，不是在系统设计会上，而是在月底账单会上。一个财务 DataAgent 刚上线时看起来效果不错：用户可以问“本月经营性现金流为什么下降”，系统会查数据、生成 SQL、解释指标、绘图并输出报告。可是两个月后，财务部门发现平台账单明显上升，业务侧却没有感觉到能力也同比例提升。

这时如果只看模型账单，结论很容易走偏。团队可能会说“把大模型换成小模型”“把输出限制短一点”“少跑几次 Judge”。这些动作都可能省钱，但也可能让现金流归因变差、让 SQL 错误变多、让人工复核压力上升。Agent 成本治理的难点就在这里：一次用户请求背后不是一次模型调用，而是一条由模型、工具、检索、存储、评测、重试组成的任务链。

Ch.38 讲 trace 时提到，平台要用 `session_id`、`run_id`、`step_id`、`trace_id` 把一次任务串起来。到了成本治理里，这些字段不再只是排障字段，也变成了成本归因字段。没有它们，平台只能知道“模型账单变高了”；有了它们，平台才能知道是哪个租户、哪个 Agent、哪个任务类型、哪个 Step 把成本打上去了。

一条 DataAgent 任务的总成本，可以先粗略拆成：

$$
Cost_{run} =
\sum_i Cost_{model_i}
+ \sum_j Cost_{tool_j}
+ Cost_{retrieval}
+ Cost_{storage}
+ Cost_{eval}
+ Cost_{retry}
$$

这里的 `model` 不只是最终回答那一次调用，也包括意图识别、SQL 生成、错误修复、报告摘要、Judge 评测等调用。`tool` 也不只是 SQL，还可能包括浏览器、代码执行器、文件解析器、BI 系统、外部 SaaS API。`retry` 常常被忽略，但在真实系统里，失败 SQL 的重试、模型超时后的重跑、用户重复点击造成的重复任务，都会把账单放大。

模型调用本身又可以拆成输入、输出和缓存折扣：

$$
Cost_{model} =
P_{in} \cdot T_{in}
+ P_{out} \cdot T_{out}
- Discount_{cache}
$$

`P_in` 和 `P_out` 是输入、输出 token 单价，`T_in` 和 `T_out` 是 token 数量，`Discount_cache` 是缓存命中带来的节省。对于自建推理服务，公式会变成 GPU 时间、吞吐、利用率和运维成本的组合，但治理思路相同：先解释每一次 Run 为什么花钱，再讨论怎样省钱。

## 2. 成本治理第一步是归因，不是降价

回到前面的财务 DataAgent。账单上涨以后，正确的第一步不是立刻换模型，而是把账单和链路对齐。也就是说，团队要能回答几个很具体的问题：本月哪些租户成本最高？哪些 Agent 成本增长最快？一个成功 Run 平均花多少钱？失败 Run 占了多少成本？成本集中在模型调用、SQL、检索、工具还是 Judge？

这些问题看起来像财务问题，实际是工程问题。因为平台必须在每个关键 Step 记录成本事件，并把它和 trace、prompt 版本、模型版本、工具版本、语义层版本关联起来。一个成本事件可以写成这样：

```json
{
  "cost_event_id": "cost_20260609_001",
  "tenant_id": "tenant_finance",
  "agent_id": "dataagent_finance",
  "run_id": "run_cashflow_042",
  "step_id": "step_sql_generate",
  "trace_id": "trace_cashflow_042",
  "cost_type": "model_call",
  "model": "gpt-5-mini",
  "input_tokens": 8420,
  "output_tokens": 620,
  "cache_hit_tokens": 5100,
  "estimated_cost_usd": 0.018,
  "prompt_version": "finance_agent:v12",
  "semantic_layer_version": "finance_semantic:v18"
}
```

这条记录的价值，不是让财务能多一行明细，而是让 Ch.38 的 trace 回放、Ch.39 的 benchmark、Ch.40 的在线评测可以和成本连起来。比如某个 Prompt 版本让现金流归因准确率提升了 1%，但平均 token 增加了 80%；又比如某个 SQL 工具版本让重试率下降了 30%，但单次查询变慢了。没有成本归因，团队只能凭感觉讨论；有了归因，团队才能比较“质量收益”和“成本代价”。

成本看板也不要只展示总金额。更有决策价值的是单位任务成本，例如每个成功 Run 的平均成本、每个被采纳报告的平均成本、每个租户每周成本、失败 Run 占成本比例、缓存命中节省金额。如果 30% 的账单花在失败重试上，优化方向通常不是换便宜模型，而是修工具错误、超时策略和重试策略。

这一节的核心逻辑是：降价是后面的动作，归因才是前置条件。否则平台很容易把显性成本压下去，却把隐性风险转移到业务侧。

## 3. 模型路由：让强模型用在值得用的地方

当成本归因清楚以后，团队通常会发现一个现象：并不是所有任务都需要同一个模型。指标定义解释、字段说明、FAQ 这类低风险任务，答案稳定、上下文短、可缓存，使用小模型通常足够。现金流下降归因、预算异常解释、人事或客户数据分析这类任务，涉及更长上下文、更高风险和更强推理能力，就需要更强模型、完整 trace 和更严格评测。

这就是模型路由的出发点。模型路由不是“永远选择最便宜模型”，而是“把合适的模型用在合适的任务上”。路由器需要参考任务类型、风险等级、上下文长度、延迟目标、预算状态和历史效果。它可以先用小模型做意图识别和风险判断，再决定后续走小模型、本地模型、强模型，还是进入人工审批。

一个简单的路由规则可以这样表达：

```yaml
routes:
  - name: low_risk_metric_explain
    when:
      task_type: metric_explanation
      risk_level: low
      max_context_tokens: 4000
    use_model: gpt-5-mini
    fallback_model: gpt-5

  - name: finance_root_cause
    when:
      task_type: root_cause_analysis
      domain: finance
      risk_level: high
    use_model: gpt-5
    require_judge: true
    require_trace: true

  - name: batch_summary
    when:
      task_type: report_summary
      latency_class: batch
    use_model: local-32b
    fallback_model: gpt-5-mini
```

更工程化一点，可以把路由看成一个效用函数：

$$
Utility(model, task) =
\alpha \cdot Quality
- \beta \cdot Cost
- \gamma \cdot Latency
- \delta \cdot Risk
$$

这个公式只是示意，不是要求每个团队都机械打分。它提醒我们，模型选择同时受质量、成本、延迟和风险约束。尤其要注意 `Risk` 不能只当作平均分里的一个扣分项。涉及越权、合规、人事薪酬、客户隐私时，风险是门禁条件；便宜模型即使看起来效用高，也不能绕过安全策略。

模型路由还必须接入评测系统。每次调整路由规则，都要跑 Ch.39 的 Regression Set 和 Safety Set，再通过 Ch.40 的在线灰度观察真实反馈。否则路由很容易在局部省钱，却造成质量退化。

## 4. 缓存不是一个开关，而是几层复用

成本治理里最容易被误解的词是“缓存”。很多人会问：“能不能加个缓存，把成本降下来？”这个问法太宽了。Agent 平台里至少有几类缓存，它们复用的对象不同，失效条件也不同。

Prompt Cache 更偏应用层。系统提示词、工具说明、稳定 schema、指标字典、权限策略说明、租户配置等内容，如果长期不变，可以作为可复用上下文片段。它解决的是“这些应用上下文是否需要每次重新拼装、传输和计费”的问题。

Prefix Cache 和 KV Cache 更偏推理服务层。Transformer 模型生成时会保存前文计算结果，也就是常说的 Key-Value Cache。如果多个请求共享相同 token 前缀，推理服务可以复用前缀计算，降低首 token 延迟和计算成本。vLLM、SGLang 等推理系统都会围绕 prefix caching、paged attention、KV Cache 管理来提升吞吐。

Result Cache 复用的是确定结果。比如同一个 SQL 查询、同一个低风险指标解释、同一份固定报表说明，在版本和权限都相同的情况下可以直接复用。Semantic Cache 则更进一步，它判断两个问题语义是否足够接近，从而复用已有答案、SQL 模板或分析路径。

在企业 DataAgent 里，缓存失效条件比普通 Web 缓存更复杂。Prompt 版本变化、工具版本变化、schema 变化、指标口径变化、租户和角色变化、数据快照变化、安全策略变化，都可能让缓存不能复用。因此缓存 key 不能只用 prompt 文本 hash，还要带上版本和权限上下文：

```json
{
  "cache_key": "sha256(...)",
  "tenant_id": "tenant_finance",
  "prompt_version": "finance_agent:v12",
  "tool_version": "sql_tools:v5",
  "semantic_layer_version": "finance_semantic:v18",
  "policy_version": "finance_policy:v7",
  "role": "finance_manager",
  "data_snapshot": "warehouse:2026-06-09"
}
```

如果缓存没有权限隔离，就可能把 A 租户或高权限用户的上下文复用给 B 租户或低权限用户。这不是性能瑕疵，而是安全事故。所以在企业场景里，缓存命中率不是越高越好。真正要追求的是“在权限、版本、时间窗口和数据新鲜度都正确的前提下命中”。

## 5. Semantic Cache：最应该保守的一层缓存

Semantic Cache 可以翻译为语义缓存。它不是要求两个问题字面完全相同，而是判断它们语义上是否足够接近。比如“经营性现金流的定义是什么”和“什么叫经营性现金流”，大概率可以复用同一条指标解释。又比如“本月现金流为什么下降”和“这个月现金流减少的原因”，语义上相似，但能不能复用就要谨慎得多，因为时间窗口、租户、权限、数据快照和上下文都可能不同。

语义缓存通常需要同时满足三个条件：

$$
CacheHit =
Similarity(q, q') > \tau_s
\land Fresh(data) = true
\land PolicyAllowed(user, result) = true
$$

`Similarity` 是语义相似度，`\tau_s` 是阈值，`Fresh` 表示数据是否仍然新鲜，`PolicyAllowed` 表示当前用户是否有权看到缓存结果。这个公式的重点不在相似度，而在后两个条件。很多缓存事故不是因为相似度算错，而是因为忽略了数据新鲜度和权限。

在 DataAgent 里，最稳妥的做法通常不是直接复用最终答案，而是分层复用。低风险的指标定义、固定说明、公开 FAQ 可以直接复用；SQL 模板可以复用，但查询要重新执行；分析路径可以复用，但数据要重新读取；高风险场景下，缓存结果最好只作为候选提示，不直接返回给用户。

这样做看起来保守，但很符合企业场景。最终答案最容易受时间、数据快照和权限影响；SQL 模板和分析计划相对稳定，更适合作为复用对象。也就是说，Semantic Cache 的目标不是“尽可能返回旧答案”，而是“尽可能复用已经验证过的中间能力”。

## 6. Token 预算：让系统在花钱前做决定

Token 是大模型成本和延迟的核心变量。输入 token 越多，模型需要处理的上下文越长；输出 token 越多，生成时间和费用越高。Agent 场景里 token 增长尤其快，因为系统提示词、工具说明、schema、历史对话、Memory、检索文档、工具结果都可能进入上下文。

预算控制不能等账单出来以后再做。更合理的方式是在 Run 执行前和关键 Step 执行前做估算：生成长报告前估算 token，执行大 SQL 前估算扫描量，调用 Judge 前判断是否需要全量评测。预算可以按租户、用户、Agent、场景、项目和时间窗口设置，例如某租户每月最多 5000 美元，某低优先级 Agent 每天最多 50 万 token，某交互式请求最多 30 秒和 20K token。

预算控制也不应该只有“允许”和“拒绝”。接近预算时可以提示管理员；上下文过长时可以压缩摘要、减少检索文档数；高峰期可以切小模型、关闭非关键工具、转异步任务；高成本任务可以要求人工审批；超过硬限制或违反策略时才拒绝执行。

一个预算策略可以这样表达：

```yaml
budget_policy:
  tenant_id: tenant_finance
  monthly_usd_limit: 5000
  per_run_token_limit: 50000
  warning_threshold: 0.8
  actions:
    - when: monthly_usage_ratio > 0.8
      action: notify_admin
    - when: per_run_estimated_tokens > 30000
      action: summarize_context
    - when: per_run_estimated_cost_usd > 2.0
      action: require_approval
    - when: monthly_usage_ratio > 1.0
      action: block_low_priority_tasks
```

预算还要和 Ch.42 的 SLO 联动。高优先级任务可以保留更多预算和更强模型，低优先级任务在高峰期可以排队或降级。预算不是单纯的财务控制，而是稳定性控制的一部分。

## 7. 成本优化的取舍顺序

成本优化最危险的做法，是只看省了多少钱。更便宜的模型可能降低任务成功率，更短的上下文可能丢失关键证据，更激进的缓存可能返回过期答案，更少的重试可能降低可用性。省下来的钱，可能会以用户点踩、人工接管、错误决策和合规风险的方式还回来。

工程上可以采用一个简单顺序：先守住安全和正确性，再守住可用性和延迟，最后优化成本。越权、泄漏、关键口径错误不能用成本节省来交换；交互式任务至少要让用户尽快看到进度或可用结果；只有在质量和稳定性护栏内，才应该压降模型、上下文、工具和评测成本。

可以把优化动作放进一个门禁公式：

$$
AllowedOptimization =
Quality \ge Q_{min}
\land Safety = pass
\land Latency \le L_{max}
\land CostReduction > \epsilon
$$

这个公式表达的是上线条件：质量、安全和延迟都满足底线时，成本优化才成立。否则它不是优化，而是风险转移。对于低风险高频任务，例如指标解释、字段说明、固定 FAQ，可以更积极地使用缓存和小模型；对于高风险低频任务，例如财务归因、人事数据、合规报告，应优先使用强模型、完整 trace 和人工复核。

## 8. 案例回放：财务 DataAgent 成本下降

现在回到开头那张异常账单。团队把成本事件和 trace 对齐后发现，现金流归因任务占总成本的 42%。其中大部分成本来自两类问题：稳定 schema 和工具说明每次都被完整塞进上下文；SQL 生成失败后，Agent 会重复生成类似错误 SQL，导致多次重试。

第一轮优化没有换模型，而是先做上下文复用。团队把稳定系统提示词、工具说明、财务 schema 和指标字典拆成可复用片段，为同一租户、同一语义层版本建立 Prompt Cache 和 prefix 复用。这样做以后，输入 token 明显下降，但任务逻辑没有改变。

第二轮优化处理 SQL 重试。Trace 显示很多失败来自 Join Key 错误，模型在没有足够工具反馈时会反复猜测。团队把 Ch.39 中的 Join Key 过程标签加入回归集，优化工具错误反馈，并让 Agent 在第二次重试时优先使用语义层推荐 Join，而不是自由猜测。

第三轮优化调整 Judge 策略。原来所有报告都跑完整 LLM-as-Judge，现在改成分层评测：低风险、缓存命中的指标解释只做规则检查；高风险财务归因保留完整 Judge；灰度版本提高抽样比例。这样既保留了质量护栏，又减少了不必要的评测成本。

灰度上线后，成本下降 35%，P95 延迟下降 22%，Regression Set 质量没有下降，线上点踩率保持稳定。这个案例说明，成本治理不是“换便宜模型”这么简单。真正有效的路径是先归因，再路由，再缓存，再预算控制，最后通过评测和灰度证明质量没有退化。

## 本章小结

### 关键结论

1. Agent 成本要按任务链路归集，不能只统计单次模型调用。
2. 模型路由的目标是“合适任务用合适模型”，不是简单选择最便宜模型。
3. Prompt Cache、Prefix Cache、KV Cache、Semantic Cache 解决的问题不同，失效条件也不同。
4. 预算控制要和租户、权限、SLO、评测指标联动。
5. 成本优化必须保留质量和安全护栏，否则会把隐性风险转移到业务侧。
