# Chapter 31 Framework Cross-Comparison

---

The preceding chapters used `mini-platform` to build Runtime, Tool Registry, MCP adaptation, multi-Agent Handoff, and HITL from the bottom up. In real enterprises, business teams often use several tools at the same time: Dify for knowledge-base Q&A, Coze for campaign Bots, or LangGraph in notebooks for SQL analysis experiments. At this point, the technical committee needs to decide which layer each tool should own. The question is not which tool is strongest.

If Dify already exists, the enterprise still has to decide who owns Runtime. If LangGraph defines a Planner graph, the enterprise still has to decide who owns the Run six-state lifecycle. If a Coze Bot faces users, the enterprise still has to decide whether database access and write operations pass through unified governance. From a feature checklist perspective, these products can all "run Agents." Once production governance enters the picture, they cover different layers.

This chapter uses one selection standard: frameworks and application platforms can accelerate experimentation and delivery, but production write operations, tool versions, approvals, audits, and multi-tenant boundaries must return to the enterprise platform. A framework that can plug into the unified Runtime, Registry, Policy, and Trace is an accelerator. A framework that closes the loop inside its own runtime becomes a new production silo.

This judgment does not dismiss existing tools. Building every Planner, Console, Bot channel, and knowledge base from scratch can slow down business pilots. The risk is layer misalignment: using a low-code platform for group-level audit, defining production permission in a notebook tool, or connecting a Bot plugin directly to a core database. These shortcuts look convenient during pilots and concentrate risk at production scale.

Framework selection is easily distorted by feature lists. LangGraph can draw state graphs, AutoGen can organize multi-Agent conversations, CrewAI can express roles and tasks, and Dify or Coze can help business teams create applications quickly. If the review compares only node types, plugin counts, and UI convenience, the conclusion will be that every product can do the job. The real difference is production responsibility: who owns tool permissions, who persists Run state, who handles approval timeout, who exports audit bundles, and who can replay a task after an incident.

Most enterprises will not choose only one route. The data team may continue to use LangGraph for complex analysis flows, operations may use a low-code platform for campaign assistants, the platform team may maintain Runtime and Registry, and the security team may own policy and audit. This mix is acceptable when boundaries are explicit. A framework can implement Planner logic, a low-code platform can serve as an entry point, and a vendor Bot can provide a channel. Write operations, data permissions, approvals, and Trace must still return to the enterprise foundation.

The goal of this chapter is to provide executable questions for selection. Ask which layer a tool occupies before asking how many features it has. Ask whether tool calls can be forced through Registry before asking whether the canvas is convenient. Ask whether logs can return to Trace before asking whether it supports another Agent mode. When the layers are clear, frameworks accelerate delivery. When the layers are unclear, they become another platform to govern.

## 31.1 Responsibility Layers for Frameworks, Platforms, and Applications

### 31.1.1 Framework, Platform, and Application

A framework is a development library into which business code is embedded, such as LangGraph's graph, AutoGen's multi-Agent conversations, and CrewAI's roles and tasks. A platform is the enterprise's unified runtime and governance layer, such as `/run`, the Run Six States, Tool Registry, Policy, HITL, Trace, and Console. An application is an Agent oriented to specific roles or business processes, such as DataAgent, customer service assistants, and business analysis Workflow Agents.

*Table 31-1: Responsibilities of the Framework, Platform, and Application layers. Source: compiled in this book.*

| Layer      | What Problem It Solves                          | Typical Products                          | Primary Maintainers      |
|------------|------------------------------------------------|------------------------------------------|-------------------------|
| Framework  | How to organize Planners, tools, roles, and multi-Agent flows | LangGraph graph, CrewAI crew, AutoGen chat | Application or data teams |
| Platform   | How to unify runtime, governance, auditing, and recovery         | `/run`, Registry, Policy, Trace, Console   | Platform teams           |
| Application| How to serve specific roles and processes                      | DataAgent, customer service Agents, report Agents | Business lines and platform teams |

These three layers can coexist, but their responsibilities must not be mixed. A framework can serve as the implementation of a Planner; low-code platforms may act as business entry points or operation consoles. However, all actions that modify business state must pass through the enterprise Runtime and Registry.

### 31.1.2 Production Environment Constraints

First, all production write operations must go through platform Policy and Registry. Regardless of whether the Planner is LangGraph, CrewAI, or custom-built, actions like sending emails, writing tickets, executing SQL, and modifying master data cannot be scattered inside the framework itself. Second, framework state is not equivalent to the Run Six States. LangGraph nodes, Dify Workflow instances, and Coze Bot conversations can all map to a `run_id`, but external SLAs, approvals, checkpoints, and auditing should be based on the Run. Third, if a low-code application platform serves as a production entry point, it must integrate with the enterprise Gateway, Registry, and Trace. Otherwise, the enterprise will maintain multiple sets of permissions, logs, tool connections, and rollback methods simultaneously.

### 31.1.3 Runtime Responsibility Beyond Low-Code Platforms

Purchasing Dify does not mean you have the Runtime described in Chapter 22. Dify has its own runtime and Workflow, but whether it meets enterprise multi-tenancy, IAM, approval, auditing, and tool version requirements must be verified item by item. LangGraph also does not allow teams to skip the Tool Registry. In development environments, tools may be defined directly in notebooks with `tools=[]`, but in production environments, tools must be registered as versioned assets with unified authentication, auditing, and gradual rollout. Allowing each business line to choose its own framework freely does not equal technical diversity. Frameworks can be varied, but platform interfaces must be unified. Otherwise, the more business there is, the higher the governance cost.

When reviewing a framework, first ignore marketing claims and instead ask three questions: Can it be included in the enterprise `/run`? Can its tool calls be forced through the Registry? Can its state and logs enter Trace and Export Bundle? Only after answering these can you reasonably discuss development experience and ecosystem plugins.

Clear answers protect business teams as well. Business teams can keep using familiar tools for prototypes instead of waiting for the platform to become perfect. The platform team can gradually take over the high-risk parts through unified interfaces. A mature hybrid architecture does not replace everything in one step. It leaves low-risk innovation in frameworks and pulls production responsibility back into the platform core.

Framework evaluation also has to include organizational capability. Building a platform in-house requires long-term ownership of Runtime, Registry, Console, evaluation, permissions, and deployment. Buying or adopting open-source frameworks can accelerate the start, but the enterprise accepts their runtime model, extension boundary, and upgrade pace. A team without platform engineering capacity can slow the business by overbuilding. A team with strict governance needs can pay later if it hands production permission to a low-code application. Selection should follow organizational capability and risk constraints, not preference.

A hybrid architecture often lands in three steps. First, let business teams keep existing frameworks in low-risk scenarios while routing tool calls through the enterprise Registry. Second, pull high-risk actions, approvals, Trace, and Export Bundle into the unified Runtime. Third, consolidate Console, evaluation, and operational reports. This sequence is more practical than replacing every tool at once and lets business lines keep producing during migration.

Vendor products need acceptance criteria as well. The review should verify whether the product supports enterprise IAM, exports full execution logs, disables built-in tools from directly connecting to core systems, supports private deployment or data-residency constraints, maps to enterprise approval workflows, and allows platform-side custom evaluation. These issues rarely appear in demos, so they must be checked during contracting and pilots.

The output of framework comparison should be a responsibility map instead of a ranking table. The map should state which capabilities remain in the framework, which move into the platform, which are owned by vendors, and which internal teams must fill. Once this map is clear, new frameworks, new business applications, and new suppliers can be reviewed with the same questions instead of restarting the debate.

The responsibility map changes by stage. During pilots, frameworks may own more runtime details because the user group is small, the data scope is controlled, and the risk is limited. In production, tool permissions, approval, audit, and SLO should move back toward the enterprise platform. At scale, onboarding, evaluation, cost, and compliance also need unified operation. If a team freezes the pilot-stage responsibility split, the second or third scenario will quickly expose governance gaps.

Technical committees should ask every proposal who handles failure. If a LangGraph subgraph fails, who can recover the Run? If a Dify workflow calls an external API, who can export the audit record? If a Coze Bot hits a sensitive field, who can intercept and review it? If the internal Runtime scheduler fails, who owns the SLO? These questions reveal architecture boundaries more clearly than node counts.

Framework adoption also needs an exit path. A low-code platform may fit the current entry point and later be replaced because of data boundaries, cost, private deployment, or audit requirements. If tools, state, logs, and artifacts can all map to the enterprise platform, exit cost stays controlled. If core business logic and permissions are locked inside a single product, migration becomes a rebuild. Writing the exit path during selection is part of responsible platform design.

## 31.2 How to Use Open Source Frameworks

### 31.2.1 LangGraph

At the core of LangGraph are directed graphs, state objects, and checkpointers. It is well-suited for expressing complex branching, loops, reflection, and manual interruption. The Planner in DataAgent can use LangGraph to define subgraphs for "table selection, SQL generation, execution, reflection, and correction"; the internal state of these subgraphs is rich, but externally they should still be folded into Runtime states like `planning`, `executing`, `waiting_human`, `succeeded`, or `failed`.

The recommended approach is to place LangGraph at the Planner layer, instead of making it an independent production entry point. The `thread_id` can map to the `run_id`, graph node tool calls should route to the enterprise Registry, and `interrupt()` can map to `waiting_human`. This way, teams can leverage LangGraph's expressive power while preserving the tool governance and audit boundaries of the enterprise platform.

This embedded usage also reduces replacement costs. The Planner graph can be iteratively adjusted as tasks evolve, while the outer Runtime events, error codes, approvals, and checkpoints remain unchanged. Business teams still see the same Agent application, and the platform team sees the same execution chain, so framework changes do not directly affect audit and SLO.

```python
# Concept example: LangGraph as a Planner plugin, not an independent Run
def next_step(ctx: PlannerContext) -> PlannerOutput:
    graph = build_sql_graph(registry=ctx.registry)
    thread_id = ctx.run_id
    for event in graph.stream(ctx.input, config={"thread_id": thread_id}):
        emit_to_runtime_sse(event)  # Fold into state / action / result
    return graph.get_final_output()
```

### 31.2.2 AutoGen

AutoGen excels at multi-Agent conversations, code execution, and research exploration. It is suitable for quickly validating how multiple roles collaborate during the Lab phase and for prototyping GroupChat or Swarm-style task decomposition. However, in production, free-form group chats bring token cost, tool boundary, and auditing challenges.

The production recommendation is to converge AutoGen experiments into explicit Handoff contracts. Agents do not chat indefinitely but pass tasks, input schemas, and return results through the Handoff Tool introduced in Chapter 28. This preserves the multi-Agent experimental insights while avoiding bringing free dialogue directly into production Runtime.

For example, during experiments, "Analyst Agent" and "Report Agent" can freely discuss in AutoGen to see how they break down problems. In production, this discussion becomes two explicit steps: DataAgent returns structured metrics and evidence, and Report Agent drafts the report based on these inputs. Whether human review is needed is decided by Runtime and Policy.

### 31.2.3 CrewAI

CrewAI's Role / Task / Crew abstractions are well-suited for expressing job responsibilities and closely resemble the multi-Agent narrative of Chapter 28. Business teams find it easy to describe role chains such as "data analyst, report writer, reviewer" using CrewAI. It is better used as a role configuration or Planner organization method instead of as a replacement for the enterprise Runtime. Crew role definitions, goals, and tool whitelists can be exported as platform AgentSpecs; execution is still managed by RunLoop for state, Registry for tools, and Policy for approvals. This usage lets business users describe workflows using role language while engineers avoid introducing a second runtime. Role configurations become business assets, and RunLoop remains the platform asset. Separating the two means role changes won't break state machines or tool audits.

## 31.3 How to Use Application Platforms

### 31.3.1 Dify

Dify is suitable for quickly building workflows, RAG (Retrieval-Augmented Generation), knowledge Q&A, and internal applications. Its advantages lie in visual orchestration, self-hosting, and a rich plugin ecosystem. Enterprises can use Dify as an early-stage experimental environment or for certain business entry points: business teams initially build workflows on Dify, and once matured, migrate core capabilities to the platform's `agent_id`, or allow Dify nodes to call the enterprise `POST /agents/{id}/run` API.

The key is to prevent Dify nodes from bypassing enterprise data and tool governance. For example, if the marketing department's workflow requires real inventory data, it should call a read-only API exposed by the Registry, instead of connecting directly to the database through a Dify plugin. Publish-type actions should flow through the `waiting_human` mechanism described in Chapter 30, instead of relying solely on a manual UI node.

### 31.3.2 Coze

Coze is optimized for rapid release of marketing, customer service, and light-weight bots, especially for multichannel outreach. Its risks revolve around data residency, plugin permissions, and internal system integration. Coze plugins can act as HTTP facades for the enterprise Registry, carrying `tenant_id`, `user_id`, and scope information for the platform to authorize calls. For edge bots, it is recommended to only expose read-only or low-risk tools. Sensitive write operations, master data modifications, batch outbound messaging, and queries containing personal data should be routed back to the enterprise Runtime and HITL (Human-in-the-Loop).

This does not diminish the value of Coze. It is well suited as an entry point for outreach and interaction, such as event inquiries, store Q&A, and lightweight customer service. As long as plugin boundaries are clearly defined, Coze can help businesses quickly validate user entry points without taking on the role of the enterprise's core runtime environment.

The success criteria for edge bots should align with the platform: user identity can be transmitted back to the platform, tool calls can be traced through the Registry, important actions enter approval workflows, and logs can be merged with enterprise tracing. Without these, Coze bots can only serve as low-risk pilots. Only when they integrate into the platform chain is there a chance to move from pilot to long-term operation.

### 31.3.3 Bisheng

Bisheng is more enterprise and government-focused, commonly deployed for knowledge bases, process orchestration, and combinations of domestic AI models. It can serve as a knowledge control center, process UI, or base for localized deployments, but it is still necessary to evaluate whether it can align with enterprise `run_id`, Trace, Policy, and Export Bundle requirements.

In government and enterprise scenarios, a feasible approach is: Bisheng manages the knowledge base and part of the process approval UI, while execution nodes call the enterprise Workflow Agent Run via HTTP; Bisheng instance IDs map to `run_id` and write to Observability for unified search and replay capabilities.

## 31.5 In-House Development, Procurement, and Hybrid Approaches

### 31.5.1 When In-House Development Is Suitable

In-house development is suitable for core platform components such as unified `/run`, the Run six states, Tool Registry, MCP/A2A adaptation, Policy, HITL, Trace, Export Bundle, and deep integrations with ERP and the semantic layer. These capabilities directly impact compliance, SLA, auditing, and cost, and are generally not appropriate for each business team or every low-code platform to develop independently. The cost of in-house development is clear: it requires a platform team, SRE, test systems, and continuous maintenance. This approach is not chosen simply because it "looks more controllable," but because the production boundary cannot be managed by any single business application.

### 31.5.2 When Procurement Is Suitable

Procurement or managed services are suitable for rapid entry and standard use cases: marketing bots, campaign landing pages, standard RAG knowledge Q&A, non-core read-only assistants, operations consoles, and some workflow UIs. These scenarios emphasize launch speed and business configuration capabilities. Using Dify, Coze, or Bisheng can reduce reinventing the wheel. When procuring, clarify data residency, IAM, log export, tool permissions, approval integration, model routing, and SLA. Rich product features do not guarantee readiness for production core workflows.

Procurement failures mostly happen not because the product is unusable, but because the enterprise misplaces it. Treating a low-code platform as an experimental entry point is manageable risk; making it the enterprise's sole runtime without unified tool permissions and audit export risks turning every interface into an exception later. Procurement contracts must clearly specify data ownership, log exports, plugin approvals, SLA, and exit mechanisms; otherwise, a successful pilot will make migration back to a unified platform even harder.

### 31.5.3 Hybrid Is the More Common Approach

Most enterprises will eventually adopt a hybrid approach: building or deeply customizing the platform core in-house; embedding LangGraph or CrewAI within the Planner; either building the Console in-house or procuring/integrating a low-code UI; using Coze for edge bots while writing back operations to the platform; and running experiments with AutoGen before migrating mature cases to `agent_id`.

The key to a hybrid route is the master-slave relationship. The low-code platform may serve as a traffic entry point but must not own a separate set of production tool permissions. The framework can manage the Planner graph but must not hold an un-auditable tool execution chain. All actions that change business state must ultimately reside in the enterprise Runtime and Registry.

A typical hybrid topology is: Coze Bot or Dify Workflow receives user requests, calls the enterprise Runtime via API Gateway; the Runtime calls the Planner, which can use LangGraph; tool invocations go through the Registry; approvals enter `waiting_human`; and ultimately Trace and Export Bundle data flow back to the enterprise monitoring and compliance systems. This structure allows each layer to leverage its strengths without scattering ownership boundaries.

The rollout order for a hybrid architecture also matters: unify tool entry points first, then unify runtime events, and finally unify Console and operational experience. Tool entry is closest to risk and must be consolidated first; runtime events determine traceability and recovery and come second; Console experience can be integrated gradually and should not delay tool governance for the sake of UI unification.

## 31.6 Migration and Procurement Checks

### 31.6.1 Three-Step Integration

Existing frameworks or low-code platforms do not need to be rebuilt from scratch. Integration can be done in three steps. First, change internal tool calls within the framework to call the Registry HTTP API instead. Unify tool definitions, versions, permissions, and audit logging. Second, fold the framework's or low-code platform's runtime events into the six Run states and the `state` / `action` / `result` event model. Third, migrate write operations, HITL (Human-In-The-Loop), Export Bundle, and high-risk approval processes back into the enterprise Runtime. This order prioritizes resolving the most dangerous risk first: tools bypassing governance. Runtime and Console can be unified gradually, but tools and permissions should not be left until last.

### 31.6.2 Questions to Ask During Procurement

When procuring Dify, Coze, Bisheng, or evaluating internal platforms, you can directly focus on the capabilities outlined in Part V:

- Does it support stable `run_id` and export Run-level logs?
- Can tool calls be forced to go through the enterprise Central Registry?
- Are MCP Server or plugin tools version-registered?
- Is human approval engine-level suspension or a fake button on the UI?
- Can checkpoints restore the full context of the Planner?
- Can Traces export compliance replay packages?
- When writing ERP via low-code Workflow, does it enforce passing through Policy?

These questions matter more than "how many nodes are supported" or "is there a built-in knowledge base." Nodes and knowledge bases determine build speed; run-time contracts determine production readiness. The same questions apply to internally developed platforms. Being self-developed does not automatically mean governable. Without stable `run_id`s, tool versioning, approval checkpoints, or export packages, it is no different from an uncontrolled framework. Procurement and evaluation should compare vendor products and internal platforms against the same capability checklist instead of assuming internal equals compliant by default.

### 31.6.3 Relationship with Part V Practical Implementation

`projects/multi-agent-workflow/` already covers the main workflow from Chapters 22 to 30: within the same `run_id`, from RunLoop, Registry, Planner, Handoff to `waiting_human` and approve/resume. When evaluating frameworks, this project can be used as a baseline: can the external framework integrate with the same Run, Registry, approvals, and checkpoints?

LangGraph can replace the Planner implementation; CrewAI can contribute role configuration; Dify can call the enterprise `/run` API; Coze can serve as the edge Bot entry point. Yet `core/runtime` and `core/registry` remain the platform's core and cannot be directly replaced by these frameworks.

If an external framework cannot integrate with this baseline workflow, it must at least clarify who will fill in the gaps: the platform adapter layer may map events, the vendor may provide log exports, or the business may abandon production rollout for that scenario. Gaps do not necessarily mean unusable, but must be documented clearly before pilots. This is the core standard of this chapter: not "which framework is best," but "what layer of responsibility does it assume in the enterprise Agent platform?" When responsibility boundaries are clear, more frameworks can accelerate development; when unclear, more frameworks increase governance difficulties.

In the initial rollout, you can first choose one read-only scenario and one write-operation scenario for comparison. The read-only scenario verifies framework integration speed. The write-operation scenario validates whether Registry, Policy, HITL, and Trace can converge properly. Both scenarios running smoothly is what makes a hybrid architecture viable. This approach is more reliable than comparing product features alone.

## 31.7 Governance Takeover After Framework Selection

After framework selection, the harder work is governance takeover. Many teams can quickly build demos with LangGraph, Dify, or Coze, then discover at launch that tool permissions, log export, approval suspension, and cost allocation have no unified owner. The platform team does not need to force business teams to abandon these tools, but it must define which capabilities can stay inside the framework and which must move back to the enterprise foundation.

Tool execution is the first capability to take over. Any tool that accesses enterprise data, triggers a write operation, or sends an outbound notification should pass through the unified Registry and Policy. The framework may keep Planner graphs, node orchestration, and UI entry points, but tool calls must enter the enterprise audit chain. The second takeover target is runtime events: framework-internal states should map to Run six states, Tool Calls, and Handoff events. Console, release flow, and operating dashboards can be consolidated after these two steps are stable.

Procurement evaluation should also cover exit mechanisms. After a pilot succeeds, the ability to export data, logs, prompts, plugin configuration, evaluation samples, and user feedback determines whether the enterprise is locked into one platform. Contracts should specify data residency, log retention, plugin approval, SLA, private deployment options, and migration interfaces. Otherwise, the more successful the pilot becomes, the harder it is to return to a unified platform.

A hybrid route still needs one primary runtime. A low-code platform can be an entry point, an open-source framework can implement Planner logic, and a commercial Copilot can be embedded in a business system. Business-changing actions still need to return to enterprise Runtime. Once two systems can independently execute tools, approve actions, and audit outcomes, platform governance splits. Chapters 22 through 30 define the capabilities needed to prevent that split.

## 31.8 Migration Strategy for Procurement, In-House Development, and Hybrid Routes

Framework selection is not a one-time decision. Enterprises usually start with low-risk scenarios, use open-source frameworks or commercial platforms to build pilots, and then move identity, permission, audit, cost, SLO, and tool governance into the platform layer as pilots enter production. Without a migration strategy, teams either become locked into an early framework or rebuild from scratch.

A more stable approach is to define replaceable boundaries during the pilot. Three interfaces matter most. The first is the Agent definition interface: roles, tools, prompts, policies, and model configuration. The second is the run-record interface: Run, Step, Tool Call, Trace, and Artifact. The third is the governance interface: permission, approval, evaluation, release, and rollback. As long as these three categories can be exported from an external framework or mapped into platform objects, later migration from a procured platform to an internal Runtime, or from one framework to a hybrid architecture, will not break the chain.

The hybrid route also needs source-of-truth ownership. The enterprise may use LangGraph, Dify, MCP Servers, and an internal Runtime together, but each system should not maintain its own identity, tools, logs, and approvals. Tool catalog belongs to Tool Registry, run ledger belongs to Runtime, evaluation samples belong to Eval, and permissions belong to enterprise IAM or data permission systems. Other frameworks can act as executors or developer-experience layers. They should not absorb governance responsibility.

## 31.9 Organizational Constraints in Framework Selection

When technical capability is similar, organizational constraints usually decide the route. A small team without platform engineering capacity may slow down business pilots if it insists on building a Runtime. A large enterprise with complex permission and audit requirements may be unable to put core processes fully on a low-code platform. Selection should evaluate what the team can operate for years, not which framework looks faster in a demo.

Organizational constraints include developer profile, operations ownership, data-governance maturity, and procurement process. Business-led pilots often fit low-code platforms, but the platform team should define safety boundaries early. Platform-led shared capabilities are better suited to Registry, Runtime, and Eval, but business teams still need fast iteration paths. If the enterprise already has a unified gateway, IAM, data catalog, and observability platform, Agent frameworks should adapt to that foundation instead of creating isolated infrastructure.

The conclusion is therefore not a recommendation for one framework. The point is to evaluate frameworks inside platform engineering. Frameworks can accelerate development, but they cannot replace the enterprise's responsibility for runtime governance. Once this principle is clear, procurement, in-house development, and hybrid routes can be combined by stage instead of becoming a route argument.

## 31.10 Exit Mechanisms After Introducing a Framework

Framework adoption discussions often focus on onboarding speed and ignore exit mechanisms. The enterprise platform should answer early: if the framework is no longer maintained, commercial terms change, performance is insufficient, or security review fails, can existing Agents, tools, Trace records, and evaluation samples move out? Without an exit mechanism, every successful pilot makes future migration harder.

The exit mechanism does not require abstracting every capability from day one, but the core assets should remain under enterprise control. Prompts, tool definitions, evaluation samples, run logs, user feedback, and process diagrams should live in repositories or platform stores controlled by the enterprise instead of only in a framework's private configuration. Capabilities that cannot be exported should be treated as risk and kept out of core flows.

Exit mechanisms also affect procurement negotiation. Vendors should explain how to export data, logs, tool definitions, model configuration, and account permissions. Non-exportable parts should be marked as risk and limited to non-core scenarios. This makes procurement reflect long-term platform cost instead of only feature lists and price. Internal learning cost matters as well. A powerful framework with poor debugging and opaque run records may fit a research team but fail across many business lines. Platform teams should include documentation, community quality, observability, and extension boundaries in evaluation.

Every selection should leave review material behind. Six months later, if the team adjusts its route, it should know why the original choice was made, which assumptions held, and which assumptions failed. Without such material, framework replacement turns into opinion debate. The review record should include pilot scenario, launch scope, dependent capabilities, unmet needs, and migration constraints. It should also record real team experience: whether debugging was smooth, documentation reliable, online incidents diagnosable, and vendor response timely. These details often explain long-term maintenance cost better than a feature matrix.

Two reference scenarios can anchor rollout. One read-only knowledge Q&A scenario validates framework integration speed and business self-service. One high-risk scenario with write operations or export validates Registry, Policy, HITL, Trace, and audit convergence. A hybrid architecture is credible only when both scenarios pass. Low-risk scenarios alone tend to overestimate product readiness.

Framework governance should also track version drift. When a business team upgrades LangGraph, a Dify plugin, or a Coze Bot configuration, platform ToolSpec, approval policy, and Trace fields may not update at the same time. A stable pattern is to write framework version, application version, and platform adapter version into each Run. During incidents, the team can tell whether behavior changed because the framework upgraded or the platform policy changed.

The platform team should still leave room for experiments. Requiring every new idea to enter unified Runtime first slows innovation; allowing every experiment to bypass the platform accumulates risk. A clear boundary is more useful: read-only, low-risk, small-user-count prototypes may move fast inside frameworks; production data, write operations, external users, and high-value processes must enter platform governance. Frameworks then serve delivery pace, while the platform keeps the responsibility boundary stable.

This kind of responsibility map can also become procurement material. Supplier commitments, enterprise-owned adapters, and remaining risks should appear in one table, so contracts, pilots, and acceptance reviews do not use different definitions. The same material helps business teams understand why some capabilities can keep moving quickly in the framework while others must return to the platform before production.

The more frameworks an enterprise uses, the more stable the platform interface must be. A stable unified entry point lets business teams keep experimenting above it. Later migration, procurement, and in-house development decisions should revolve around that entry point instead of around the short-term experience of one tool. This is why the framework chapter ultimately returns to platform responsibility.

## 31.11 Framework Governance Ledger And Periodic Review

After a framework is introduced, the platform needs a governance ledger. The ledger records which responsibility layer each framework or low-code platform carries: entry point, Planner, tool execution, knowledge base, Console, approval, Trace, evaluation, or export. It should also record owner, version, integration method, data boundary, tool boundary, log-export capability, exit condition, and review date. Without this ledger, an enterprise may discover after a few months that the same capability is scattered across products: one team calls tools through Dify, another uses LangGraph as a runtime, a third maintains a separate knowledge base in Coze, and the platform has no shared run evidence.

Periodic review should ask three questions. First, do early assumptions still hold? A low-code platform that now touches production data and write operations should no longer be managed as a pilot. An internal Runtime with little business use may need a scope review. Second, are core assets portable? Prompts, tool definitions, evaluation samples, run logs, user feedback, and approval records should be exportable or mappable into platform objects. Third, is the platform interface stable enough? Business teams are willing to experiment above the platform only when `/run`, Registry, Policy, and Trace do not change unpredictably.

Review material should feed the roadmap. If a framework works well for read-only scenarios but cannot connect write actions to Policy, it can remain an entry point but should stay out of core workflows. If an internal capability is reused across several businesses, it should become a platform module. If a vendor product has opaque logs, the next contract should add export and audit clauses. Framework governance does not exist to slow business experiments. It gives successful experiments a migration path and failed experiments a safe exit path.

The first ledger can stay simple: framework name, scenario, responsibility layer, interface to enterprise foundation, non-portable assets, current risk, and next review date. If it tells the platform team who owns runtime responsibility, it already prevents framework selection from being treated as a one-time procurement conclusion.

## 31.12 Pilot Acceptance and Production Transition

Framework pilot acceptance should produce two kinds of material. The first proves business value: what task the pilot solved, whether users kept using it, which manual steps were removed, which samples users rejected, and which artifacts entered downstream workflows. The second proves production handover: whether tools enter Registry, runtime events map to Runs, logs can be exported, permissions use enterprise IAM, approval is a real suspension, and cost and SLO can be counted by tenant. The first set says whether the pilot is worth continuing. The second says whether it can enter production. Many framework projects fail because these two standards get mixed: the business sees value, while the platform cannot see handover evidence.

Production transition needs explicit gates. Read-only, internal, low-volume scenarios can keep iterating inside the framework. Once the application touches production data, external users, write operations, or high-value workflows, tool execution, state, audit, and release controls should move into the enterprise foundation. Transition does not always require rewriting the application. An adapter can first register framework events, tool calls, and artifacts into platform objects. If the adapter cost is too high, the framework may be better kept as an entry point or prototype layer. That judgment should enter the review record so a successful pilot does not quietly create production responsibility that the platform cannot govern.

## 31.13 Organizational Constraints In Framework Migration

After an Agent framework enters production, migration cost often exceeds technical validation cost. Moving from a single SDK to a state graph, connecting a self-built Runtime to an external framework, or moving a vendor Agent into the enterprise platform touches tool schema, Memory, Trace, evaluation samples, frontend events, and approval state. A migration plan that compares framework features alone will underestimate organizational and data work.

Migration should first stabilize platform contracts. Model calls go through the gateway. Tools register through Registry. Run state enters Trace. Evaluation samples can replay. High-risk actions go through HITL. When these contracts are stable, the underlying framework can be replaced in stages. If contracts are unstable, framework migration becomes a full-chain rewrite. The first stage can connect existing Agents to observability and cost. The second stage can unify tools and permissions. The third stage can migrate Planner, Memory, and Runtime. Each stage should deliver visible value instead of only changing the technology stack.

Business continuity must be preserved. Existing Agents should not lose support because a new platform is under construction. A new framework should not bypass current approval and security processes. Platform teams provide compatibility adapters. Business teams provide representative tasks and acceptance samples. SRE confirms rollback windows. Security teams confirm high-risk actions have not expanded. Framework selection ultimately serves platform operability, not a development-experience preference.

## 31.14 Runtime Samples For Framework Evaluation

Framework evaluation should not rely on official examples alone. Production evaluation needs runtime samples: long-task recovery, tool failure, human approval, Memory read, concurrent cancellation, cross-device recovery, Trace export, and cost records. A framework that can quickly build a chat demo but cannot express Run state and recovery paths clearly should not carry the enterprise Agent mainline directly.

Runtime samples should cover development experience and operational responsibility. Developers care whether state graphs are easy to write, tools are easy to connect, and debugging is convenient. SRE cares whether failures are observable, versions are rollbackable, and tasks can be interrupted. Security teams care whether permission and approval can be inserted. Business teams care what users see during exceptions. Framework evaluation led only by developers misses production responsibility.

The first version can keep a hybrid route. Core Runtime and governance contracts stay under platform control, while business applications may use external frameworks or SDKs locally. As long as they obey interfaces for model gateway, Tool Registry, Trace, HITL, and Eval, they can enter the platform gradually. Framework choice then avoids a one-time bet and reduces migration risk.

## 31.15 Framework upgrades and platform contracts

Agent framework upgrades should not be judged by feature lists alone. The framework's message model, tool-call format, state storage, callback events, error types, and concurrency semantics may all change, and these changes affect platform contracts. Once the framework is connected to Runtime, Trace, Eval, Tool Registry, and Guardrails, an upgrade is no longer a simple dependency change. It becomes a runtime-chain change.

Before upgrading, the team should prepare compatibility samples. The samples should cover multi-turn dialogue, tool calls, long-running suspension, human approval, tool failure, retry, cancellation, and artifact generation. Event order, error codes, state transitions, and Trace fields in the new framework version should match the platform contract. If the framework changes internal state structure, the platform must decide whether to write an adapter, freeze the old version, or migrate runtime data. A direct upgrade can make old Runs impossible to replay and weaken incident review.

A first platform version can put framework upgrades behind release gates: run samples in shadow environment, connect low-risk Agents, then expand to core paths. Each upgrade should record affected modules, migration script, rollback target, and known differences. Frameworks can then evolve without letting external project cadence rewrite platform contracts.

## 31.16 Turning framework outputs into platform assets

After a framework pilot succeeds, asset capture is easy to miss. A business team that launches an Agent with an external framework usually creates prompt templates, tool adapters, sample data, user feedback, failed traces, permission rules, and frontend interaction lessons. If those materials stay inside the framework project, the next team repeats the same learning, and the platform cannot build shared capability. At pilot close, reusable materials should be mapped to platform objects: prompts to the template repository, tools to Registry, failure samples to Eval, runtime logs to Trace, approval rules to HITL and Policy, and frontend events to the common interaction contract.

This capture process should separate reusable assets from application-specific work. Some node orchestration belongs only to one business flow and can stay at the application layer. Tool contracts, state events, evaluation samples, and security policies that appear across scenarios should move into the platform foundation. The platform team does not need to rewrite every framework project into one implementation, but it should extract repeated capabilities so the next integration starts with less uncertainty. Business teams benefit as well: new scenarios can reuse mature tools, samples, and audit paths instead of beginning from an empty project.

Asset capture also affects procurement judgment. A vendor product or open source framework that can export runtime logs, tool definitions, evaluation samples, and prompt versions into platform objects has lower migration risk. A product that locks all assets inside a private console may offer good short-term development speed while creating long-term governance debt. Framework selection should compare development efficiency and whether pilot outputs can become enterprise-owned platform assets. That standard keeps experimentation fast while protecting the long-term operating boundary.

## 31.17 Responsibility split after framework integration

After a framework enters the enterprise platform, the responsibility split should be written down early. The business team usually owns scenario design, user flow, and acceptance samples. The platform team owns Runtime, Registry, Trace, Policy, HITL, and release gates. The security team owns permissions, data egress, and high-risk actions. A vendor or open source maintainer owns framework capability and version notes. If these responsibilities do not map to concrete interfaces, incidents will be blamed on framework usability or model instability while the real gap remains unresolved.

The split should describe runtime paths and assign them to teams. When a tool call fails, who checks ToolSpec, who verifies external-system idempotency, and who decides whether retry is allowed. When a long-running task suspends, who defines timeout, who notifies the approver, and who handles user cancellation. When a framework upgrades, who provides compatibility samples, who runs shadow replay, and who approves production entry. These questions should appear in the integration document, or the framework pilot will stay at demo depth.

For procurement, responsibility split also affects contracts and acceptance. If a vendor promises canvas usability but cannot export runtime logs, tool definitions, evaluation samples, or error codes, the platform team inherits governance cost. If the vendor provides open interfaces but the business team has no acceptance samples, the system still cannot prove production fitness. Framework selection should bring these constraints forward so technical evaluation, procurement evaluation, and security evaluation use the same material.

A first version can create a one-page responsibility matrix for every framework integration project. The matrix can stay compact, but it should cover runtime entry, tool registration, state mapping, Trace fields, evaluation samples, permission approval, upgrade rollback, and asset capture. Chapter 31's framework comparison then becomes an executable integration process instead of a product-feature list. The aim of adopting a framework is to bring pilot capability into platform governance, not to leave each business team with isolated runtime semantics.

## 31.18 Compatibility governance for framework upgrades

Agent framework upgrades often carry hidden risk. A minor version can change tool-call format, Memory defaults, callback events, error types, retry behavior, or streaming output. The application team may see only a dependency bump, while the platform may see changes to message contracts, Trace fields, evaluation samples, and security policies. Enterprise platforms should not treat framework upgrades as ordinary package maintenance.

Compatibility governance starts by identifying platform contracts. Before upgrading, the team should check Runtime events, ToolSpec mapping, Memory interfaces, HITL states, Trace fields, error codes, and frontend message models. If the framework is used only for experiments, upgrades can move faster. If it supports production Agents, upgrades need sample replay, shadow running, and canary release. Vendor changelogs are useful, but they cannot replace local business samples because enterprise tools and permission policies have their own boundaries.

A first version can set a minimum process: pin the old version, create an upgrade branch, replay core samples, compare Trace, check security samples, confirm rollback path, and then expand traffic. If the upgrade brings new capability, the team should decide whether it belongs in the platform contract before an application uses it directly. Framework evolution can then serve platform capability building without letting production systems follow external framework cadence blindly.

## 31.19 Moving framework capabilities into the platform standard library

After a framework is introduced, teams often use built-in components directly: Agent executors, Memory modules, tool wrappers, callbacks, and evaluation helpers. This speeds up pilots, but over time it creates incompatible implementations. The platform should absorb validated framework capabilities into a standard library and keep application code away from framework internals. The standard library keeps stable interfaces, while the framework becomes one implementation source.

Entry into the standard library needs a bar. A component should have clear inputs and outputs, error semantics, Trace fields, permission boundaries, sample tests, and version policy. A framework tool wrapper may be convenient, but if it cannot record idempotency keys and compensation state, it should not enter production standard library. A Memory module may be quick to connect, but if it lacks deletion and audit capability, it should remain in pilots. The standard library selects governable capabilities, not the richest API.

A first version can use four states: candidate, pilot, standard, and retired. Candidate capabilities come from frameworks. Pilot capabilities are tested in one scenario. Standard capabilities enter platform documentation and templates. Retired capabilities keep migration notes. The platform can then absorb framework value without being locked to the lifecycle of one external framework.

## 31.20 Exit conditions for framework capability migration

When exit conditions for framework capability migration reaches production, the platform needs a shared evidence standard for standard interface, dependency version, replacement path, sample coverage, known gap, migration cost, and retirement plan. This standard is not paperwork for its own sake. It lets business, platform, data, security, and operations teams discuss the same facts. Without this material, incident review depends on memory and personal judgment. With it, the team can see which inputs were valid, which actions executed, which artifacts can still be used, and which results need correction or withdrawal.

This evidence should connect to Chapter 22 on Runtime, Chapter 23 on tool registration, and Chapter 38 on Trace. The upstream chapters provide the capability base, downstream chapters consume the runtime result, and this chapter explains how the middle layer is verified. If a capability looks complete inside one chapter but cannot enter Trace, Eval, release records, or the compliance evidence package, the production system still has a break in the chain. Readers should treat cross-chapter interfaces as engineering contracts, not as a reading order.

Common risks include application code binding to framework internals, Trace fields changing during upgrades, and teams maintaining incompatible wrappers. A successful demo rarely exposes these problems because demo samples are usually clean, short, and direct. Real business traffic brings stale data, abnormal input, permission changes, user withdrawal, budget limits, and long-running state. If the platform does not turn those situations into samples and ledgers, later scenarios will hit the same class of issues again.

Application code binding to framework internals should be turned into a tracked review item when it appears repeatedly. The operating record should at least state owner, version, sample, affected scope, action, and review time. It does not need to become a long process report, but it must be clear enough for a later maintainer to understand the decision. For high-risk capability, the record should also state which conditions allow wider use and which failures require degradation or withdrawal.

A first version can build this habit in a few representative scenarios. It is better to make high-traffic, high-risk, externally visible paths solid first, then copy the sample, ledger, and review method to related capabilities in other chapters. This makes the chapter read like engineering guidance: it explains how the capability is integrated, validated, operated, and retired.

## Chapter Recap

Frameworks, platforms, and applications should be evaluated as separate layers. LangGraph is useful for Planner expression, AutoGen for experimental collaboration, and CrewAI for role configuration. Dify, Coze, and Bisheng are stronger in entry points, Bot channels, and knowledge management. They can all enter an enterprise architecture, but they should not replace Runtime, Registry, Policy, HITL, and Trace. Selection should prioritize `/run`, Registry, Policy, checkpoints, Trace, and audit replay before canvas convenience or node counts. A hybrid architecture is acceptable when business-changing actions ultimately return to enterprise Runtime and Registry under unified permission, audit, checkpoint, and rollback constraints. Framework pilots also need asset capture and a written responsibility split so reusable work can become platform capability.
