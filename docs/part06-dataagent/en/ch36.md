# Chapter 36 Data Analysis, Visualization, and Reporting

Chapters 34 and 35 have already handled data retrieval and analysis. The system knows that operational GMV in East China fell last week, which SKUs contributed most, and how the decline looks when grouped by category. Business users now need material they can bring into a discussion, rather than SQL output or Python stdout. A business report must explain what happened, where the evidence came from, which statements are facts, which statements are inferences, and which recommendations still require human confirmation. When a controller approves the report, they should be able to open the footnotes and inspect the original query, Python artifact, metric version, and data freshness. Smooth prose without that evidence only raises the risk.

The presentation layer receives the outputs of Chapters 34 and 35 and turns tables, statistics, charts, and evidence into a report. Chapter 30 defines the HITL state machine, Chapter 48 defines frontend rendering and interaction, and Chapter 39 defines the evaluation pipeline. This chapter defines what a DataAgent deliverable should look like.

Many DataAgent prototypes perform well during data retrieval, then fall back into chatbot behavior at delivery time. They can find the number but write every result as loose prose; they can draw charts but omit the chart definition; they can make recommendations but fail to state which ones need business approval. The first report may look convenient. The second time it is taken into a meeting, finance may ask which metric version was used, operations may ask why these SKUs were selected, a regional manager may ask whether the report can be forwarded, and audit may ask whether the chart contains unauthorized fields. If the presentation layer lacks evidence, versions, and responsibility, even rigorous SQL and Python work will have limited value in the business workflow.

Report generation also serves two audiences. Business owners need to know what happened, how large the impact is, and who should act next. Platform and audit teams need to know where the evidence came from, whether the calculation is reproducible, and which conclusions are only inferences. The report body should be concise enough for business review, while EvidenceRef, chart data references, SQL/Python artifacts, and metric versions provide the review path. A DataAgent report should be readable and defensible at the same time.

Another common mistake is treating insight as a better-written summary. A useful insight changes the user's next action: continue drilling into stores, involve the category manager, verify data quality before approval, save the report as weekly meeting material, or keep it as an ad hoc analysis. The model can help organize language, but the business value comes from evidence structure, context, and approval boundaries. Charts, reports, and recommendations in this chapter all serve one purpose: helping the reader see what can be used directly and what still needs verification. The presentation layer is the business delivery layer of DataAgent. It receives upstream calculations, prevents the model from inventing numbers, selects appropriate charts, generates editable reports, and sends user feedback back into evaluation.

## 36.1 From Results to Insights

An insight is more than a number translated into natural language. Restating a result, such as "operational GMV in East China fell 12.3%," only tells the user what happened. An insight continues the analysis: where the decline is concentrated, which structural changes may be related, and whether the situation warrants action. A usable insight usually contains a comparison baseline, magnitude, concentration, evidence, and uncertainty.

*Table 36-1: Differences between restatement, insight, and recommendation. Source: Compiled by the author.*

| Type | Example | Purpose |
|---|---|---|
| Restatement | Operational GMV in East China fell 12.3% | Tells the user what happened |
| Insight | SKU-A, SKU-B, and SKU-C account for 58% of the decline gap | Shows where the problem is concentrated |
| Inference | The personal care category share increased, which may relate to the end of a promotion | Proposes a direction for verification |
| Recommendation | Ask the category manager to review SKU-A promotion and replenishment strategy | Moves toward business action and approval |

When the Planner generates insights, the model should not recalculate contribution rates. Contribution, share, year-over-year change, and week-over-week change should be calculated by SQL or Python first. The model is responsible for organization, prioritization, and evidence linkage. If the Python artifact does not contain "58%," that number should not appear in the report. Insight text must also distinguish facts, inferences, and hypotheses. Facts come directly from aggregation or scripts. Inferences are evidence-based but still need business confirmation. Hypotheses are directions to test. Mixing these categories makes a report sound decisive while making it harder to approve.

The number of insights also needs control. A business meeting report should not list every anomaly the system can find. It should select three to five conclusions around the main question. In the East China decline case, the main question is regional GMV decline, so top SKUs, category contribution, data quality, and causes to verify are enough. Adding stores, channels, members, and weather would turn the report into a collection of observations. DataAgent should help users converge, rather than show every pattern it discovered.

Insight language should avoid unsupported certainty. Without promotional calendar evidence, the report should not write "the decline was caused by the promotion ending." It can write "the decline timing may be related to the end of a promotion and should be checked against the promotional calendar." This wording is more useful to a controller because it shows which conclusions can be quoted and which need follow-up.

Insights also need to preserve the analysis path. A conclusion from SQL aggregation should cite the query and metric version. A conclusion from Python contribution analysis should cite the corresponding artifact. A conclusion that relies on business knowledge or external context should state the source and confirmation status. A conclusion without a path is only a model judgment, even when the wording sounds professional. Production reports should contain fewer claims if necessary, as long as each claim can return to evidence.

Report structure should support review. Business material usually starts with conclusions, then evidence and action items. Technical post-mortems often start with data scope, method, and limitations before the conclusions. DataAgent should not force every task into one template. The system can select structures by task type: anomaly diagnosis reports emphasize impact scope, contribution items, causes to verify, and owners; metric definition reports emphasize definition, source, calculation, and applicability; management weekly reports emphasize trends, risks, and action items. The right structure keeps users from searching for the point inside a pile of correct numbers.

Charts are a common source of misreading. The model can suggest a chart type, but chart titles, axes, units, sorting, filters, and data ranges must be generated from structured fields. A bar chart without a clear distinction between "sorted by GMV gap" and "sorted by contribution percentage" can lead to different interpretations in a meeting. The simpler the chart, the clearer the data contract behind it must be.

Reports also need editing support. Business owners will change titles, remove paragraphs that are unsuitable for external circulation, or request language that better matches organizational style. Editability does not mean facts can be changed freely. The system should separate factual fields from wording fields. Factual fields come from SQL/Python artifacts and should be marked as manually edited if changed. Wording fields can be adjusted, but publication should still require EvidenceRef. This respects business writing habits without breaking traceability.

The presentation layer should leave users with a clear sense of status: what is supported by data, what is an inference, and what requires approval. Such material may look less flashy, but it is easier to use inside enterprise workflows. It lets meeting discussion focus on business judgment instead of repeatedly asking where a number came from.

## 36.2 Chart Selection

Charts serve comparison intent. Use line charts for trends, horizontal bars for ranking and contribution, stacked bars or area charts for composition, histograms or box plots for distributions, and scatter plots for correlation. DataAgent should avoid 3D bars, excessive colors, and pie charts with many slices simply because they look attractive. Chart selection should also consider how the material will circulate. Interactive pages can support hover, filtering, and drill-down. Exported reports must make the definition clear in a static page.

Many business reports are copied into screenshots or forwarded outside the original interface. Once a chart leaves the system page, EvidenceRef and context are easier to lose. Exported versions should place key definitions, time range, and data source near the chart, rather than relying on a technical appendix at the end.

Automated EDA also needs a clear position. It is useful for detecting data quality issues, distribution shifts, outliers, and candidate patterns. It should not send every finding to business users. A higher missing rate in one field may be an upstream synchronization issue or a process change. A store outlier may come from a promotion or an input error. DataAgent can put these signals into "quality notes" or "leads to verify" rather than presenting them as established business conclusions.

Recommendations must be controlled as well. The system may recommend that a category manager verify the impact of a SKU-A promotion ending. That is an investigation task. Writing "restore SKU-A promotion immediately" enters business decision territory. Recommendations should state the responsible role, evidence source, risk level, and approval status. This lets users turn a recommendation into a task, meeting topic, or approval action, while the platform records whether the recommendation was accepted, modified, or rejected.

Report quality evaluation should cover edited versions. A first draft may have complete evidence, then the user deletes footnotes. A first draft may use cautious wording, then the user changes it into a definite claim. A chart may be correctly referenced inside the system but lose context when copied into an external document. The platform should validate reports at save, approval submission, and export. Otherwise the evaluation only proves that the model's draft passed, not that the final deliverable is fit for use.

Feedback after publication should also enter the learning loop. Which paragraphs users remove, which charts they keep, which explanations they add, and which recommendations approvers reject all indicate whether the template and insight strategy are appropriate. The platform can turn editing behavior into template signals, rejected conclusions into evaluation samples, and accepted action recommendations into downstream tasks. The presentation layer becomes an entry point for continuous DataAgent improvement.

In enterprise work, a report often has a second life. It may enter messaging tools, email, slide decks, BI pages, or workflow systems. Each circulation path should retain minimum context: report version, data scope, generation time, owner, and evidence entry point. If a second circulation only carries a chart or one sentence, later discussion can detach from the original evidence. DataAgent export design should account for this from the beginning.

In the East China case, top SKU contribution is better represented by a horizontal bar chart. Bars show which SKUs have the largest gaps and can bind cleanly to row-level EvidenceRef. Category share change can use stacked bars or grouped bars. Multi-week trends should use line charts.

Chart specs are controlled artifacts, not image prompts. `x_field`, `y_field`, sorting, aggregation, and color fields must exist in the input schema or Python artifact. `chart_renderer` should validate field existence, data reference consistency, and metric version visibility, then return `chart_id`, `data_ref`, and an evidence hash. Chapter 48 covers frontend rendering and interaction. The model should not directly produce untraceable images.

Automated visualization tools such as LIDA provide a multi-stage pattern: summarize data, enumerate goals, generate charts, and evaluate results. DataAgent can borrow this pattern, but enterprise usage should be more conservative. Rules determine the chart type first, the model completes the spec, and schema plus EvidenceRef validation decides whether the chart can be used.

Charts should fit their reading context. Business meeting materials require fast comparison, so titles, axes, units, and definitions should be visible immediately. Interactive exploration pages can support filtering and drill-down, but exported PDFs and screenshots must retain enough context. A chart title such as "Operational GMV `gmv_ops@2025Q1`" is better for audit and forwarding than "GMV Gap Contribution." Colors and sorting should also be controlled. Positive and negative deltas need consistent semantic colors. Ranking charts must state whether they sort by absolute gap or contribution percentage, and whether non-top-N items are merged into "Other." Arbitrary colors and sorting make weekly reports hard to compare.

## 36.3 Automated EDA and Anomaly Detection

Automated EDA should not dump a profiling report into the user's view. In diagnostic tasks, it should generate a small number of structured signals: missingness, outliers, distribution shifts, top-category changes, and sample-size risks. In the East China decline scenario, if the `sku_id` missing rate is slightly elevated, the report should warn that top-SKU ranking may be affected.

Automated EDA must record its method. "SKU-A is 2.4 standard deviations below last week's mean" is more credible than "AI detected an anomaly." Statistical method, threshold, input data, and hash should be written into an artifact that the report can reference. Anomaly alerts without method descriptions become another form of model hallucination.

The EDA level should also be controlled. Simple data lookup can disable automated EDA. Diagnostic tasks can run lightweight EDA. Subscription workbenches are better suited to proactive anomaly detection. Excessive EDA makes simple questions slow and fills reports with marginal signals, which can distract business judgment.

EDA findings should not automatically become insights. Missingness, outliers, and distribution shifts are candidate signals. The Planner must decide whether they relate to the user's problem. In the East China case, a slightly higher `sku_id` missing rate affects top-SKU ranking and belongs in a quality note. An unrelated field anomaly should not enter the executive summary. Otherwise the report confuses data quality noise with business findings.

When EDA detects severe quality issues, the system should lower the delivery level. Minor warnings can become report footnotes. Severe quality failures should reject formal report generation or enter HITL. The presentation layer must not polish away upstream quality risk, because business users often see only the final material.

## 36.4 EvidenceRef and Report Structure

EvidenceRef is the core contract of a DataAgent report. It binds an insight to the data source that produced it: which tool, which artifact, which row or aggregation, which metric version, and which content hash. Reports may rephrase wording, but they must not drop the evidence reference.

```json
{
  "ref_id": "ev_002",
  "source_tool": "python_sandbox@v1",
  "artifact_id": "df_sku_contrib",
  "locator": "row:sku_id=SKU-A",
  "metric_id": "gmv_ops",
  "metric_version": "2025Q1",
  "content_hash": "sha256:8f3a2b1c"
}
```

When a user opens a footnote, the frontend should locate the corresponding row or aggregate based on `artifact_id` and `locator`, verify the content hash, and display metric version, data timestamp, and source tool. Free text such as "source: Python analysis" has little audit value.

The report structure can stay stable. Business analysis reports usually include an executive summary, question and scope, data overview, findings, recommendations, and appendix. The executive summary should contain three to five points, each with EvidenceRef. The question and scope section should show the Question Frame, metric version, and time range. The appendix should retain Run ID, SQL hash, Python hash, freshness, and the full evidence list.

Business meetings, regulatory submissions, and external-facing material should enter HITL. DataAgent can generate drafts, but controllers, legal reviewers, or business owners must confirm before publication. HITL records rejection reasons, annotations, and revision history into Trace, creating samples for future evaluation.

Report editing must protect the evidence chain. Users may change titles, adjust wording, delete paragraphs, or request a new chart. A conclusion should not keep its original claim while losing its EvidenceRef. If an edited sentence is no longer supported by the original evidence, the system should require evidence rebinding or mark the sentence as manually supplemented. People can edit the report; traceability should remain intact.

EvidenceRef also needs version pinning. After publication, upstream artifacts may be cleaned or recalculated, and semantic layer metrics may be upgraded. Published reports should continue pointing to the artifact hash and metric version used at publication time. They should not silently follow the latest data. If the user refreshes a report, the system should generate a new Run and a new EvidenceRef set.

The appendix is not an optional engineering dump. Formal reports should at least keep Run ID, metric version, SQL hash, Python hash, chart `data_ref`, freshness, and quality status. Business users may not open the appendix during every review, but when a dispute appears, the appendix determines whether the report can be audited.

## 36.5 Actionable Recommendations and Business Feedback

An Agent should not make business decisions on behalf of the business. Recommendations in reports should be written as "who needs to evaluate what" rather than direct business commands. Replenishment, price changes, promotion, and channel changes involve inventory, margin, supply chain, and organizational responsibility. DataAgent can propose evidence-based options.

Recommendations should be tiered. Observations state facts. Inferences explain possible reasons. Recommendations propose executable actions and mark approval status. In the East China case, "SKU-A contribution fell 32%" is a fact. "It may relate to promotion ending" is an inference. "Ask the category manager to review SKU-A promotion and replenishment strategy" is a recommendation.

Business feedback should return to evaluation and templates. User acceptance, rejection, delay, or modification of recommendations should enter Eval and the Playbook candidate pool. Recommendations with high acceptance can become standard analysis procedures. High rejection rates usually indicate over-inference, insufficient evidence, or a report template that does not fit the current business.

Recommendations also need an owner. "Optimize promotion strategy" is too vague. "Ask the category manager to check SKU-A promotion end date and replenishment changes" is actionable. The owner can come from organizational context or manual selection, but the model must not invent one. If the platform does not know the owner, the report should say "requires business owner assignment" rather than fabricate a department.

Recommendations should state prerequisites. "Evaluate replenishment strategy" only makes sense when inventory data, promotional calendar, and margin definition are available. If the current Run covers only sales, the report should say that inventory and promotion calendar data are needed before making the judgment. This restraint reduces repeated meeting questions about the basis of an action.

Feedback results can improve report templates. If a certain recommendation type is rejected repeatedly, it may exceed the DataAgent's evidence scope. If a certain insight type is accepted repeatedly, it can become a Playbook and fixed chart. DataAgent reporting is not a one-time text generation feature. It is a feedback-calibrated business delivery capability.

## 36.6 Output Quality Evaluation

DataAgent evaluation cannot stop at SQL exact match. User complaints often concern charts that miss the question, conclusions without evidence, missing follow-up questions, and unclear report definitions. Output evaluation should cover answers, insights, charts, reports, refusals, and clarification questions.

*Table 36-2: DataAgent output evaluation dimensions. Source: Compiled by the author.*

| Dimension | Checkpoints |
|---|---|
| Answer | Numerical tolerance, ranking consistency, definition footnotes |
| Insight | Whether each insight has a valid EvidenceRef |
| Chart | Whether the chart type matches intent and fields come from the schema |
| Report | Run ID, metric version, freshness, and appendix completeness |
| Interaction | Whether ambiguous definitions trigger follow-up questions and insufficient evidence triggers refusal |

Groundedness is the central metric. Rule validation first checks whether `locator`, `content_hash`, metric version, and artifact exist. LLM-as-Judge can help assess whether the narrative matches the evidence, but it cannot replace hard validation of hashes and locators. Every insight in the report should survive click-through and replay verification.

Negative samples matter. The evaluation set should include bad cases: insights without EvidenceRef, chart fields that do not exist, model-calculated contribution rates, inferences written as facts, and missing definition versions. Without such negative samples, a system can look good in demos and lose control in real reporting.

Online signals should also enter evaluation. Manual report revision rate, approval rejection rate, chart regeneration rate, negative feedback, and recommendation adoption rate all reflect presentation-layer quality. If SQL is correct but reports are often rejected, the issue lies in insight organization, definition explanation, or recommendation boundary.

Evaluation should cover edited reports. Many systems assess only the first draft and ignore whether evidence remains valid after manual edits. The report editor should revalidate EvidenceRef on save: whether references exist, hashes match, sentences remain evidence-supported, and chart fields still correspond to the data. The edited artifact is the final deliverable, so draft-only evaluation is incomplete.

LLM-as-Judge can evaluate language clarity and insight relevance, but numerical values, sorting, evidence location, metric version, and data freshness should be checked by rules or programs. Model review is suitable for judging whether a paragraph reads like a business report. It is not suitable for deciding whether a specific 58% came from the data.

## 36.7 East China Decline Report Sample

The following skeleton illustrates a report for the East China decline Run. The sample data is used only to show structure. Currency units and decimal precision should be handled consistently by the report template.

**Title:** East China Operational GMV Decline Review for Last Week (Pending Controller Confirmation)

**Run ID:** `run-8f3a`

**Metric:** Operational GMV `gmv_ops@2025Q1`

**Data as of:** 2025-06-14 06:00

**Executive Summary**

- Operational GMV in East China fell 12.3% week over week, with a regional gap of about RMB 13.1 million. [`ev_001`]
- SKU-A, SKU-B, and SKU-C together contributed 58% of the decline gap; SKU-A alone contributed 32%. [`ev_002`]
- The personal care category share increased by four percentage points, consistent with the category of the top declining SKUs. This is an inference and requires confirmation from the category manager. [`ev_004`]
- The `sku_id` null rate is slightly elevated, so top-SKU ranking should be treated as indicative. [`ev_005`]

**Findings**

Fact: SKU-A has a gap of RMB 4.2 million, accounting for 32% of the regional decline.

Inference: The higher personal care category share may relate to promotion ending or a mix shift.

Hypothesis: Cross-region channel diversion has not been determined because channel-level detail was not included in this Run.

**Recommendations**

Ask the category manager to run a post-mortem on SKU-A promotion, checking promotion end date, replenishment, and shelf placement changes. This recommendation requires controller approval before it enters downstream tasks.

**Appendix**

The appendix should contain the semantic SQL hash, Python script hash, metric version, freshness, quality warnings, EvidenceRef list, and report approval record. Readers do not need to open the appendix every time, but audit and post-mortem review must be able to open it.

If the controller rejects the report, the system should retain the rejection reason. For example, "the personal care mix shift lacks promotional-calendar evidence" can become a data-backfill task for the next Run, or it can enter the evaluation set so the model does not present the same type of inference as fact in future reports. A report is the interface through which data work enters the organization's operating process.

If the report is approved, the publication event should also enter Trace. The online page, PDF, exported file, and notification message should bind to the same `report_id` and `run_id`. When someone forwards the PDF later, the Run ID in the footer can connect the static file back to the platform's evidence chain.

## 36.8 Presentation Layer and DataAgent Report Artifacts

The presentation layer can be divided into two tools and one template directory. `chart_renderer` generates and validates chart specs. `report_renderer`, or the DataAgent template system, produces report artifacts. `core/eval/` evaluates the output. Charts and reports read upstream artifacts and do not recalculate core numbers.

```text
mini-platform/tools/chart_renderer/
├── handler.py
└── spec_schema.json

mini-platform/agents/data_agent/templates/
└── weekly_ops.md
```

Before launch, at least four things should be checked: chart `data_ref` uses the same source as the SQL/Python artifact, every insight has EvidenceRef, external reports enter HITL by default, and the report footer includes Run ID plus `metric_id@version`. These requirements look small, but missing any one of them makes replay difficult.

Templates should limit information density. The executive summary should contain no more than five points, charts should not crowd a page, and inferences and hypotheses should be marked. DataAgent reports help business users meet and decide. They are not a showcase for generating long text.

In implementation, `chart_renderer`, report templates, and Eval should not maintain separate evidence fields. The EvidenceRef schema should be centrally defined and reused across tool results, report artifacts, frontend footnotes, and evaluators. Once fields diverge, charts may become traceable while reports are not, or report evidence may fail to open in the frontend.

The first release can support one report template and two or three chart types. A narrower scope with solid evidence chains and HITL is more valuable than supporting PPT, PDF, web pages, and many chart types at the same time. Presentation-layer complexity comes from evidence and editing, not from the number of file formats.

Report withdrawal should also be supported after publication. Data quality rollback, metric definition revision, approval changes, or permission changes may require withdrawal of published material. Withdrawal should not delete historical evidence. It should mark report status, retain the reason, and prevent old links from continuing to circulate as valid conclusions. If the report has been sent to multiple recipients, the withdrawal event should notify the original recipients and show the latest status on the report page. That status synchronization also belongs in the audit record.

## 36.9 Report Review and Publication Boundary

A DataAgent report is not a long chat answer. Once it enters a business review, weekly report, or approval process, it becomes a business artifact that can circulate. The platform should distinguish draft, reviewed, published, and archived states, recording the owner and evidence version for each state. Without that lifecycle, users cannot tell whether a report is a model-generated suggestion or a conclusion confirmed by a business owner.

Chart explanations must bind to data evidence. The model can generate natural-language attribution, but every key judgment should return to EvidenceRef: SQL query, Python calculation, chart parameters, time range, metric definition, and data snapshot. If a report says "the East China decline mainly came from one category" without the corresponding contribution calculation and chart reference, that sentence should not enter the published version.

Human review is more than polishing language. The reviewer should check whether conclusions are evidence-supported, whether recommendations are executable, whether obvious business constraints are missing, and whether the report contains sensitive fields or internal judgments that should not be shared. Review comments should also enter Trace. Later evaluation can distinguish model errors, insufficient evidence, and human rewrites, giving report quality work a clearer direction.

Report templates should not force every task into the same structure. Anomaly diagnosis, business post-mortem, compliance explanation, and action planning need different organization. The platform can maintain a template library, but template version should be recorded together with the Agent, semantic layer, and chart components. After publication, a template upgrade should not change the replayed content of historical reports.

## 36.10 Evaluation Samples for Report Quality

Report quality should not depend only on human reading impressions. DataAgent reports need dedicated evaluation samples covering conclusion correctness, evidence completeness, expression fit, action recommendations, and security compliance. Each sample should include the user question, allowed data domains, expected evidence, prohibited sensitive content, and the human-approved conclusion boundary. Model changes, template changes, and chart component changes can then be regressed against the same set of samples.

Evaluation should separate factual errors from expression issues. Factual errors include wrong metric definitions, incorrect contribution calculations, wrong time ranges, and missing evidence references. Expression issues include excessive certainty, non-executable recommendations, inappropriate audience fit, and confusing report structure. These two categories have different fixes. Factual errors usually lead back to the semantic layer, SQL, Python, or chart contract. Expression issues lead back to templates, prompts, and model judging.

Reports also need security evaluation. Business reports may contain sensitive fields, internal judgments, non-public plans, or conclusions unsuitable for external circulation. The platform should check for detail-level data, personal information, unmasked customer names, unapproved export links, and high-risk recommendations. For publishable reports, the human review card should clearly state whether the material is suitable for internal circulation, cross-department circulation, or external disclosure.

Evaluation results should feed template governance. If many samples show missing evidence footnotes, the template structure should change. If the model often writes correlation as causality, conclusion constraints should change. If action recommendations often exceed user authority, Policy and HITL should intervene earlier. Report quality is a concentrated expression of the full DataAgent chain.

## 36.11 Operational Feedback After Publication

Publication is not the end of the chain. Business users may comment, forward, copy charts, ask for more dimensions, or point out during a meeting that a conclusion does not fit the field situation. If this feedback remains only in chat records, it cannot improve future Agents. The platform should classify feedback into factual correction, expression revision, recommendation adjustment, and permission issue, then route each category back to the semantic layer, analysis chain, report template, or security policy.

Factual correction has the highest priority. When a user points out a wrong metric definition, a chart denominator problem, or a misclassified anomaly, the platform should request evidence and add the sample to evaluation. Expression revisions can improve templates but should not directly change factual calculation. Recommendation adjustment depends on user role and business process, so one meeting comment should not automatically become a global rule.

Report circulation also needs operation. Internal business reviews, cross-department post-mortems, and external disclosure materials require different evidence and wording standards. DataAgent may generate multiple publication versions, but each version should record which fields were removed, which evidence remained, and who approved publication. If the same report is copied to different channels without version records, responsibility becomes hard to trace later.

Operational feedback also exposes product experience issues. Frequent requests to "show details" indicate that the chart summary is insufficient. Frequent requests to "use another definition" indicate that the semantic layer entry point is unclear. Frequent requests to "send this to someone" indicate that reports need integration with workflow systems. The report layer is therefore not a standalone writing module. It is the interface where DataAgent moves from analysis into business action.

## 36.12 Degradation Strategy for the Report Layer

When report generation fails, the system should not discard the analysis already completed. If SQL, Python, and chart artifacts are available but the model cannot generate a full report, the frontend can degrade to an evidence panel and data summary. If chart generation fails but calculation results are usable, the page can show tables and textual explanations first. If evidence is insufficient, the system should produce a draft marked "evidence needed" rather than forcing a conclusion.

Degradation should avoid pretending to succeed. A report that has not passed review should stay in `draft` or `needs_review` state. Missing evidence should be visible in the title and body. Users may continue editing, but the system should not treat the artifact as published. This boundary matters for management readers because a complete-looking report format can make unverified conclusions look validated.

Degraded artifacts should also enter Trace. A failed report may still contain completed SQL, Python artifacts, chart drafts, and model errors. Keeping these intermediate states helps the team decide whether to fix the model, template, data, or permissions. Without degradation records, a failure appears only as "generation failed" and loses much of the evidence needed for improvement.

## 36.13 Publication Responsibility of EvidenceRef

EvidenceRef is not a decorative field in a report. It is a precondition for business circulation. Every conclusion should return to concrete evidence: SQL query, Python artifact, chart spec, metric version, data freshness, and human review record. When implementing the report layer, readers should avoid treating EvidenceRef as a simple list of links. It carries publication responsibility: which Run produced the conclusion, which semantic layer version was used, which checks passed, whether permissions were trimmed, and whether visible content was desensitized.

Review before publication should revolve around EvidenceRef. Reviewers cannot check every word of natural language, but they can check whether conclusions match evidence. If a report says "the East China revenue decline mainly came from Channel A," EvidenceRef should point to channel-level decomposition, time window, revenue metric definition, and anomaly detection result. If the evidence supports only "East China revenue declined" but not "mainly came from Channel A," the report layer should lower the expression strength and write "Channel A is one candidate cause." This is production discipline around evidence level.

EvidenceRef also supports later feedback. When business users report that a conclusion is wrong, the platform needs to know whether the error came from data, analysis, chart selection, text interpretation, or action recommendation. If each conclusion has evidence, feedback can be written back to the right step. Chapter 38 stores the run process in Trace. Chapter 39 turns issues into evaluation samples. The report layer aligns user-visible conclusions with underlying evidence. Together they turn a wrong report into an engineering problem that can be fixed.

## 36.14 Operational Loop After Report Publication

Report generation is not the end of the workflow. Enterprise reports are forwarded, commented on, downloaded, copied into meeting material, and sometimes trigger approvals, tickets, or sales actions. If the platform records only generation success, it cannot tell whether the report created value. A fuller operating loop should record reading, follow-up questions, subscriptions, human revisions, conclusion adoption, and downstream actions. This does not require turning every behavior into a complex metric, but the platform should at least distinguish "opened and closed," "continued asking," and "published to a team."

Operational data should influence report generation strategy. Paragraphs that are often followed by questions may need more explanation. Recommendations that are often deleted may exceed the generation boundary. Charts that are often copied may deserve fixed templates. The platform can feed this back into evaluation samples and template governance rather than treating it as product analytics only. In this way, the report layer moves from one-time text generation to a continuously improved analytical product.

Published reports also need version retention. A report seen at different times may differ because data refreshed, the semantic layer changed, analysis code updated, or human reviewers edited content. The platform should store published versions, draft versions, and revision records so that the same link does not show different conclusions later without explanation. For reports used in decisions or compliance review, version records matter more than fluent language. DataAgent reporting becomes suitable for formal collaboration only when it has this publication discipline.

## 36.15 Constrained Language in Reports

The most common quality issue in the report layer is writing uncertain conclusions as definite judgments. Models are good at fluent explanations, but business reports need to separate fact, inference, hypothesis, and recommendation. Facts come from data results. Inferences come from analytical methods. Hypotheses need further verification. Recommendations involve business action. If these categories are mixed, readers may assume that every sentence has the same evidence level.

The platform can constrain report language through output structure. Each conclusion should mark its evidence source. Each inference should mark its dependency. Each recommendation should mark prerequisite assumptions and risk. When evidence is insufficient, the report should use phrases such as "candidate cause," "requires further confirmation," or "the current data supports this interpretation." The purpose is to help readers judge what can be used directly and what requires review.

Language constraints also improve evaluation. Evaluation should check whether evidence level matches expression strength in addition to fluency. A strong conclusion without evidence should fail. A response with insufficient evidence that clearly states its limitation may be an acceptable degraded output. This matches how reports are used inside enterprises.

After the presentation layer goes live, the team should regularly sample real reports. Review should cover whether conclusions address the question, charts support the conclusions, EvidenceRef can be opened, and edited versions still keep evidence. This manual sampling can work with the online evaluation in Chapter 40, turning rejected reports, heavily revised paragraphs, and frequently removed charts into improvement samples.

A mature DataAgent reporting system reduces the user's manual verification cost over time. Users can still challenge conclusions, but they can quickly see the data source, calculation process, and responsible party. At that point, the report layer is no longer only model writing capability. It is part of the enterprise data workflow.

## Chapter Recap

The presentation layer organizes SQL and Python results into business material that can be discussed, approved, and traced. Insights must explain concentration, structural change, and action significance instead of restating numbers. Chart specs must bind to input data and evidence hashes, avoiding untraceable images. EvidenceRef is the central contract behind report credibility; each key insight should open back to its evidence. Output evaluation must cover reports, charts, refusals, clarification questions, and edited artifacts. SQL correctness alone cannot prove that a DataAgent has completed a business analysis task.
