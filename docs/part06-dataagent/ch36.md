!!! note "HITL 与 EvidenceRef 是什么？"
    **HITL**（Human-in-the-Loop，人在回路，Ch.30）：对外报告、经营会材料等 Run 在发布前 **暂停**，等人点「通过/驳回」，避免 Agent 直接对外发错误数字。  
    **EvidenceRef**（证据引用）：报告里每条洞察绑定的 **可追溯指针**（哪张表、哪一行、哪个 Metric 版本）——Eval 称 **groundedness（证据锚定）**，见 §4、§6。

!!! note "LIDA 是什么？"
    **LIDA**（Dibia，2023 [1]）是微软研究院的 **自动可视化** 工具：先摘要数据 → 枚举可视化目标 → 生成 Vega-Lite/matplotlib 代码 → 执行 → 评估。DataAgent 不直接嵌入 LIDA，但 **`chart_renderer` Tool 可映射为类似多阶段流水线**（规则选型 + LLM 填 spec + 校验 `data_ref`），并强制每条洞察绑定 `EvidenceRef`。

### 从查询结果到业务洞察

Ch.34 返回一张 Top SKU 表；Ch.35 的 Python 脚本算出 **品类贡献度** 与 **SKU 差额占比**。Planner 的下一项工作不是把数字念一遍，而是生成 **洞察（Insight）**——帮助读者理解 **发生了什么、集中在哪里、可能意味着什么**。

#### 「洞察」与「复述数字」有何区别？

| 类型 | 示例（华东下滑案例） | 问题 |
| --- | --- | --- |
| **复述** | 「华东运营 GMV 下降 12.3%」 | 仅重复 SQL 聚合行，无结构 |
| **复述** | 「SKU-A 上周 GMV 为 820 万」 | 单点数字，无对比或占比 |
| **洞察** | 「下滑集中在 SKU-A/B/C，三者合计贡献区域 GMV 差额 **58%**」 | 说明 **集中度**，可下钻 |
| **洞察** | 「日化品类占 GMV 比重升 **4pp**，同期 SKU-A 属日化」 | 关联 **结构变化** 与 Top SKU |
| **可行动洞察** | 「建议品类经理对 SKU-A 做促销 post-mortem；补货策略须人工评估后执行」 | 含 **建议边界**，待 HITL |

**复述** 把表里的数搬到自然语言里；**洞察** 回答「So what」——差额集中在哪、结构是否迁移、是否值得进一步动作。Eval 上二者可分档打分（§6）。

#### 走读：华东案例 Planner 如何从结果集生成洞察

Ch.35 沙箱返回的结构化 JSON（示意）：

```json
{
  "dataframe_ref": "df_sku_contrib",
  "content_hash": "sha256:8f3a2b1c",
  "metrics": [{"metric_id": "gmv_ops", "version": "2025Q1", "title": "运营 GMV"}],
  "summary": {
    "region_delta_pct": -0.123,
    "top_skus": [
      {"sku_id": "SKU-A", "gmv_delta": -4200000, "contrib_pct": 0.32},
      {"sku_id": "SKU-B", "gmv_delta": -2100000, "contrib_pct": 0.16},
      {"sku_id": "SKU-C", "gmv_delta": -1300000, "contrib_pct": 0.10}
    ],
    "category_shift_pp": {"daily_chemicals": 4.0}
  }
}
```

Planner 生成洞察时的 **输入约束**：

- 只接受 **结构化结果**（上例 JSON 或 ≤50 行的聚合表），禁止把 raw 万行塞进 prompt；  
- Question Frame 与 `metric_id@version` 来自 Ch.33 Linking（本例 **`gmv_ops@2025Q1`**）；  
- 同比、环比、贡献度等 **须由 SQL 或 Python 预先计算**，LLM 只叙述与解释（§8.3）。

**输出约束**（写入 prompt 与 Policy）：

- **每条结论至少绑定一个 `EvidenceRef`**（结构见 §4）；  
- 写入报告 artifact 的 `insights[]` 与 Trace 的 `insight.evidence_refs[]`；  
- **禁止** 仅输出「依据：见上表」类不可解析的自由文本。

生成结果示意：

```json
{
  "insights": [
    {
      "level": "fact",
      "text": "华东上周运营 GMV（gmv_ops@2025Q1）较前周下降 12.3%。",
      "evidence_refs": [{"ref_id": "ev_001"}]
    },
    {
      "level": "fact",
      "text": "SKU-A/B/C 合计贡献区域 GMV 下滑差额的 58%。",
      "evidence_refs": [{"ref_id": "ev_002"}, {"ref_id": "ev_003"}]
    },
    {
      "level": "inference",
      "text": "日化品类 GMV 占比升 4pp，与 SKU-A 品类一致，疑似结构迁移（待品类经确认）。",
      "evidence_refs": [{"ref_id": "ev_004"}]
    }
  ]
}
```

#### 常见误区

**误区 1：洞察 = 润色 SQL 注释。**  
「SELECT 结果说明华东下降」不是洞察；须回答集中度、结构或对比关系。

**误区 2：LLM 心算贡献度。**  
「约占一半」若未在 Python 输出中出现，属于 **幻觉风险**；Eval 须用金标准表校验（§6）。

**误区 3：一条洞察绑多个 unrelated 证据。**  
每条 `EvidenceRef` 应能 **独立支撑** 该句；多 SKU 合计可绑多个 ref，不可绑整张未定位的表。

### 图表类型选择与可视化表达

表格适合精确数字；经营会材料还需要 **一眼看出排名、趋势或构成** 的图表。DataAgent 的 `chart_renderer` 不直接「画图给业务看」，而是产出 **声明式 spec + 数据绑定 + 证据 hash**，由 Ch.48 Generative UI 渲染。

#### 如何让 Agent 选对图表？

Wilke 的经典原则是：**图表服务于比较意图** [4]。本书在 Planner / `chart_renderer` 中约定如下选型规则：

| 分析意图 | 推荐图表 | 避免 | 华东案例用法 |
| --- | --- | --- | --- |
| **趋势** | 折线 | 饼图展示时间序列 | 周度 GMV 走势（若多周） |
| **排名 / 贡献** | 横向条形 | 3D 柱、过多颜色 | **SKU GMV 差额 Top 10** ⭐ |
| **构成** | 堆叠条/面积 | 15+ 扇区饼图 | 品类 GMV 占比变化 |
| **分布** | 直方/箱线 | 折线硬连离散类别 | 门店 GMV 分布（异常 EDA） |
| **相关** | 散点 | 伪因果叙述 | 价格 vs 销量（探索） |

**LIDA** [1] 用 **Goal Explorer** 枚举可视化目标，再由 LLM 生成图表代码；DataAgent 不嵌入 LIDA，但 **`chart_renderer` 采用更可控的做法**：先用 **规则表（上表）** 锁定图表类型，再由 LLM 填写 spec 字段，输出 **Vega-Lite 或 ECharts JSON**（Ch.48 渲染）。

#### 走读：华东 SKU 贡献图如何绑定数据

Planner 在生成 Top SKU 表后，调用 `chart_renderer@v1`，输入：

```json
{
  "intent": "rank",
  "metric_context": {"metric_id": "gmv_ops", "version": "2025Q1", "title": "运营 GMV"},
  "dataframe_ref": "df_sku_contrib",
  "x_field": "gmv_delta",
  "y_field": "sku_id",
  "title": "华东 SKU 运营 GMV 差额贡献（gmv_ops@2025Q1）"
}
```

`chart_renderer` 返回的 **Tool Result** 须含：

| 字段 | 说明 | 华东示例 |
| --- | --- | --- |
| `spec` | Vega-Lite / ECharts 声明式 JSON | 见 §8.1 |
| `data_ref` | 与 SQL/Python **同源** 的 artifact 路径 | `/runs/run-8f3a/df_sku_contrib.parquet` |
| `chart_id` | Run 内图表标识 | `c_102` |
| `evidence` | 输入 DataFrame 的 `content_hash` | `sha256:8f3a2b1c` |

若 `data_ref` 与 `sql_result` 不一致（例如 LLM 编造序列），Eval 与审计均视为 **失败**。

#### chart_renderer 与 Generative UI 的分工

| 层级 | 职责 |
| --- | --- |
| **chart_renderer** | 选型 + 生成 spec + 校验字段存在于 schema |
| **Generative UI（Ch.48）** | 渲染 spec；点击图表元素时根据 `evidence` 高亮对应表格行 |
| **Trace（Ch.38）** | 记录 `chart_id`、`data_ref`、`content_hash` |

### 自动 EDA 与异常发现

**Auto-EDA**（自动探索性数据分析）指：对结果集 **自动** 输出缺失率、分位数、离群点等摘要，而不等用户逐列点开看。仅做 **查数、不做诊断** 时可关闭；**诊断类任务**（如华东下滑）适合在 SQL 聚合之后 **自动跑一层轻量 EDA**，把突变与空值告警写进 Planner 上下文，供 §1 洞察生成使用。

工程上可参考 **ydata-profiling** 等工具生成 profiling，但 DataAgent 只消费 **结构化 JSON 摘要**（非 HTML 长报告），避免 prompt 膨胀。Tang 等综述 [2] 将 Auto-EDA 列为分析 Agent 能力之一。DataAgent 须控制自动化 **级别**：

| 级别 | 行为 | 适用 |
| --- | --- | --- |
| **L0** | 仅用户问及时 EDA | 仅查数场景 |
| **L1** | 诊断类任务自动摘要分布、Top 变化 | **华东下滑案例** ⭐ |
| **L2** | 主动推送异常至任务工作台 | 订阅 + 告警 |

#### L1 自动 EDA 输出什么？

华东 Run 在 `python_sandbox` 之前或之后，可附加：

```json
{
  "eda_level": "L1",
  "method": "zscore_vs_prior_week",
  "flags": [
    {"sku_id": "SKU-A", "z": -2.4, "note": "低于前周均值 2.4σ"},
    {"column": "sku_id", "null_rate": 0.008, "note": "空值率略高于阈值"}
  ],
  "content_hash": "sha256:eda91f"
}
```

**关键要求**：异常描述须 **标注统计方法**（如「基于前周 SKU 级均值 2σ」），避免 **无法说明检测依据** 的「AI 发现异常」表述。上述 `flags` 可转为 §4 的 `EvidenceRef`，供 footnote 点击。

Python 沙箱（Ch.35）执行 EDA 脚本；结果以 **结构化 JSON** 回 Planner，**不得** 把 profiling 全文 dump 进 prompt。

### 报告结构、叙事与证据引用

经营分析报告不是聊天回复的加长版。Controller 审批时需要 **固定章节、口径可追溯、推断与事实分离**。对外/经营会场景下，Run 在发布前进入 **`waiting_human` 等人审批**（HITL，Ch.30）；本章定义 **报告长什么样**，Ch.30 定义 **审批状态机与 SSE 事件**。

#### 推荐六段结构

| 序号 | 章节 | 内容要点 | 华东案例 |
| --- | --- | --- | --- |
| 1 | **Executive Summary** | 3–5 bullet，**每条带 EvidenceRef** | 下滑幅度、Top SKU、结构推断 |
| 2 | **问题与口径** | Question Frame + `metric_id@version` + 时间范围 | 运营 GMV、华东、上周 vs 前周 |
| 3 | **数据概览** | 核心表 + 主图 `chart_id` | SKU 贡献表 + 条形图 `c_102` |
| 4 | **发现** | 分级：**事实 / 推断 / 假设** | SKU 集中、日化占比升 |
| 5 | **建议** | 可选；对外须 HITL | 促销 post-mortem |
| 6 | **附录** | SQL hash、数据新鲜度、Run ID | `run-8f3a…`、同步截至时间 |

#### 证据引用（EvidenceRef）最小契约

每条洞察须绑定至少一个 `EvidenceRef`，写入报告 artifact 与 Trace（`insight.evidence_refs[]`），供 Generative UI（Ch.48）渲染为 **可点击脚注**。

| 字段 | 说明 | 华东示例 |
| --- | --- | --- |
| `ref_id` | Run 内唯一引用 ID | `ev_002` |
| `source_tool` | 产生数据的 Tool | `python_sandbox@v1` |
| `artifact_id` | Run 内结果标识 | `df_sku_contrib` |
| `locator` | 行/聚合键 | `row:sku_id=SKU-A` |
| `metric_id` | 语义层指标 | `gmv_ops` |
| `metric_version` | 口径版本 | `2025Q1` |
| `content_hash` | 结果集摘要哈希 | `sha256:8f3a2b1c` |

#### 走读：从洞察句子到可点击脚注

**报告正文（渲染后）**：

> **发现 2**：SKU-A 贡献华东运营 GMV 下滑差额的 **32%**（`gmv_ops@2025Q1`）。  
> 依据：脚注 `[2]` → 点击展开来源行。

**artifact 内嵌的 EvidenceRef 对象**：

```json
{
  "ref_id": "ev_002",
  "source_tool": "python_sandbox@v1",
  "artifact_id": "df_sku_contrib",
  "locator": "row:sku_id=SKU-A",
  "metric_id": "gmv_ops",
  "metric_version": "2025Q1",
  "content_hash": "sha256:8f3a2b1c",
  "snapshot": {
    "sku_id": "SKU-A",
    "gmv_delta": -4200000,
    "contrib_pct": 0.32
  }
}
```

**用户点击脚注 `[2]` 时**，Generative UI：

1. 读取 `artifact_id` + `locator`，从 Run artifact 存储拉取 Parquet/JSON 行；  
2. 校验 `content_hash` 与当前 artifact 一致（防篡改）；  
3. 高亮表格中 `SKU-A` 行，并展示 `metric_id@version` 与 `source_tool`。

**不得** 仅展示「依据：Python 分析结果」等无法解析的字符串——Eval 的 groundedness 维度会判 **不合格**。

#### 事实 / 推断 / 假设：叙事分级

| 分级 | 定义 | 华东示例 | HITL |
| --- | --- | --- | --- |
| **事实** | 直接来自聚合/脚本 | 「SKU-A 差额 -420 万，占区域差额 32%」 | 一般不需改 |
| **推断** | 有证据的因果猜测，须标注 | 「可能与上周促销结束相关」 | Controller 可批注 |
| **假设** | 证据不足，待验证 | 「是否存在渠道窜货需补数」 | 须明确「待验证」 |

#### HITL 边界

| 报告类型 | 是否 `waiting_human` |
| --- | --- |
| 内部探索、单次问数 | 否 |
| **经营会材料** | **是** ⭐ |
| 监管报送 | 是 + 法务节点 [Ch.30] |

### 行动建议与业务闭环

Agent 能否替业务做决定？**不能。** 建议 = **基于证据的选项**，须明确 **采纳主体** 与 **未采纳时的风险**，而不是系统指令。

#### 建议三级表述

| 等级 | 表述 | 华东示例 |
| --- | --- | --- |
| **观察** | 仅事实 | 「SKU-A 运营 GMV 较前周下降 40%」 |
| **推断** | 有证据支持的因果推断 | 「下滑时点与促销结束日重合，需促销日历验证」 |
| **建议** | 明确需人工采纳 | 「建议品类经理评估 SKU-A 补货与下周促销方案」 |

报告模板中，**建议** 单独成节，默认带 `requires_approval: true`；Controller `approve` / `reject` 写入 Trace（Ch.30）。

#### 闭环跟踪

- 用户 **采纳/驳回** 建议 → 写入 Eval 反馈 [Ch.37 §5]；  
- 高采纳率建议 → 沉淀为 **Playbook**（可复用步骤模板；Ch.26 Reflexion 支持从失败 Run 中晋升有效策略）；  
- 驳回率高的建议类型 → 下调 Planner 自动输出级别或改 prompt。

### 输出质量评估：DataAgent 不能只评 SQL

**SQL EM**（Exact Match，SQL 字符串是否与金标准完全一致）在 Spider/BIRD 等公开集上常用，但 **只证明取数 SQL 对不对**。DataAgent 上线后，用户抱怨的往往是 **「图不对题」「结论没依据」「该问没问」**。输出 Eval 须与 Ch.39 流水线配合，覆盖下表维度（**本章定评什么，Ch.39 定怎么跑**）。

| 维度 | 指标示例 | 评测方式 |
| --- | --- | --- |
| **SQL/执行** | EM（SQL 完全匹配）、EX（执行结果匹配） | Spider/BIRD 公开集 [Ch.39] |
| **答案** | 数值容差、排序一致 | 山岚业务金标准表 |
| **洞察 groundedness** | 每条 insight 有合法 `EvidenceRef` | 规则校验 + LLM-as-Judge（模型辅助打分，不可替代 hash 硬校验） [7] |
| **图表** | 类型匹配意图（§2 表） | 规则 + 人工 |
| **报告** | 口径脚注、`metric_id@version` 完整 | checklist |
| **拒答/追问** | 该追问时是否追问 | 对抗样本 [5] |

BIRD-INTERACT [5] 强调 **多轮澄清与交互**（见 [Ch.32 §1](ch32-dataagent.md)）；输出 Eval 应包含 **「口径歧义时是否 ASK」** 与 **「缺证据时是否拒写洞察」**。

#### 负样本：缺乏证据的洞察

金标准集须含 **故意缺 `EvidenceRef` 的坏例**，例如：

```json
{"text": "华东下滑主要因竞品降价", "evidence_refs": []}
```

自动化规则应 **判 fail**；LLM-as-Judge 仅作补充，不可替代 `locator` + `content_hash` 的硬校验。

#### 线上信号

| 信号 | 含义 |
| --- | --- |
| 用户负面反馈 | 洞察不可信 / 图表难读 |
| 人工改 SQL 率 | NL2SQL 缺口 |
| 报告审批驳回率 | 表达/HITL/口径问题 |
| 建议采纳率 | 业务价值 |

### mini-platform 工程路径：图表、报告与 Eval

!!! note "目录说明"
    下列路径为 **Part VI 目标架构（书中契约）**；当前仓库 **尚未合入** `tools/chart_renderer/`、`agents/data_agent/templates/` 等目录。**Part V** 模块与 `mini-platform/projects/multi-agent-workflow/` 已在仓库中存在。输出质量 Eval 维度见 §6，流水线实现见 [Ch.39](../part07-observability-eval/ch39-dataagent-eval-benchmark.md)。

本章描述表达层 **chart_renderer**、报告模板与 Eval 契约。

#### 8.1 模块目录

```
mini-platform/tools/chart_renderer/
├── handler.py             # Registry Tool：chart spec 生成与渲染
└── spec_schema.json

mini-platform/agents/data_agent/templates/
└── weekly_ops.md          # 山岚经营周报模板（EvidenceRef 变量）
```

**Vega-Lite 最小 spec**（`chart_renderer` 输出；`data_ref` 须与 SQL/Python 同源）：

```json
{
  "$schema": "https://vega.github.io/schema/vega-lite/v5.json",
  "title": "华东 SKU 运营 GMV 差额贡献（gmv_ops@2025Q1）",
  "data": {"url": "artifact://df_sku_contrib?format=parquet"},
  "mark": "bar",
  "encoding": {
    "y": {"field": "sku_id", "type": "nominal", "sort": "-x", "title": "SKU"},
    "x": {
      "field": "gmv_delta",
      "type": "quantitative",
      "title": "GMV 差额（元）",
      "axis": {"format": ",.0f"}
    },
    "color": {"field": "contrib_pct", "type": "quantitative", "legend": {"title": "占区域差额%"}}
  },
  "data_ref": "/runs/run-8f3a/df_sku_contrib.parquet",
  "evidence": {"content_hash": "sha256:8f3a2b1c", "metric_id": "gmv_ops", "metric_version": "2025Q1"}
}
```

`handler.py` 职责：校验 `encoding` 字段存在于 `dataframe_ref` schema；写入 Trace 的 `chart.data_ref` 与 `chart.evidence`；返回 `chart_id` 供报告模板引用。

**报告模板变量**（`weekly_ops.md` 示意）：

```markdown
## Executive Summary
{{#each insights.summary}}
- {{text}} [{{#each evidence_refs}}{{ref_id}}{{/each}}]
{{/each}}

## 口径
指标：{{metric.title}}（`{{metric.id}}@{{metric.version}}`）
Run：`{{run_id}}`
```

#### 8.2 生产化 checklist

- [ ] 洞察条数上限（如 Executive Summary ≤5），避免信息堆砌  
- [ ] `EvidenceRef` 必填校验：`locator` + `content_hash` 不可空  
- [ ] 对外报告强制 HITL；SSE 发布事件 `report_published`  
- [ ] 图表 `data_ref` 与 SQL/Python artifact **同源校验**  
- [ ] Eval 集含 **缺 evidence 的洞察** 负样本（§6）  
- [ ] 报告页脚强制 `run_id` + `metric_id@version`

#### 8.3 实务注意

!!! warning "LLM 不得心算同比、环比或贡献度"
    上述数值须由 SQL 或 Python 预先计算；LLM 只负责叙述、分级（事实/推断）与排版。心算贡献度是 groundedness Eval 的 **头号失败模式**。

1. **饼图扇区过多（如 15 个）。**  
   规则引擎应限制扇区数量，自动改横向条形（§2 表）。

2. **报告未携带 Run ID，审计无法回放。**  
   页脚须强制包含 `run_id`、Semantic SQL hash 与 Metric version。

3. **推断写进 Executive Summary 未标注。**  
   模板层对 `level=inference` 自动加「推断」前缀，便于 Controller 审批。

## 本章小结

### 关键结论

1. **表达层** 交付决策材料，不是 SQL 日志；华东案例口径为 **`gmv_ops@2025Q1`（运营 GMV）**。  
2. **洞察 ≠ 复述**：须说明集中度、结构或对比；每条绑定 **EvidenceRef**。  
3. **图表选型可规则化**；LIDA 式多阶段生成映射到 `chart_renderer` Tool 链 [1]。  
4. **groundedness** 是 DataAgent 可信核心；Eval 必须覆盖洞察、图表与报告 checklist。  
5. **建议分级 + HITL** 控制业务风险；与 Ch.48 前端、Ch.39 评测、Ch.37 持续改进形成闭环。
