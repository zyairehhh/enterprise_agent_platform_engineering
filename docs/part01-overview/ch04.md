## 为什么总论最后必须回到一张地图

前三章分别回答了三个问题：什么是 Agent，为什么企业会走向平台化，什么是 AI 原生业务系统。到这里，读者已经能判断一个系统是不是 Agent，也能理解平台不是框架、AI 原生不是聊天框。但如果没有一张全局地图，这些判断仍然容易散开。

企业级 Agent 平台不是一个单点技术，也不是一组漂亮组件。它同时牵涉模型、数据、知识、工具、流程、前端、评估、安全、部署和组织协同。读者如果只记住“Runtime 很重要”“语义层很重要”“评估很重要”，却不知道它们之间的依赖关系，后面阅读就会变成碎片化知识积累。

山岚集团的平台负责人真正需要回答的，也不是“有哪些模块可以做”，而是更具体的问题：

- 第一个生产级 Agent 上线前，哪些底座能力必须先有？
- DataAgent 为什么不是“NL2SQL + 图表”这么简单？
- Runtime、Tool Registry、审批、trace、评估之间是什么关系？
- 为什么本书先讲模型、数据、知识，再讲 Agent 能力和业务系统？
- 不同读者应该顺序读完，还是按角色跳读？

第四章的作用，就是把前三章的概念判断压缩成一张可执行的阅读地图。它不是目录复述，而是告诉读者：后面每一章在整个平台中处于什么位置，解决哪类问题，和哪些章节存在前后依赖。

## 四层参考架构：从业务任务到治理底座

企业级 Agent 平台最容易画坏的一点，是一上来就罗列太多组件。更适合总论的方式，是先用四层理解全局。

第一层，是**业务任务层**。这里是用户真正感知到价值的地方：DataAgent、报价 Agent、工单 Agent、经营分析工作台、销售任务工作台，以及后续各种 AI 原生业务系统。它回答的是“企业用 Agent 完成什么任务”。

第二层，是**Agent 能力层**。这里决定 Agent 如何被组织和运行：任务状态、工具调用、规划策略、长任务、人工介入、多 Agent 协作、协议与框架选型。它回答的是“Agent 如何把任务推进下去”。

第三层，是**智能与数据层**。这里为 Agent 提供能力原料：模型推理、结构化输出、RAG、知识工程、语义层、湖仓、OLAP、元数据、血缘、指标口径。它回答的是“Agent 凭什么理解、判断和生成结果”。

第四层，是**基础设施与治理层**。这里保证系统能长期运行：模型部署、网关、多租户、可观测性、评估、成本、限流、降级、安全、合规和组织机制。它回答的是“Agent 如何被稳定、可信、可控地运行”。

| 层级 | 核心问题 | 主要对应章节 |
|---|---|---|
| **业务任务层** | 用 Agent 完成什么业务任务 | Part VI, Part XI |
| **Agent 能力层** | Agent 如何规划、调用工具、执行长任务 | Part V |
| **智能与数据层** | Agent 使用什么模型、数据和知识 | Part II, III, IV |
| **基础设施与治理层** | 系统如何部署、观测、评估、安全运行 | Part VII, VIII, IX, X |

这四层不是为了好记，而是为了帮助读者把问题放回正确位置。

如果没有这张层次图，企业很容易发生两种误判。第一种，是把所有问题都下沉成模型问题。明明是语义层缺失、工具契约混乱、审批边界不清，却被归咎于“模型不够强”。第二种，是把所有问题都抬高成业务问题。明明是 trace 不统一、评估样本缺失、运行状态不可恢复，却被说成“业务场景太复杂”。

四层架构的价值，在于让团队拥有定位问题的共同语言。

## 八个能力簇：平台骨架与失败模式

在四层之中，真正构成 Agent 平台骨架的，是一组会反复出现在后续章节里的能力簇。本书把它们收束成八个。

| 能力簇 | 它解决的问题 |
|---|---|
| **Runtime** | 任务如何被创建、推进、暂停、恢复和终止 |
| **Registry** | 工具、Agent、能力和版本如何统一管理 |
| **Planner** | 系统如何决定下一步，而不是只生成文本 |
| **Memory** | 会话状态、长期偏好和任务上下文如何延续 |
| **RAG / Knowledge** | 文档、元数据和知识上下文如何进入 Agent |
| **Observability** | trace、日志、指标和会话回放如何统一记录 |
| **Eval** | 如何判断版本变好还是变坏 |
| **Policy** | 权限、脱敏、审批、安全边界如何被执行 |

这八个能力簇不是功能清单，而是企业失败模式清单的反面。

没有 Runtime，系统只能演示，不能可靠执行。没有 Registry，工具会越来越多、越来越乱。没有 Planner，模型会在错误路径上自由发挥。没有 Memory，长任务和多轮任务会迅速失真。没有 RAG 和知识工程，系统无法真正连接企业上下文。没有 Observability，出错后无法解释。没有 Eval，版本好坏只能靠感觉。没有 Policy，越权、泄漏和绕过审批迟早出现。

也可以换一种方式理解这八个能力簇：

| 类型 | 包含能力 | 在平台中的作用 |
|---|---|---|
| **执行骨架** | Runtime, Registry, Policy | 让 Agent 能被统一运行和约束 |
| **智能放大器** | Planner, Memory, RAG / Knowledge | 让 Agent 能理解上下文并动态推进 |
| **反馈系统** | Observability, Eval | 让平台避免黑箱化和失控 |

后面章节虽然会展开很多具体技术，但读者可以始终回到这张表：当前主题到底是在补执行骨架、智能放大器，还是反馈系统？

## DataAgent 主线：让全栈能力显影

本书不是只讲 DataAgent，但它选择 DataAgent 作为主线场景。原因不是问数最热门，而是 DataAgent 几乎天然穿过企业级 Agent 平台的主要层级。

一个看似简单的问题，例如“上周华东区毛利率异常的原因是什么”，背后会同时触发多个系统问题：

| 层级 | DataAgent 为什么绕不过去 |
|---|---|
| **模型层** | 需要理解问题、规划路径、生成 SQL、解释结果 |
| **数据层** | 需要语义层、指标口径、湖仓、OLAP 和数据质量 |
| **知识层** | 需要元数据、历史分析、业务术语、制度和案例 |
| **Agent 层** | 需要 Runtime、工具调用、Planner、状态管理和人工介入 |
| **治理层** | 需要权限控制、trace、评估、成本和审计 |
| **前端层** | 需要图表、表格、引用、报告和任务工作台 |

这就是 DataAgent 的特殊价值：它不是唯一重要的业务场景，但它非常适合用来讲透企业级 Agent 平台为什么是全栈问题。

把这个任务拆开看，山岚集团的一次 DataAgent 请求至少要经过七个检查点：

| 检查点 | 真正要解决的问题 |
|---|---|
| **任务创建** | 用户身份、租户、问题范围是否明确 |
| **上下文加载** | 指标口径、历史分析、可访问数据是否完整 |
| **路径规划** | 先查什么、后查什么，是否需要跨多个工具 |
| **工具执行** | SQL、API、Python 等动作是否受控 |
| **结果解释** | 事实、推断、建议是否被清楚区分 |
| **治理记录** | trace、成本、审批、风险是否留痕 |
| **结果交付** | 是否给出图表、引用、结论和后续动作 |

如果把 DataAgent 误解成“自然语言转 SQL”，平台建设从第一天就会跑偏。它真正连接的是数据智能、Agent 执行、AI 原生工作台和企业治理闭环。

这也解释了为什么 Part VI 在全书中承担主线地位。它不是孤立案例，而是把前面模型、数据、知识、Agent 能力和后面评估、安全、前端全部拉通的综合场景。

## 全书组织顺序：按平台依赖关系展开

很多 Agent 书会从聊天界面、提示词或工具调用开始讲。这种顺序容易上手，但一旦进入企业落地，就会遇到一个问题：前端体验跑得很快，底层语义、评估、权限、运行时和审计却都没有准备好。

本书的组织顺序，是按企业级平台真正长出来的依赖关系安排的。

先讲模型与推理，因为没有基础推理、结构化输出、模型路由和推理优化，后面的 Agent 决策无从谈起。

再讲数据基础设施和知识工程，因为企业 Agent 的真正差异，往往不来自模型本身，而来自它能否安全地连接正确数据、正确口径和正确知识。

接着进入 Agent 基础能力，因为有了模型、数据和知识之后，才有必要讨论 Runtime、Tool Registry、MCP、Planner、Memory、HITL、多 Agent 与框架选型。

然后通过 DataAgent 主线，把前面底座能力拉通到一个真实业务场景中。

最后进入观测、评估、成本、部署、前端、安全、合规和组织，因为这些能力不是附属装饰，而是生产级平台必须闭合的运行环。

| 书中部分 | 为什么放在这里 |
|---|---|
| **Part II 模型与推理层** | 建立模型能力与结构化输出基础 |
| **Part III 数据基础设施层** | 建立可访问、可信、可治理的数据底座 |
| **Part IV 向量、检索与知识工程** | 让 Agent 能连接企业非结构化知识 |
| **Part V Agent 基础能力** | 建立任务执行、工具调用和协作机制 |
| **Part VI DataAgent 主线深潜** | 用一个综合场景拉通全栈能力 |
| **Part VII-X 生产化与治理** | 补齐观测、评估、成本、部署、安全和组织闭环 |
| **Part XI 案例集** | 把平台能力迁移到更多业务 Agent |

这个顺序不是目录美学，而是工程依赖关系。读者可以跳读，但不能误解依赖。

**不同读者怎样使用这本书。**

第四章除了给技术地图，还要给阅读地图。不同角色不应该用完全相同的方式读这本书。

| 读者 | 更适合先抓住什么 |
|---|---|
| **平台负责人 / CTO** | 平台边界、年度路线、成本、治理和组织协同 |
| **架构师** | 四层架构、八个能力簇、章节之间的前后依赖 |
| **数据智能工程师** | 语义层、RAG、DataAgent、NL2SQL、评估体系 |
| **AI 应用开发者** | Runtime、工具接入、任务工作台、结果交付 |
| **安全 / 合规负责人** | 权限边界、审批、trace、评估、Guardrails 和法规 |

如果团队要共读 Part I，也可以按主题组织：

| 共读主题 | 推荐章节 | 目的 |
|---|---|---|
| **统一概念** | Ch.01-Ch.02 | 先把什么是 Agent、什么是平台讲清楚 |
| **对齐业务方向** | Ch.03 | 判断哪些业务适合 AI 原生化 |
| **对齐建设路线** | Ch.04 | 把后续章节映射到团队路线图 |

如果读者已经有实际项目，也可以按当前痛点跳读：

| 当前问题 | 值得优先阅读的主题 |
|---|---|
| Agent 效果不稳定 | Runtime、Planner、Trace、Eval |
| 问数经常答不准 | 语义层、Schema Linking、NL2SQL、DataAgent 评测 |
| 工具越来越多，风险难控 | Tool Registry、Policy、审批、成本治理 |
| 平台路线说不清 | Part I、平台能力、成本、安全、组织路线 |
| 前端像聊天机器人，不像工作台 | AI 原生业务系统、对话 UI、Generative UI |

这张阅读地图会让本书更像工作手册，而不是只能从第一页顺序读到最后的教材。

## 一年建设路线：从首个试点到平台化复制

如果山岚集团准备用一年时间把企业级 Agent 平台从 0 做到可服务多条业务线，比较合理的节奏不是一开始就做“大而全平台”，而是围绕真实场景逐步沉淀共性能力。

| 阶段 | 技术重点 | 治理重点 | 组织重点 |
|---|---|---|---|
| **第一季度** | Runtime、模型入口、工具注册、基础 trace、首个试点 | 工具风险等级、最小审批准则 | 明确平台团队边界和业务试点负责人 |
| **第二季度** | 评估、成本归集、审批接入、基础管理界面 | 评测样本模板、上线准入标准 | 建立场景共创和复盘机制 |
| **第三季度** | 语义层、RAG、更多工具接入、第二业务线复制 | 统一数据口径、权限和 trace 规范 | 推动业务团队按模板接入 |
| **第四季度** | 灰度、降级、SLO、供应商接入、平台目录 | 事故复盘、版本治理、合规检查 | 建立平台运营节奏和年度路线 |

这张路线图的重点，不是季度数字本身，而是顺序：先把执行链路站稳，再做规模化和组织化。

每个阶段都应该沉淀一种真正可复用的公共资产：

| 阶段 | 应沉淀的公共资产 |
|---|---|
| 第一季度 | 统一任务状态模型、工具风险等级 |
| 第二季度 | 评测样本模板、上线准入清单 |
| 第三季度 | 语义层规范、数据权限和知识接入规范 |
| 第四季度 | 平台目录、复盘模板、成本和质量运营报表 |

很多平台路线图失败，不是因为技术目标写错了，而是因为只写技术目标，不写治理目标和组织目标。只有技术目标，平台会“建出来但用不好”；只有治理目标，平台会“规矩很多但没人愿意接”；只有组织目标，平台会“讨论很多但缺少可运行底座”。三者必须一起推进。

## 第四章收束：让后续章节不再像一堆散点

如果把整本书想象成一张图，Part I 是图例，第四章就是索引页。它的价值不在于总结前三章，也不在于把目录念一遍，而是帮助读者把后面遇到的每个主题重新定位。

读到模型路由时，知道它位于推理层，也会影响成本治理。读到语义层时，知道它为什么对 DataAgent 是刚需。读到 Human-in-the-loop 时，知道它不是局部交互，而是企业责任边界的一部分。读到评估、安全和组织时，知道它们不是收尾章节，而是平台闭环的一部分。

这一章真正想交付的，是整本书的使用方式。

第一，企业级 Agent 平台更适合先看成四层，而不是一堆独立组件。

第二，平台核心可以收束成八个能力簇，后续很多章节其实都在围绕这些能力簇展开。

第三，DataAgent 之所以被选为主线，不是因为它唯一重要，而是因为它最能拉通全栈问题。

第四，本书的组织顺序来自真实依赖关系，而不是目录美学。

第五，Part I 的意义不是替代后续深入章节，而是让读者在进入技术深水区之前，先拥有一张可靠的地图。

从下一部分开始，本书将正式进入模型与推理层。之所以先从这里开始，不是因为模型最重要，而是因为它是后续所有能力得以成立的起点之一。
