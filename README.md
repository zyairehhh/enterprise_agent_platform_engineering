## 三层阅读法

每章固定三段，配合页眉与页脚的层级标注，支持按角色跳读：

| 层 | 篇幅 | 内容 | 适合谁读 |
|---|---|---|---|
| **L1 概念** | ~30% | 业务定义、边界、相邻概念对比、典型误区 | CTO / 安全合规 / 跨职能读者 |
| **L2 架构** | ~40% | 组件划分、数据流、接口契约、状态机、失败模式 | 架构师 / 平台负责人 |
| **L3 工程** | ~30% | mini-platform 可运行代码、配置、生产化 checklist、踩坑 | 工程师 / 应用开发者 |

## 参与贡献

本书采用**主编 + Part Owner + 章节贡献者**三级治理。

1. 在 Issues 中查找待认领章节，使用 [`templates/chapter-claim.md`](templates/chapter-claim.md) 认领
2. fork 仓库，在 `feature/partXX-chYY-<topic>` 分支撰写
3. 提 PR，CI 自动校验（Markdown / 链接 / 术语 / 模板完整性 / 代码可运行 / 敏感信息）
4. Part Owner + 主编 + 至少 1 名 reviewer 评审通过即合并

详细贡献指南见 [CONTRIBUTING.md](CONTRIBUTING.md)。

### v0.1 阶段优先招募

- 主编 1-2 名（负责跨 Part 一致性与全书风格）
- Part Owner 10 名（每 Part 一名，统筹本 Part 章节）
- Part II / III / IV / VI 主笔（模型、数据、知识、DataAgent 四个核心 Part）

## 案例统一背景：山岚集团

全书所有业务案例均基于虚构公司 **「山岚集团」**：

> 山岚集团是一家综合性企业，下设零售、制造、金融、物流四大子板块。年营收百亿规模、3 万员工、年产生数据量 PB 级别。在 Part I 第 3 章会详细介绍其业务体量、数据底座现状与 AI 落地目标。

所有真实公司、真实业务数据均**禁止**进入仓库；CI 流水线会扫描敏感信息。
